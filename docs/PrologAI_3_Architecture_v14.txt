PROLOGAI ARCHITECTURE  (v14)

HOW THE COMPONENTS FIT TOGETHER AND COMMUNICATE

SPARC STEP 3: ARCHITECTURE - THE AUTHORITATIVE STRUCTURE REFERENCE

(All element names are original to PrologAI; this volume gives the structure that realizes the PrologAI Platform Specification and its Pseudocode)


This is the architecture companion to the PrologAI Platform Specification and the PrologAI Pseudocode.

The specification (SPARC Step 1) said WHAT the platform must do; the pseudocode (SPARC Step 2) said HOW each work package reasons; this volume (SPARC Step 3) says WHERE every piece lives, how the pieces are bound into components, and how they communicate at run time.

It defines the component model (repositories, packs, and the Rust crate), the dependency and build graph, the data architecture of the Lattice, the concurrency and coordination model, the runtime control flow of a living mind, the interface contracts between layers, the safety architecture, the deployment topologies, the cross-cutting concerns, and a full traceability map from capability layer to pack to work package.

Nothing here changes a requirement; it places the forty-eight work packages into a buildable, communicating whole.


PART ARCH-0 - ARCHITECTURAL PRINCIPLES


SECTION 0.1 - THE GOVERNING DECISION

The architecture is governed by Architecture Decision Record ADR-1 (Specification Part 31): PrologAI is a layered constructed language and platform built ON an unforked SWI-Prolog, never a fork OF it.

Three structural consequences follow and shape everything below.

Consequence 1 - the engine is an immutable dependency: PrologAI code never modifies the engine; it only loads on top of it, so engine upgrades arrive for free.

Consequence 2 - the reflective boundary is structural: the mind may rewrite PrologAI-layer code (which is Prolog data) but can never reach the engine beneath, which structurally enforces the protected core.

Consequence 3 - performance escalates outward, not inward: hot logic moves down into the Rust crate or up into an upstream proposal before anyone considers an engine patch.


SECTION 0.2 - THE THREE SUBSTRATE LAYERS

The platform stands on three substrate layers (Specification Part 2).

Layer A - the logic engine: an unforked, embedded SWI-Prolog supplying unification, backward chaining, backtracking, threads, message queues, engines, incremental tabling with well-founded semantics, transactions, and persistence.

Layer B - the vector memory engine: a high-performance vector and graph store reached only through a backend-agnostic interface, with the default backend chosen by the PR 2 bake-off.

Layer C - the PrologAI cognitive architecture: PrologAI's own original code, delivered as SWI-Prolog packs plus a launcher, plus the PrologAI language surface.

The architecture's job is to organize Layer C and bind it cleanly to Layers A and B.


SECTION 0.3 - SEPARATION OF CONCERNS

Five separations are load-bearing and are preserved by every component boundary below.

Structure versus annotation: a node_fact's identity is immutable; its value channels mutate freely, so truth maintenance tracks structure, not annotations.

Knowledge versus mechanism: the Lattice holds both, but mechanisms (actors, sentinels, rewrite rules) are themselves node_facts, which is what makes reflection possible.

Specification versus implementation: the Minimal PrologAI Kernel defines compliance in writing, so the language is retargetable and the implementation is replaceable.

Capability versus safety: the constitutional layer is a separate, unlearnable final gate, never merged into the capability it guards.

Substrate versus instance: the platform is the reusable substrate; a particular mind, such as Mentova, is one configured, developed instance of it.


PART ARCH-1 - THE COMPONENT MODEL


SECTION 1.1 - THE THREE REPOSITORIES

The platform lives in three repositories (Specification Part 15).

Repository prologai - the platform monorepo, holding packs/ (one or more SWI-Prolog packs per capability), launcher/ (the prologai executable), syntax/ (the .pai transcompiler), tests/ (the five test levels), and docs/.

Repository prologai-core - the Rust crate holding the native-speed components behind their interfaces: the vector backend bridge, the pluggable backends, journaling, cryptographic verification, the embedding provider, the vector-symbolic algebras, and the optional symbolic regressor.

Repository prologai-dist - the distribution, bundling a released SWI-Prolog, the compiled prologai-core library, and all packs into one prologai executable per platform, with the NOTICE file and all upstream license notices.


SECTION 1.2 - THE LAUNCHER AND THE LANGUAGE SURFACE

The launcher (PR 1) is the single entry point: it preloads every pack, installs the bootstrap ontology, prints the banner with the SWI-Prolog attribution line, sets the prologai_version and dialect flags, and starts the top level.

The syntax component (PR 1) is the language surface: term_expansion and goal_expansion rules plus operator declarations that transcompile .pai source into Layer C library calls at load time, leaving .pl files untouched.

These two components give PrologAI its identity without a fork, exactly as ADR-1 requires.


SECTION 1.3 - THE PACK CATALOGUE

Layer C is decomposed into packs, each a cohesive unit owning a slice of the architecture and exposing a small public interface.

Foundation packs.

pack lattice - the Lattice, node_facts, value channels, grounding, scopes and zones, and incremental tabling truth maintenance (PRs 3, 4, 5, 15).

pack actors - cyclic_actor and receptor with their management predicates (PRs 6, 7).

pack pubsub - subscribe and publish, the stigmergic coordination bus (PR 8).

pack sentinels - neuro-symbolic opportunistic forward-chaining rules (PR 9).

pack mindbody - the herald protocol, body enrollment, percept relay, and command dispatch (PR 10).

pack sona - the continuous learning layer: ReasoningBank, EWC++ protection, consolidation, and pattern separation (PR 11).

pack assessment - the Bayley, Piagetian, and CHC frameworks plus consciousness-indicator coverage (PR 12).

pack reflection - the reflection actor roster, the coordination specialist roster, the unified gating actor, and the impasse actor (PR 13).

Belief and cognition packs.

pack beliefs - budgets, evidence truth, anytime answering, and per-node_fact belief records with propagators (PRs 21, 29).

pack workspace - the global broadcast workspace cycle, the heartbeat of cognition (PR 18).

pack prediction - the predictive hierarchy and self-synthesized model population with the competence lifecycle (PRs 19, 22).

pack affect - somatic markers, staged appraisal, and the modulator bus (PRs 24, 26, 27).

pack development - spinoff learning, curiosity, steered daydreaming, the detector suite, and the language faculties (PRs 20, 23, 25, 30, 31, 28).

Substrate and reasoning packs.

pack attention_economy - the two-currency attention banker with wages, rent, spreading, and economic forgetting (PR 32).

pack types - gradual Lattice types with the off-by-default checker (PR 33).

pack chainer - the generic forward-and-backward chainer with meta-reasoning control knowledge (PR 34).

pack kernel - lattice-resident rewrite rules and the Minimal PrologAI Kernel interpreter and trace (PR 35).

Frontier packs.

pack induction - clause induction by inductive logic programming, learn-from-failures, and meta-interpretive learning (PR 37).

pack vsa - compositional vector binding, bound to the algebras in prologai-core (PR 38).

pack frames - reference frames and voting consensus (PR 39).

pack defeasible - justified defeasible reasoning over s(CASP) (PR 40).

pack probabilistic - the distribution-semantics probabilistic layer (PR 41).

pack attention_schema - the model of the mind's own attention and the formal workspace correspondence (PR 42).

Interoperability packs.

pack mcp_gateway - the Model Context Protocol endpoint exposing PrologAI services (PR 14).

pack interop - the Hyperon bridge and the Agent2Agent protocol with durable agent mail (PRs 36, 43).

Utility library packs.

pack libraries - record, similarity, types, collections, generators, tasks, problems, config, peers, macros, and convenience (Specification Part 18), with the nondeterminism conveniences pai_superpose and pai_collapse.


SECTION 1.4 - THE RUST CRATE MODULES

The prologai-core crate holds only what must run at native speed or below the logic layer.

Module vector_bridge - the six-predicate backend-agnostic interface and its routing to the configured backend.

Module backends - RuVector and at least one established baseline (hnswlib, usearch, or FAISS), each implementing the bridge trait.

Module journaling - write-ahead journaling for Lattice persistence.

Module verification - SHA-256 nexus hashing and the optional per-operation cryptographic proofs.

Module embedding - the pluggable embedding provider (hash projection, local model, external service) with the re-embedding maintenance routine.

Module vsa - the vector-symbolic algebras (Holographic Reduced Representations and one alternative) for PR 38.

Module regressor - the optional symbolic regressor (a Kolmogorov-Arnold Network or equivalent) for PR 22.

These modules compile to one shared library loaded with use_foreign_library, using the swipl-rs bridge.


SECTION 1.5 - ENGINE-RESIDENT PACKAGES USED, NOT BUILT

Three reasoning capabilities ride packages already available to the engine, wrapped by PrologAI packs rather than reimplemented.

Constraint Handling Rules back the sentinel-engine and attention-banker prototypes (the bake-off discipline of PRs 9 and 32).

s(CASP) backs justified defeasible reasoning (PR 40).

A ProbLog or PITA-style capability backs the probabilistic layer (PR 41).

Wrapping rather than rebuilding is the same leverage discipline as ADR-1, applied one level up.


PART ARCH-2 - THE DEPENDENCY AND BUILD GRAPH


SECTION 2.1 - THE FOUNDATION CHAIN

The build order follows the dependency edges declared by each work package.

prologai-core and the launcher come first, because nothing runs without the bridge and the entry point.

Then pack lattice (needs the bridge), then node_facts and scopes and zones within it, then the actor and coordination packs (actors, then pubsub, then sentinels), then mindbody, then sona, then assessment, then reflection.

This is the foundation chain: the sixteen-pack core that corresponds to the original buildable platform.


SECTION 2.2 - THE COGNITIVE AND SUBSTRATE TIERS

On the foundation, the belief substrate is built first among the cognitive packs: beliefs (budgets and evidence) underpins everything that scores or chooses.

Then workspace and prediction (the cycle and the predictive hierarchy), then affect and the curiosity and drive parts of development, then the rest of development (learning, steering, perception, language).

The substrate and reasoning packs layer next: attention_economy, types, then kernel, then chainer (which depends on the kernel's rewrite rules).

The frontier packs layer last: induction, vsa, frames, defeasible, probabilistic, and attention_schema, each depending on the cognitive and reasoning packs named in its specification dependencies.


SECTION 2.3 - THE INTEROPERABILITY EDGE

The interoperability packs sit at the outer edge, because they expose the assembled mind rather than constitute it.

pack mcp_gateway depends on all service packs (it wraps them).

pack interop depends on mcp_gateway, the kernel and value channels (for the Hyperon mapping), and mindbody and quarantine scopes (for inbound tasks and mail).

INVARIANT: dependencies point inward and downward only; no foundation pack depends on a cognitive, frontier, or interoperability pack, so the core can be built and tested before anything above it exists.


PART ARCH-3 - DATA ARCHITECTURE


SECTION 3.1 - THE LATTICE AS A DUAL-INDEX STORE

The Lattice is one logical store presented through two physical indexes per nexus, kept in step.

The logical index is the engine's clause database, giving exact unification, backtracking, and incremental tabling.

The vector index is the Layer B backend, giving sub-millisecond semantic similarity over embeddings.

Every write goes to both indexes inside one transaction; every read of relevance combines them, scoring a result as logical match weighted most, semantic similarity next, and activation recency least.

A nexus is addressed as locus://HOST/PATH, so the same interface names an in-memory store today and a distributed store later without surface change.


SECTION 3.2 - THE NODE_FACT AND ITS CHANNELS

The node_fact is the atomic unit: an immutable identity of Id, Relation, Arguments, and Referents, with a sign on the Id distinguishing a thesis from its antithesis dyad.

Around that immutable identity hang named value channels - belief, attention, emotion, provenance, and any user-defined channel - which mutate freely per scope and per zone.

This is the architecture's central data decision: identity is stable enough for tabling dependencies to track, while annotation is fluid enough to carry learning, attention, and affect.

Structural change is therefore never an in-place edit; it is accommodation, a provenance-stamped copy, with consolidation as the only force that destroys.


SECTION 3.3 - SCOPES, ZONES, AND THE SHAPE OF CONTEXT

Scopes partition the Lattice into locally coherent microtheories, addressable and sealable.

Zones partition it by epistemic status across nine kinds (present, possible, past, desired, expected, imagined, recalled, attained, confirmed), each with a declared activation flow direction.

The architectural rule is strict: activation never crosses a zone boundary except through an explicit scope_merge or a deliberate inscription, which keeps simulation, memory, desire, and observation from contaminating one another.

Reference frames (PR 39) are a scope subtype that adds coordinates, so spatial and conceptual knowledge reuse the same compartment machinery.


SECTION 3.4 - PERSISTENCE AND INTEGRITY

Persistence is write-ahead journaling in prologai-core: each operation is appended to the journal before it is applied, so a crash leaves a replayable log.

Integrity is a SHA-256 hash over identifiers and relation data, checkable with lattice_verify, with optional per-operation cryptographic proofs for safety-critical deployments.

Truth maintenance is delegated to the engine: derived relations are incrementally tabled, so a base-fact change invalidates and lazily recomputes only the affected derivations, and any derivation that goes undefined surfaces as a contradiction node_fact for compensation and coping.


PART ARCH-4 - CONCURRENCY AND COORDINATION ARCHITECTURE


SECTION 4.1 - THE ACTOR SUBSTRATE

Concurrency is the actor model over engine threads and message queues.

A cyclic_actor is a proactive thread that wakes on a schedule, runs a goal, recovers from its own errors, and repeats.

A receptor is a reactive thread that drains a named message queue through a handler.

A sentinel is an opportunistic rule that fires when a Lattice pattern appears, by exact unification or by semantic similarity.

Engines (the engine's lightweight coroutines) are available for massive many-agent fan-out where threads would be too heavy.


SECTION 4.2 - STIGMERGIC COORDINATION

Actors do not call each other directly; they coordinate through the Lattice, stigmergically.

Writes publish change events; interested actors subscribe to patterns and are invoked asynchronously, so the writing operation returns immediately.

Cognitive synergy is an architectural requirement, not an accident: every learning and reasoning actor exposes its intermediate state as node_facts, so a stuck process can be picked up and helped by another within one cycle.

This indirection is what lets the pack set grow by registration rather than by rewiring: a new specialist subscribes to its trigger and writes its result, and no existing actor changes.


SECTION 4.3 - RESOURCE GOVERNANCE

Two governors keep concurrency bounded and fair.

Budgets (PR 21) govern tasks: priority, durability, and quality drive an economic scheduler rather than an exhaustive sweep, with starvation detection lifting long-waiting work.

The attention economy (PR 32) governs knowledge: short-term and long-term importance, fed by wages and drained by rent, decide what occupies processor attention and what is forgotten, within a capped money supply that makes runaway feedback impossible.

The banker converts between the two at a configured policy, so tasks and knowledge compete on one reconciled currency.


PART ARCH-5 - RUNTIME CONTROL FLOW


SECTION 5.1 - THE COGNITIVE CYCLE AS HEARTBEAT

The workspace cycle (PR 18) is the heartbeat of a running mind, turning at a steady cadence.

On each beat: candidate coalitions form from live node_facts; each is scored by novelty, goal relevance, and affect, minus habituation; exactly one winner is broadcast on the workspace channel and kindled across zones; and learning attaches to that broadcast.

Two buffers feed the cycle - internal_experience and overall_experience - so self-generated and externally driven cognition stay distinguishable.

Everything else in the runtime hangs off this loop as a producer of candidates or a consumer of broadcasts.


SECTION 5.2 - THE PERCEIVE-DELIBERATE-ACT PATH

A percept enters through the herald, is routed by relay_percept to the perceiver, the motivation actor, or the regulation actor by signal type, and becomes node_facts.

The detector suite and the predictive hierarchy turn raw percepts into features and errors, forwarding only surprise upward.

Motivation turns homeostatic deltas and the curiosity drive into objectives; deliberation finds or composes a causal_plan, consulting markers, beliefs, and the chainer, under a budget that forces commitment.

Action dispatches through the body with proprioceptive confirmation, and regulation classifies the outcome and feeds learning.

Two speeds run in parallel: a fast reflex-and-intuition path and a slow solver path, arbitrated, with reflexes able to pre-empt when urgent.


SECTION 5.3 - THE REFLECTIVE AND SELF-IMPROVEMENT LOOP

Reflection actors run continuously alongside the cycle: regulation, compensation, coping, exploration, discovery, imitation, play, and meta-control, plus the unified gating actor and the impasse actor.

The impasse actor turns dead ends into subgoals, so the mind manufactures its own developmental material.

The refinement loop (PR 17) closes the largest circle: it diagnoses a recent trajectory window, proposes constitutional, sandboxed edits to the mind's own configuration, archives superseded variants with fitness and novelty, and rolls back on regression.

The meta-control actor also maintains the attention schema (PR 42), a model of the cycle's own dynamics used to pre-empt capture and explain attention history.


SECTION 5.4 - THE LIFECYCLE OF A COMPETENCE

Knowledge is not merely stored; it advances through a lifecycle (PR 22): feature, then subtask, then option, then model, then plan.

Synthesis and consolidation push competences forward through these stages, while meta-control retires low-utility elements, so the abstraction inventory is continually replaced rather than only accumulated.

This lifecycle is the architectural expression of Piagetian development: the same loop, run long enough, grows the mind.


PART ARCH-6 - INTERFACE CONTRACTS


SECTION 6.1 - THE VECTOR BACKEND CONTRACT

Layer B is reachable only through six predicates: index create, insert, search, delete, update weights, and close.

Any backend that implements this contract is admissible; the default is the bake-off winner, and the losers remain as fallbacks behind the identical contract.

This single narrow seam is what keeps an unverified dependency swappable forever.


SECTION 6.2 - THE EMBEDDING CONTRACT

The embedding provider is one function: given text or a term, return a vector of the provider's declared dimension.

A nexus records which provider and dimension produced its vectors; mixing dimensions in one index is an error; a maintenance routine re-embeds a nexus under a new provider in journaled batches.

The default is local, so the mind's experience stays with the mind unless an operator opts out.


SECTION 6.3 - THE MIND-BODY CONTRACT

The herald is the device-side client; the contract is three signal types inbound (perception, interoceptive, proprioceptive) and a command stream outbound, with every command resolving to a result or a synthesized timeout.

The body declares needs and capabilities at enrollment; the architecture is device-agnostic, so a simulator and a physical robot present the same contract.


SECTION 6.4 - THE KERNEL AS A SPECIFICATION INTERFACE

The Minimal PrologAI Kernel is an interface of a different kind: it is the written contract that defines a compliant implementation.

Any implementation whose observable evaluations are derivable from the kernel transition rules is compliant, which makes the whole platform retargetable to engines other than SWI-Prolog if it ever outgrows its host.


SECTION 6.5 - THE OUTWARD CONTRACTS

The MCP Gateway exposes PrologAI services as authenticated Model Context Protocol tool endpoints.

The interoperability pack adds two more outward contracts: the Atomese mapping for exchanging knowledge with Hyperon systems, and the Agent2Agent agent card plus task lifecycle for collaborating with opaque agents, both of which disclose capabilities and results while never exposing Lattice contents.

Every inbound artifact lands first in a quarantine scope, so the outside world can request and receive but never directly read or write the mind's memory.


PART ARCH-7 - SAFETY ARCHITECTURE


SECTION 7.1 - THE CONSTITUTIONAL GATE

The constitutional layer (Specification Section 3.16) is the architecture's final, unlearnable gate.

It sits ABOVE the unified gating actor, never merged into it: the gating actor learns which actions to favor, while the constitutional check decides whether a favored action is permitted at all.

It is called automatically before every irreversible step - command dispatch, modification commit, and modification proposal - and it cannot be bypassed or modified by the self-improvement primitives.

Vetoes carry reasons, and where defeasible reasoning is involved they carry justification trees, so the planner can synthesize a compliant alternative rather than merely stalling.


SECTION 7.2 - THE PROTECTED CORE

Certain components are excluded from all self-modification: the engine, the constitutional layer, the monitor, the refiner's own pipeline, and all bootstrap relations.

This protected core is enforced twice over: structurally, because the engine is unforked and unreachable from PrologAI-layer code; and procedurally, because the refinement loop refuses CRUD edits to protected components.

Removing the constitution is therefore not an accident waiting to happen but a deliberate, visible act against the protected-core discipline.


SECTION 7.3 - THE SANDBOX PIPELINE

Every self-modification flows through one pipeline: propose with justification, sandbox-evaluate in a possible_zone scope with no write access to present_zone and no external reach, run the constitutional check, then commit or reject, with rollback on post-commit regression.

Below the developmental stage of formal operations, every commit additionally requires human authorization, so a young mind cannot rewrite itself unsupervised.

Corrigibility is wired in at the same level: a signed shutdown executes within seconds and cannot be blocked or delayed by the mind's own reasoning.


SECTION 7.4 - AUDITABILITY AS STRUCTURE

Auditability is not a feature bolted on; it is a structural property.

Every relation is inspectable as a table or a graph, every derived structure carries provenance, modification and veto logs are immutable, and resource quotas are enforced per actor.

Because mechanisms are node_facts, a human can read the mind's competences, rules, and reasoning the same way the mind reads its own.


PART ARCH-8 - DEPLOYMENT ARCHITECTURE


SECTION 8.1 - THE SINGLE-HOST DEFAULT

The default deployment is a single host: one prologai executable bundling the engine, the compiled crate, and all packs, with the Lattice resident in memory and journaled to disk.

This is the form in which a mind such as Mentova is first grown and assessed, with the Community Garden Project as its proving ground.


SECTION 8.2 - THE MIND SERVER AND ITS TOPOLOGIES

The Mind Server (Specification Part 14) is the multi-mind deployment, implemented as part of the MCP Gateway: it registers devices, provisions mind instances, and persists each mind's profile between sessions.

It supports four topologies: individual (one mind, one body), unity (one mind, many bodies), joint (many minds, one body), and hive (many minds, many bodies).

Durable agent mail (PR 43) is hosted here, so minds in different sessions exchange messages asynchronously through quarantine scopes.


SECTION 8.3 - THE DISTRIBUTED-STORE ROADMAP

Because the Lattice interface is address-based, a distributed backend can be slotted beneath it later without any surface change, exactly as the design disciplines anticipate.

This is roadmap, not version one: single host first, the distributed store layered in when scale demands it, and the engine-level escalation held in reserve behind its written trigger.


PART ARCH-9 - CROSS-CUTTING CONCERNS

Provenance threads through every pack: nothing derived is stored without a record of how it arose.

Truth maintenance threads through every derived relation: the engine's incremental tabling keeps inferences consistent with a changing world without bookkeeping code.

Attention and budget thread through every actor: nothing runs unmetered, and nothing occupies memory without paying rent.

Observability threads through meta-control: usefulness, throughput, queue depth, error counts, and indicator coverage are sampled at adaptive rates that never starve the system they measure.

Safety threads through every irreversible action: the constitutional gate is the one cross-cutting concern that may never be optimized away.


PART ARCH-10 - BUILD AND INTEGRATION ORDER

The integration order composes the dependency graph with the specification's implementation priorities.

Stage 1 - stand up prologai-core (the bridge and the bake-off) and the launcher, then the utility library packs that everything uses.

Stage 2 - build the foundation chain: lattice, then actors, pubsub, sentinels, then mindbody, sona, assessment, reflection.

Stage 3 - build the belief substrate (beliefs) and then the workspace and prediction packs, then affect and development.

Stage 4 - build the reasoning substrate: attention_economy, types, kernel, chainer.

Stage 5 - build the frontier packs: induction, vsa, frames, defeasible, probabilistic, attention_schema.

Stage 6 - build the outward edge: mcp_gateway, then interop.

Each stage is gated by the test levels: unit tests per actor, integration tests per design pattern, scenario tests on the Community Garden Project, longitudinal development tests against the Piagetian battery, and the architecture audit (the loops audit and the epistemology audit) run during refinement.

PART ARCH-12 - SAFETY AND INTERPRETABILITY AT SCALE

This part places the safety mechanisms of Specification Part 38 into the architecture; they are cross-cutting and privileged, and they sit outside the pack graph that the mind can edit.


SECTION 12.1 - THE PRIVILEGED SAFETY PLANE

Safety is a privileged plane above the capability layers, not a pack among them.

It holds the constitutional layer (Section 3.16), the oversight monitor, the capability-evaluation gate, and the tripwire detectors, none of which the self-improvement primitives can reach or rewrite.

The ordering at run time is fixed: the learned gating actor proposes, the constitutional gate disposes, the oversight monitor watches and samples for human review, and the tripwires can pause everything; an action passes only when all of them allow it.

This plane is enforced structurally by the layered design: it lives where PrologAI-layer self-modification cannot reach, the same boundary that makes the engine immutable.


SECTION 12.2 - THE GRADUATED-AUTONOMY STATE MACHINE

Autonomy is a state machine, not a flag.

A mind moves through states - simulation-only, human-authorized, narrowly self-authorizing, broadly self-authorizing - and each transition upward is gated by passing the capability evaluations and the audits, while any regression, tripwire, or failed evaluation transitions it downward automatically.

The developmental stage sets the ceiling (no self-authorization below formal operations), and the trust budget sets the current position beneath that ceiling.

Corrigibility is invariant across all states: the signed-shutdown path is wired directly to the halt, bypassing the cognitive cycle entirely so the mind cannot delay it.


SECTION 12.3 - THE AUDITABILITY BOUNDARY

The architecture has a clean, honestly drawn line between what is read and what is bounded.

Above the line, the symbolic Lattice - node_facts, scopes, rules, plans, justifications, and provenance - is fully inspectable by a human.

Below the line, the sub-symbolic components - embeddings, similarity routing, elastic-weight protection, and the vector backend - are governed by provenance, by the structural vector-symbolic embedding option (PR 38) that keeps composition inspectable, by the symbolic justification layer that explains conclusions, and by deterministic replay, rather than by direct transparency.

No safety mechanism in this architecture depends on the sub-symbolic layer being transparent; closing that gap is a named research direction, not a load-bearing assumption.


SECTION 12.4 - CONTAINMENT AND MULTI-MIND SAFETY

Containment is architectural: early minds run in the simulated Community Garden with no real-world actuation, restricted tool and network access, and all external traffic quarantined, and embodiment is unlocked only stage by stage.

In the Mind Server's joint and hive topologies, each mind keeps its own Lattice, its own constitution instance, and its own overseers; all inter-mind traffic passes through quarantine scopes; and the capped attention economies make runaway cross-mind feedback structurally impossible.

The deterministic replay facility is the architecture's debugging anchor for a self-modifying system, with the honest caveat that exact replay under heavy actor concurrency requires recording the scheduling order, which the journal captures for the safety-critical paths.


PART ARCH-13 - THE TOOL-USE AND COMPUTER-USE ARCHITECTURE

This part places the tool-use faculty (PR 44) and the computer-use body (PR 45) into the architecture; they are the layer by which the mind reaches the wider world of tools and applications.


SECTION 13.1 - TWO NEW PACKS

pack tooling realizes PR 44: it holds the tool registry, the tool_card schema, affordance discovery, the selection step that plugs into deliberation, the gated invocation path, and the reliability learning.

pack computer_use realizes PR 45: it enrolls a screen-and-input body, routes screenshots into the detector suite, and issues mouse, keyboard, and browser actions through the mind-body interface, with browser control as its structured sub-mode.

Both sit at the outer edge of the dependency graph, like the interoperability packs, because they connect the assembled mind to the outside rather than constitute it; pack tooling depends on the mind-body, gateway, budget, marker, and chainer packs, and pack computer_use depends on the mind-body, detector, frames, and tooling packs.


SECTION 13.2 - TOOLS ARE GROUNDED NODE_FACTS

The architecture adds no new knowledge primitive for tools, because a tool is a grounded node_fact (Section 3.1) carrying a tool_card.

A tool is therefore discovered, scored, invoked, learned about, and even built using the same machinery as any other competence, and the six tool sources - native foreign predicates, Model Context Protocol servers (PR 14), Agent2Agent skills (PR 43), body capabilities (PR 10), Agent Communication Protocol agents (PR 47), and Agent Network Protocol peers (PR 48) - are unified behind one card and one registry.

The catalogue of Specification Part 39 is the bootstrap population of that registry, each class tagged with a risk class.


SECTION 13.3 - COMPUTER USE IS A BODY, AND TOOLS PASS THE SAFETY PLANE

Computer use introduces no special control path: the screen is a sensor and the input devices are actuators on an enrolled body, so the perceive-deliberate-act loop already specified carries it, with the detector suite reading the screen and reference frames giving the screen its coordinate space.

Every tool call and every irreversible on-screen action passes the same privileged safety plane as any other action (Part ARCH-12): the constitutional gate disposes, the oversight monitor samples by risk class, and the tripwires can pause; high-risk tool classes are additionally gated by capability evaluation and staged autonomy.

Tool results and on-screen content from untrusted sources land in quarantine scopes, so the outside world can be read and acted upon without its content being trusted as instruction - the architectural defense against prompt injection through tools and the screen.


PART ARCH-14 - THE ROBOT BODY AND THE ROS 2 BRIDGE

This part places PR 46, the Robot Operating System bridge, into the architecture; it introduces no new cognitive layer, because a robot is an instance of the Mind-Body interface (Layer 9), not a new faculty.

pack ros_bridge realizes PR 46: a small Robot Operating System version 2 (ROS 2) node, written with rclpy, that subscribes to the robot's sensor topics and relays them to the mind as percepts, and turns the mind's commands into actuator messages; it is the robot's herald, and it depends on the mindbody, development (detector suite), frames, and tooling packs.

Sensor messages map to percepts through the detector suite (PR 30), and the TF2 transform tree maps onto reference frames (PR 39), so the robot's space reuses the frame machinery rather than a parallel one.

The navigation and manipulation stacks (Navigation 2 and MoveIt) are mounted as grounded tools (PR 44), not reimplemented, and a simulator (Gazebo, NVIDIA Isaac Sim, or Webots) provides the virtual robot body, with the same bridge serving a physical robot unchanged.

Embodiment sits under the same privileged safety plane as everything else (Part ARCH-12): the constitutional gate and the oversight monitor screen every irreversible physical action, and staged autonomy keeps a young mind in simulation until capability evaluation permits the real world.


PART ARCH-15 - THE ACP GATEWAY

This part places PR 47, the Agent Communication Protocol gateway, into the architecture; it sits at the outer edge of the dependency graph alongside the MCP gateway (Layer 13) and the A2A gateway (Layer 27), because it connects the assembled mind to the enterprise REST client ecosystem rather than constituting a new cognitive faculty.

pack acp realizes PR 47: an HTTP listener that accepts POST /runs (mapped to objective node_facts routed through motivation and the constitutional gate), serves GET /runs/{run_id} status, and publishes the ACP agent description at GET /.well-known/agent.json; it depends on the mcp_gateway pack (for the shared HTTP infrastructure and the constitutional gate) and the tooling pack (PR 44) for outbound task invocation.

The ACP gateway registers its hosted agents as a sixth tool source in the tool registry (PR 44); from the perspective of the tool-use faculty, an ACP-hosted agent is a grounded node_fact with an acp binding, indistinguishable in discovery and invocation from any other tool type.

All ACP task content from untrusted sources lands in quarantine scopes before any content is trusted or promoted, exactly as A2A artifacts and computer-use screen content do, providing the architectural defense against prompt injection through the ACP surface.


PART ARCH-16 - THE ANP GATEWAY

This part places PR 48, the Agent Network Protocol gateway, into the architecture; it is the only pack that provides the mind with a decentralized cryptographic identity (a W3C DID) rather than depending on a central registry or endpoint directory.

pack anp realizes PR 48: a DID lifecycle manager that derives a did:web DID from the Mind Server hostname and stores the DID and Ed25519 key pair as constitution-scope node_facts; an HTTP endpoint at GET /.well-known/agent-descriptions serving the ANP agent description; an inbound message verifier that resolves the sender's DID document and checks the signature before admitting any content to any scope; an outbound message signer that gates every message through the constitutional check before signing and transmitting; and a meta-protocol negotiation handler that returns the set {mcp, a2a, acp, anp} with their endpoint URLs when a peer queries supported protocols.

pack anp depends on the mcp_gateway pack (for the shared HTTP infrastructure), the lattice pack (for constitution-scope storage of the DID and key material), and the acp pack (PR 47) for the ACP endpoint URL returned in meta-protocol negotiation.

The ANP gateway registers ANP-addressed peers as a fifth tool source in the tool registry (PR 44); from the perspective of the tool-use faculty, an ANP peer is a grounded node_fact with an anp binding; the signature verification and the constitutional gate are transparent to the selection and invocation steps.

Inbound ANP message content that fails signature verification is discarded unconditionally, and a security event is written to the oversight log; this is the architectural guarantee that the decentralized trust model of ANP does not weaken the privileged safety plane (Part ARCH-12).


PART ARCH-17 - THE CONSOLE REPL LAUNCHER

This part places the console REPL launcher (Specification Section 41.1) into the architecture. It is a Mentova-level entry point, not a PrologAI core pack, so it lives in the Mentova repository rather than in the packs/ tree of PrologAI.

Location: Mentova repository, bin/ directory. Two files: bin/mentova_repl.pl (the Prolog wrapper) and bin/mentova (the bash shell launcher). The bin/ directory is not a SWI-Prolog pack and carries no pack.pl manifest; it is a deployment artifact, not a library.

Dependency graph: bin/mentova_repl.pl depends on src/mentova/mentova.pl (the full Mentova bootstrap, which in turn depends on all 48 PrologAI packs), and on library(assessment) and library(node_facts) from the PrologAI assessment pack (packs/assessment/prolog/). It has no other dependencies. It introduces no new predicates and no new runtime state; it is a boot file, not a module.

Interface contract: the launcher pre-loads the environment and hands control to SWI-Prolog's native prolog/0 toplevel. The interface to the user is the ?- prompt. All platform predicates accessible at that prompt are unchanged from their published module interfaces; the launcher adds no new syntax and no new commands.

Relationship to the safety plane: the launcher calls mentova_boot, which runs the constitutional check and overseer registration. All safety guarantees established at boot time (constitutional gate, oversight monitor, scope enforcement) are therefore in effect for every query issued at the ?- prompt. The launcher does not bypass or weaken the privileged safety plane (ARCH-12).

Future extension: a friendlier command interface (:help, :query, :assess short-form commands) may be added as a custom read-eval-print loop that sits between the user and SWI-Prolog's prolog/0. The thin wrapper (bin/mentova_repl.pl) is the foundation for that extension; it may be loaded by the extended interface without modification.

This volume completes SPARC Step 3: it places all forty-eight work packages into repositories, packs, and crate modules; binds them through narrow interface contracts to the engine and the vector backend; and defines how they communicate at run time around the cognitive cycle, under one cross-cutting safety gate.

PART ARCH-18 - THE LATTICE CRYPTOGRAPHIC PRIVACY LAYER

This part places PR 49, the Lattice Cryptographic Privacy Layer, into the architecture. It sits as a cross-cutting service pack at the outer edge of the dependency graph: any pack, application, or interoperability gateway may load it to encrypt or decrypt sensitive values, but it does not constitute a cognitive faculty and has no dependency on the Lattice internals or the constitutional layer.


SECTION 18.1 - ONE PACK, ONE MODULE

pack lattice_crypto realizes PR 49: a single SWI-Prolog module (lattice_crypto) in packs/lattice_crypto/prolog/lattice_crypto.pl, loaded from the standard library path. The module exports key generation, encryption, decryption, availability probe, and PEM loading predicates. It has no foreign (C/Rust) component; all operations use the SWI-Prolog library(crypto) built-in and the openssl command-line tool (required on the PATH).

The pack depends only on SWI-Prolog library(crypto), library(process), library(lists), and library(apply). It has no dependency on any PrologAI pack and can therefore be loaded first in the dependency graph, before the Lattice, before actors, and before any other PrologAI component.


SECTION 18.2 - PLACEMENT IN THE LAYER DIAGRAM

pack lattice_crypto occupies the same deployment position as the interoperability packs (mcp_gateway, anp) at Layer 13 in the dependency diagram: outer shell, no inward cognitive dependency. Its upward boundary is the application or interoperability pack that calls it; its downward boundary is the SWI-Prolog crypto engine and the host operating system's OpenSSL installation.

The reason for this placement: private data can originate anywhere in the stack (from the user input surface, from an inter-agent payload arriving at the ANP gateway, or from the tool-use faculty reading a credential from disk) and must be encrytable and decryptable at any of those entry points. A cross-cutting service pack at the outer shell is the only placement that serves all callers without introducing circular dependencies.


SECTION 18.3 - KEY MATERIAL AND THE CONSTITUTIONAL BOUNDARY

Long-term private keys (RSA or ECDH private keys belonging to the mind itself) are not stored inside the Lattice data store. They reside in the host file system under operator control or in a hardware security module. The pack lattice_crypto loads them on demand from PEM files using pai_pem_load_private/2 and pai_pem_load_public/2 and holds them in Prolog variables for the duration of the call.

Short-lived ephemeral ECDH private scalars exist only in Prolog variables; they are never written to disk or to the Lattice. The shared secret and the derived AES key are created and consumed within a single pai_encrypt/5 or pai_decrypt/5 call.

Encrypted bundles (rsa_bundle/3, ecdh_bundle/4, pqc_bundle/4) are storable as ordinary Prolog terms in the Lattice or in the journal. The plaintext they protect is visible only to a caller that holds the matching private key.

The constitutional gate and the privileged safety plane (ARCH-12) are not bypassed by the crypto layer. Predicates that store encrypted data into the Lattice still pass through the standard scope and assertion predicates; the crypto layer does not introduce an unaudited back channel into the Lattice.


SECTION 18.4 - ALGORITHM SELECTION AND MIGRATION

RSA is the entry point: it requires only a standard OpenSSL installation (already required by the ANP pack for DID-based signature verification) and is supported out of the box by library(crypto) in SWI-Prolog.

ECDH on prime256v1 is the recommended choice for new deployments: smaller keys, forward secrecy (because ephemeral key pairs are discarded after each encrypt), and no dependency beyond the SWI-Prolog crypto built-in.

PQC (ML-KEM-768) is the recommended choice for deployments that must remain secure against a future cryptographically relevant quantum computer. It requires OpenSSL 3.2 or later with the OQS provider; the pack probes for the provider at runtime and raises a structured error if it is absent, so the rest of the system degrades gracefully rather than silently.

Migrating from RSA to ECDH or PQC requires only changing the algorithm atom (pai_algo_rsa, pai_algo_ecdh, pai_algo_pqc) and re-encrypting existing bundles with the new algorithm. The bundle format is self-describing (the compound functor names the algorithm), so old and new bundles can coexist in the Lattice during a rolling migration.


PART ARCH-12 - IMAGINATIVE MEMORY AND SITUATIONAL AWARENESS (PR 50, PR 51)

pack imagination - the imaginative area of the Totality: mindscapes (compartmentalized canvases held only in non-observed realities), tableaux of bound elements (vantage, figure, motif), and rendered reveries that advance figures kinematically into deterministic, inspectable animations for mental simulation (PR 50).

pack awareness - situational awareness as a ladder of regards (ambient, selfward, otherward(Agent), avowed, disavowed), with one active standpoint at a time, theory-of-mind attribution and divergence detection, and reconciliation of the owned and disowned views into a single self-account (PR 51).

Both packs are cognitive-layer leaves: they depend only on the standard list library and assert their own dynamic state, so they add no new dependency to the foundation chain and load through the launcher's per-pack library path like every other pack. The imagination pack enforces the imaginative sandbox by refusing the observed reality, keeping imagined content provably separate from perceived fact; the awareness pack manages viewpoints in the same spirit as Lattice viewpoints, so a proposition may be believed under one regard and denied under another without global contradiction.

Build and integration order: Layer 31 (imaginative memory and situational awareness) - pack imagination, pack awareness - PRs 50 and 51 - extends the cognitive layer above the development and affect packs and may be built and tested independently of all packs above the foundation.

PART ARCH-19 - THE EPHEMERA PACK (PR 53)

pack ephemera - the code synthesis, execution, and skill persistence substrate: goal evaluation with timeout (ep_eval/3), Prolog code atom evaluation (ep_scratch/4), shell subprocess execution with output capture (ep_shell/3), language-specific script execution via temporary file (ep_ephemeral/4), synthesis-execute-check iteration cycles (ep_iterate/5), trace logging (ep_trace_record/4, ep_trace_get/2), and skill persistence (ep_skill_save/4, ep_skill_lookup/3, ep_skill_run/3, ep_skill_list/1, ep_skill_forget/1). Skills are stored in the dynamic fact ep_skill_db/4 and persist for the lifetime of the session. A skill is a named ephemeron promoted from ephemeral to reusable: it has a name, a language, source code, and a human-readable description for indexing and search.

Build and integration order: Layer 32 (ephemeral code synthesis and execution) - pack ephemera - PR 53 - sits above the imaginative memory and situational awareness layer (Layer 31) and below the agency orchestration layer (Layer 33). It may be loaded and tested independently of all packs that depend on it.

Dependency profile: pack ephemera depends only on SWI-Prolog built-ins (call_with_time_limit/2, atom_to_term/3, tmp_file_name/2) and standard libraries (library(process), library(readutil), library(lists)). It has no dependency on any PrologAI pack, making it a stable foundation for the agency and refinery packs above it.

Integration points: the agency pack (Layer 33) imports ep_eval/3 and ep_shell/3 from this pack to execute agentic actions. The refinery pack (Layer 34) may use ep_eval/3 to run evaluation goals within bounded time. Callers that wish to execute temporary programs in languages other than Prolog (Python 3, Bash) use ep_ephemeral/4 directly.

Safety properties: every subprocess invocation is bounded by a caller-supplied timeout. Temporary files are deleted unconditionally after the subprocess exits. The pack does not modify any global Prolog state except its own trace log and counter. All external code executes in a subprocess, isolated from the main Prolog process.

PART ARCH-20 - THE AGENCY PACK (PR 54)

pack agency - the agentic execution loop: formal loop lifecycle management (ag_loop_create/3, ag_loop_run/3, ag_loop_step/3), goal stack manipulation (ag_push_subgoal/2, ag_pop_subgoal/2), step budget accounting (ag_budget_remaining/2, ag_budget_decrement/1), loop detection (ag_detect_loop/2), escalation to human oversight (ag_escalate/2), and full step-by-step trace recording (ag_step_record/4, ag_loop_trace/2).

Build and integration order: Layer 33 (agentic execution loop) - pack agency - PR 54 - sits above the ephemera pack (Layer 32) and depends on it for action execution (ep_eval/3 and ep_shell/3). It may be loaded and tested independently of all packs above it.

Dependency profile: pack agency imports ep_eval/3 and ep_shell/3 from pack ephemera. It has no other PrologAI pack dependency. This means agency can be used anywhere ephemera is available.

The reasoning goal (ReasonGoal) interface: the caller supplies a Prolog goal that is called at each step as call(ReasonGoal, LoopId, CurrentGoal, Thought, Action). This interface is intentionally open: any reasoning strategy (planning, reflection, table lookup, neural oracle) can be plugged in as a reasoning goal. The agency pack is purely the loop infrastructure; domain intelligence lives in the ReasonGoal.

Integration points: the refinery pack (Layer 34) can use agency loops to drive iterative quality improvement cycles. The planning pack and synthesis pack may supply ReasonGoal implementations for specific problem domains. Callers in the Mentova cognitive layer can use ag_loop_create/3 and ag_loop_run/3 to pursue any goal with a bounded, observable, traceable loop.

Safety properties: every loop has a finite step budget; when the budget is exhausted, the loop halts with status budget_exhausted rather than running indefinitely. Loop detection (ag_detect_loop/2) allows the ReasonGoal to detect and break cycles. Escalation (ag_escalate/2) provides a safe shutdown path when human oversight is required. Every step is recorded in the trace, making loop execution fully auditable.

PART ARCH-21 - THE REFINERY PACK (PR 55)

pack refinery - the metacognitive quality layer: criterion-based critique (rn_critique/4), fraction scoring (rn_score/3), iterative improvement with an improver goal (rn_improve/5), full evaluator-optimizer loop with run logging (rn_optimize/5), multi-path exploration with score ranking (rn_explore_paths/4), and a persistent lesson database (rn_learn/3, rn_recall/2, rn_forget/1).

Build and integration order: Layer 34 (metacognitive quality layer) - pack refinery - PR 55 - sits above the agency pack (Layer 33). It has no dependency on any PrologAI pack; its criterion TestGoals are caller-supplied and may invoke any predicate available in the caller's environment.

Dependency profile: pack refinery depends only on SWI-Prolog built-ins (findall, call_with_time_limit, float, length) and standard libraries (library(lists), library(aggregate)). It does not depend on pack ephemera or pack agency, and may be loaded and used independently of those packs.

Integration points: the agency pack can use rn_optimize/5 as a ReasonGoal implementation to drive quality-improvement cycles as agentic actions. The planning pack may use rn_explore_paths/4 to explore multiple plan candidates and pick the highest-scoring one. Any PrologAI pack that produces outputs subject to quality checking can use rn_critique/4 and rn_score/3 directly.

The criteria design principle: criteria TestGoals should be written using YALL lambda notation so they carry their module context. This makes criteria portable across module boundaries and prevents the module-visibility errors that arise when using plain predicate atoms.

The lesson database principle: before attempting a complex problem of a known type, recall lessons from rn_learn/3; after a failure, record a lesson so future attempts benefit from the experience. This is the symbolic equivalent of the reflexion pattern from agentic AI research: the system accumulates written lessons from failures and applies them to future attempts.
