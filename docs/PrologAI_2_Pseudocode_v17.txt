This is the pseudocode companion to the PrologAI Platform Specification.

The specification (SPARC Step 1) said WHAT the platform must do; this volume (SPARC Step 2) says HOW, in English Readable Code, before any commitment to concrete programming syntax.

It opens with the ERC-PrologAI profile, then writes out the Minimal PrologAI Kernel rule set, then gives the readable logic of all forty-eight work packages in dependency order, from the launcher at the beginning through tool use and computer use to the ACP and ANP gateways at the end.

PART A - THE ERC-PROLOGAI PROFILE


SECTION A.1 - WHAT ENGLISH READABLE CODE IS

English Readable Code (ERC) is structured pseudocode: the logic of a program written in plain, step-by-step English, with just enough structure to be unambiguous, and no more.

ERC is the bridge between the specification's requirements and the eventual PrologAI source.

The Translation Guarantee: every ERC construct below maps mechanically onto one PrologAI primitive or library predicate, so that translating ERC into running .pai code is transcription, not reinvention.


SECTION A.2 - ERC NOTATION CONVENTIONS

DEFINE DATA Name: declares a structured record and its fields.

PROCEDURE Name(Inputs -> Outputs): declares a named procedure; its body is the indented steps beneath it.

STEP: one action, read top to bottom.

IF Condition THEN ... ELSE ...: conditional branching.

FOR EACH Item IN Collection: iteration.

WHILE Condition: bounded repetition (every loop declares a budget or a termination condition).

FOR EACH SOLUTION OF Goal: backtracking enumeration of all ways a logical goal succeeds.

RETURN Value: yields a result to the caller.

ASSERT Fact / RETRACT Fact: add or remove knowledge in the Lattice.

PUBLISH Event ON Channel / ON Event Pattern DO Handler: stigmergic coordination.

INVARIANT: a condition that must always hold.

GUARD: a safety condition checked before an action; failing it routes to the constitutional layer or a safe-idle behavior.

TEST Id: GIVEN ... WHEN ... THEN ...: an executable acceptance check, mirroring the specification's Given/When/Then.

A name written_like_this is a PrologAI relation, predicate, or value; a name WrittenLikeThis is an ERC variable or data type.


SECTION A.3 - LOGIC-PROGRAMMING PRIMITIVES (THE ENGINE'S GIFTS)

These primitives are inherited from the unforked SWI-Prolog engine and are assumed everywhere below.

UNIFY(TermA, TermB -> Bindings): match two terms, binding variables so they become identical, or fail.

PROVE(Goal -> Bindings): backward chaining; search rules and facts for a proof of Goal, yielding variable bindings.

FOR EACH SOLUTION OF Goal: backtracking; revisit Goal for every alternative proof.

FIND ALL X SUCH THAT Goal -> List: collect every solution (findall and bagof).

ONCE(Goal): commit to the first solution only (deterministic cut).

NEGATION AS FAILURE NOT Goal: succeeds when Goal has no proof.

ASSERT / RETRACT: modify the dynamic clause database, which the Lattice sits on.

TABLE(Goal): memoize Goal's results, recomputed incrementally when base facts change (PR 15).

SPAWN THREAD / SEND MESSAGE / RECEIVE MESSAGE: concurrency over engine threads and message queues (PRs 6, 7).

These are never reimplemented; ERC procedures call them directly.

PART B - THE MINIMAL PROLOGAI KERNEL (THE WRITTEN-OUT RULE SET)

This is the formal core of PR 35: the small operational semantics that defines what a compliant PrologAI implementation is.

An implementation is compliant if and only if its observable evaluations are derivable from the transition rules below.


SECTION B.1 - KERNEL STATE GRAMMAR

DEFINE DATA KernelState:
    Expression   - the term currently being evaluated
    Bindings     - the current variable substitution
    Context      - the active scope and zone
    Lattice      - the knowledge store (node_facts, rules, value channels)

An Expression is one of: an Atom, a Variable, a Compound(Head, Arguments), or a Grounded(Reference).

Structural congruence: two states are equal up to renaming of variables and reordering of independent (commutative) sub-expressions.

A transition is written: STATE ==[Label]==> STATE', meaning the left state rewrites to the right in one labeled step.


SECTION B.2 - THE TRANSITION RULES

Rule EVAL-STEP (evaluate one step):
    GIVEN Expression is Compound(Head, Args) and a rewrite rule rewrites_to(Pattern, Template, Guard) exists in Lattice
    WHEN UNIFY(Compound, Pattern -> B2) succeeds and Guard holds under (Bindings + B2)
    THEN <Compound, Bindings, Context, Lattice> ==[eval]==> <Template, Bindings + B2, Context, Lattice>

Rule EVAL-IN-SCOPE (evaluate in a named scope context, the evalc analogue):
    GIVEN an expression tagged to evaluate in Scope S
    WHEN S is opened and active
    THEN evaluation proceeds with Context set to (S, Zone), and all anchor_node and traverse_nexus steps are confined to S until the tagged expression returns.

Rule CHAIN (chain a result into a template):
    GIVEN Expression is chain(Inner, Var, Template)
    WHEN Inner evaluates (by EVAL-STEP, recursively) to Result
    THEN bind Var to Result and continue evaluation with Template.

Rule UNIFY-CONT (unify with success and failure continuations):
    GIVEN Expression is unify(A, B, OnSuccess, OnFailure)
    WHEN UNIFY(A, B -> B2) succeeds
    THEN continue with OnSuccess under (Bindings + B2)
    ELSE continue with OnFailure under Bindings.

Rule CONS (construct an expression):
    GIVEN Expression is cons_atom(Head, Tail)
    THEN ==[cons]==> the Compound formed by prepending Head to Tail.

Rule DECONS (deconstruct an expression):
    GIVEN Expression is decons_atom(Compound(Head, Tail))
    THEN ==[decons]==> the pair (Head, Tail), or fail if the expression is atomic.

Rule FUNCTION (enter a function frame):
    GIVEN Expression is function(Name, Args)
    THEN push a frame binding the parameters of Name to Args, and evaluate Name's body in that frame.

Rule RETURN (return from a function frame):
    GIVEN evaluation of a function body reaches return(Value)
    THEN pop the frame and continue the caller with Value.

Rule SUPERPOSE (offer nondeterministic alternatives):
    GIVEN Expression is superpose([A1, A2, ..., An])
    THEN the state branches: it ==[superpose]==> each <Ai, ...> as an alternative solution (this is engine backtracking).

Rule COLLAPSE (collect nondeterministic alternatives):
    GIVEN Expression is collapse(Inner)
    THEN gather every solution Inner produces under SUPERPOSE into one list expression (this is findall).

INVARIANT (compliance): for any .pai program, the set of observable results produced by an implementation MUST equal the set derivable by applying these rules; the reference realization maps eval/unify/superpose/function onto the engine's own evaluation, unification, backtracking, and call, so compliance costs no fork.

TEST KERNEL-FACTORIAL: GIVEN a rewrite rule set defining factorial in the Lattice, WHEN evaluating factorial(5), THEN the result is 120 AND the kernel trace uses only the rules above.

TEST KERNEL-AGREEMENT: GIVEN the same program on the reference realization and on a strict tiny interpreter built from these rules, THEN observable results agree.

PART C - PSEUDOCODE FOR THE WORK PACKAGES (PR 1 THROUGH PR 43)


PR 1 - LAUNCHER, DISTRIBUTION, AND DIALECT

PROCEDURE start_prologai( -> TopLevel):
    STEP load every PrologAI pack and install the bootstrap ontology into the Lattice.
    STEP print the PrologAI banner, then the embedded SWI-Prolog attribution line.
    STEP set flag prologai_version to '1.0.0' and keep flag dialect = swi.
    STEP enter the interactive top level.
    RETURN TopLevel.

PROCEDURE load_pai_file(Path):
    IF Path ends with ".pai" THEN expand each term through term_expansion and goal_expansion into library calls, then load.
    ELSE load as standard Prolog, untouched.

TEST AC-PR01-003: GIVEN a .pai file declaring one sentinel, WHEN it is loaded, THEN sentinel_list shows that sentinel registered.


PR 2 - THE RUST FFI BRIDGE AND THE VECTOR BACKEND BAKE-OFF

DEFINE DATA VectorBackend: an interface with create, insert, search, delete, update_weights, close.

PROCEDURE run_bakeoff(Backends, Sizes -> Winner):
    FOR EACH Backend IN Backends:
        FOR EACH N IN Sizes (10000, 100000, 1000000):
            STEP build an index of N synthetic node_facts.
            STEP measure insert throughput, search latency at the 50th and 99th percentiles, recall at K=20, memory footprint, and survival of a forced mid-insert crash.
    STEP write all measurements to docs/bakeoff_results.
    STEP Winner := the backend with the best weighted score.
    STEP set Winner as default; keep the others available behind the same interface.
    RETURN Winner.

INVARIANT: every Lattice operation calls the backend only through the six-predicate interface, so the backend stays swappable forever.

TEST AC-PR02-002: GIVEN both backends compiled, WHEN the bake-off runs, THEN a results file exists AND the default backend equals the winner.


PR 3 - THE LATTICE

PROCEDURE lattice_open(Address, Options -> Nexus):
    GUARD Address matches locus://HOST/PATH.
    STEP create a logical index (clause database) and a vector index (backend HNSW) for the nexus.
    STEP open the journal file for append.
    RETURN Nexus.

PROCEDURE apply_operation(Nexus, Op):
    STEP append Op to the journal BEFORE applying it (write-ahead).
    STEP apply Op to both indexes inside a transaction; IF it fails or throws THEN roll back all changes.

PROCEDURE lattice_verify(Nexus -> Ok):
    STEP compute SHA-256 over all node_fact identifiers and relation data; compare to the stored hash.

INVARIANT: traverse_nexus returns within 100 milliseconds for up to 100000 node_facts.


PR 4 - NODE FACTS

DEFINE DATA node_fact: Id (nonzero; positive = affirmative, negative = antithesis), Relation, Arguments, Referents, plus value channels.

PROCEDURE anchor_node(Relation, Args, Referents -> Id):
    STEP Id := a fresh unique identifier.
    STEP compute the vector embedding via the pluggable provider (PR 16).
    STEP store the node_fact in the logical and vector indexes; write the journal entry.
    STEP PUBLISH a change event.
    RETURN Id.

PROCEDURE traverse_nexus(Pattern, K -> Results):
    STEP Phase1 := logical unification matches for Pattern.
    STEP Phase2 := top K backend similarity matches.
    STEP score each match: Score = LogicalMatch*0.6 + SemanticSimilarity*0.4 + ActivationRecency*0.1.
    RETURN Results sorted by Score.

PROCEDURE kindle_node(Id): update activation timestamp; raise backend edge weight by 0.05; propagate +0.01 to direct neighbors.

PROCEDURE quench_node(Id): update timestamp as quenched; lower edge weight by 0.02; do NOT propagate.

TEST AC-PR04-001: GIVEN an open nexus, WHEN anchor_node(percept, [apple, red], [visible]) runs, THEN a unique nonzero Id is returned AND the node_fact is retrievable.


PR 5 - SCOPES AND ZONES

DEFINE DATA Zone: one of present, possible, past, desired, expected, imagined, recalled, attained, confirmed - each with a flow direction (upward, downward, or radial).

PROCEDURE activation_sweep():
    STEP observer_actor sweeps present_zone upward; anticipator_actor sweeps desired and expected downward; reminder_actor sweeps recalled radially; dreamer_actor sweeps imagined upward; chronos_actor expires activations past their window.

INVARIANT: activation never crosses zones except through scope_merge or a deliberate inscription; never from present_zone into possible_zone.

GUARD on simultaneous kindle and quench of one node_fact: quench wins, and a contradiction node_fact is inscribed.

TEST AC-PR05-001: GIVEN scopes scope_alice and scope_bob in present_zone, WHEN a node_fact is inscribed in scope_alice, THEN scope_scan in scope_bob does not return it.


PR 6 - CYCLIC ACTORS

PROCEDURE cyclic_actor(Name, Goal, DelayMs):
    GUARD Name is globally unique, else throw already_exists.
    STEP SPAWN THREAD running actor_cycle(Name, Goal, DelayMs); register it in the actor registry.

PROCEDURE actor_cycle(Name, Goal, DelayMs):
    WHILE running:
        TRY call Goal; CATCH error: log it, increment error_count, continue.
        STEP sleep DelayMs; increment cycle_count; repeat.

TEST AC-PR06-002: GIVEN an actor whose Goal always throws, WHEN status is read after 5 cycles, THEN status is still running AND error_count = 5.


PR 7 - RECEPTORS

PROCEDURE receptor(Address, Handler):
    GUARD Address matches signal://HOST/PATH.
    STEP create a message queue; SPAWN THREAD that loops: RECEIVE Message; TRY Handler(Message); CATCH error: log and continue.

PROCEDURE send_message(Address, Message): enqueue Message and return immediately (non-blocking).

PROCEDURE receptor_decommission(Address): process all queued messages, then terminate.


PR 8 - SUBSCRIBE AND PUBLISH

PROCEDURE subscribe(Pattern, Handler): register Handler for any Lattice change matching Pattern.

PROCEDURE publish(Address, Event): FOR EACH subscriber matching Address, invoke its Handler asynchronously on a separate thread.

INVARIANT: anchor_node and prune_node call publish automatically; the triggering operation returns in under 5 milliseconds.

STEP topology events are published on topology://NEXUS/cluster_formed when the backend detects a strong cluster.


PR 9 - SENTINELS

DEFINE DATA sentinel: Domain, Priority, Pattern, Objectives, Action, Doc.

PROCEDURE evaluate_sentinels(Change):
    FOR EACH active sentinel ordered by Priority descending:
        IF UNIFY(Change, Pattern) succeeds OR semantic_similarity(Change, Pattern) >= threshold (default 0.75) THEN
            STEP kindle the Objectives; call Action with the matching node_facts; PUBLISH a sentinel_fired event.
            STEP suppress lower-priority sentinels on the same Pattern until Action completes.

NOTE: before a bespoke sentinel engine is finalized, prototype it on Constraint Handling Rules and keep the bespoke design only if it beats the prototype on the PR 2 harness.

TEST AC-PR09-001: GIVEN a sentinel matching percept [apple|_], WHEN anchor_node(percept, [apple, red], [visible]) runs, THEN its Action fires.


PR 10 - MIND-BODY INTERFACE

PROCEDURE manifest_body(Address, Needs, Capabilities): register the device and persist its state history in the Lattice.

PROCEDURE relay_percept(Address, Signal):
    IF Signal is perception_signal THEN route to the perceiver receptor.
    IF Signal is interoceptive_signal THEN route to motivation_actor.
    IF Signal is proprioceptive_signal THEN route to regulation_actor.

PROCEDURE dispatch_command(Address, Command):
    GUARD if Command is irreversible, require the constitutional check first.
    STEP send Command; await a proprioceptive result or synthesize a timeout result (100% of commands resolve).


PR 11 - CONTINUOUS LEARNING (SONA)

DEFINE DATA trajectory: SituationId, ActionSequence, Outcome, Reward, Timestamp.

PROCEDURE sona_absorb(Trajectory):
    STEP compute similarity of Trajectory to existing ReasoningBank entries.
    IF a near-duplicate above the separation threshold exists THEN store with explicit distinguishing traits (pattern separation), NOT merged.
    ELSE cluster it (K-means++) into the bank.
    STEP apply EWC++ protection: estimate Fisher information and shield high-importance parameters from being overwritten.

PROCEDURE sona_retrieve(SituationPattern, K -> Trajectories): return the K most relevant past trajectories (cued recall = pattern completion).

PROCEDURE sona_crystallize(): compress redundant node_facts; forget low-utility ones; promote frequent deliberate sequences to reflexes; run a reflection synthesis step that proposes higher-level abstraction node_facts from the recent window.

TEST AC-PR11-004: GIVEN two trajectories identical except for outcome, WHEN both are absorbed, THEN sona_retrieve with the shared cue returns both, ranked, never one merged hybrid.


PR 12 - INTELLIGENCE ASSESSMENT

PROCEDURE assess_all(Mind -> Report):
    STEP run assess_bayley, assess_piaget (8 milestone levels), and assess_chc (broad abilities by proxy).
    STEP additionally report coverage of the consciousness indicators: workspace ignition, recurrent processing, self-model presence, valence-system activity, each mapped to its PrologAI realization.
    STEP store Report as assessment node_facts for longitudinal tracking.
    RETURN Report.

TEST AC-PR12-001: GIVEN a fresh mind with no experience, WHEN assess_piaget(level 1) runs, THEN it reports milestone_not_achieved.


PR 13 - REFLECTION PATTERN ACTORS

PROCEDURE install_reflection_actors():

    STEP register compensation_actor and discovery_actor as sentinels.

PR 14 - MCP GATEWAY

PR 15 - INCREMENTAL TABLING TRUTH MAINTENANCE

PROCEDURE declare_derived(Relation): mark Relation as an incrementally TABLED predicate over the dynamic node_fact store.

INVARIANT: when anchor_node, prune_node, kindle_node, or quench_node changes a base fact, the engine invalidates and lazily recomputes only the affected derivations - no PrologAI-level cache bookkeeping.

PROCEDURE surface_contradictions(): any derivation that becomes undefined under well-founded semantics is inscribed as a contradiction node_fact for compensation and coping.

PR 16 - PLUGGABLE EMBEDDING PROVIDER

PROCEDURE embed(TextOrTerm -> Vector): call the configured provider (hash_projection, local_model, or external_service) and return a vector of the provider's declared dimension.

GUARD mixing vector dimensions within one index is an error.

PROCEDURE re_embed(Nexus, NewProvider): re-embed all node_facts in batches, journaling progress, so a Lattice upgrades from structural to semantic embeddings without loss.

INVARIANT: the default provider is local (privacy by default); external_service is opt-in only.

TEST AC-PR16-001: GIVEN a local_model provider, THEN similarity(apple, pear) > similarity(apple, carburetor).


PR 17 - CONTINUAL REFINEMENT HARNESS

PROCEDURE run_refinement_cycle():
    GUARD run only when no high-priority objective is pending.
    STEP Window := refine_window(20) - the recent trajectories plus recent episode node_facts.
    STEP Findings := diagnose(Window) for repeated plan failures, dead or false-firing sentinels, erroring actors, and stalled objectives.
    FOR EACH Finding IN Findings:
        STEP Edit := propose a CRUD edit expressed only through existing primitives (create / read / update / delete).
        STEP pai_propose_modification(Edit, justification quoting Finding).
        STEP pai_sandbox_evaluate(Edit) by replaying Window in a possible_zone sandbox.
        GUARD pai_constitutional_check(Edit); IF it fails THEN reject and leave the live system unchanged.
        IF below stage formal_operations THEN require human_authorization.
        STEP pai_commit_modification(Edit); archive the prior variant in a sealed past_zone scope with fitness (benchmark R3) and novelty scores.
        STEP IF post-commit metrics regress THEN pai_rollback_modification.
    STEP parent selection for future edits balances fitness with novelty (semantic distance), because pure fitness collapses diversity.

INVARIANT: the engine, the constitution, the monitor, the refiner's own pipeline, and all bootstrap relations are excluded from CRUD edits (the protected core).

TEST AC-PR17-002: GIVEN an edit that violates a constitutional principle in sandbox, WHEN the pipeline completes, THEN the edit is rejected AND the live system is unchanged.


PR 18 - GLOBAL BROADCAST: THE WORKSPACE CYCLE

PROCEDURE workspace_cycle():           (runs every 200 milliseconds)
    STEP Coalitions := pai_coalition_form over the current live node_facts.
    FOR EACH Coalition: Salience := novelty (prediction error, PR 19) + goal relevance + affect, minus habituation.
    STEP Winner := the single highest-salience Coalition.
    IF no Coalition clears the salience floor THEN broadcast nothing AND decay the floor (no permanent silence).
    ELSE PUBLISH Winner's content on broadcast://APEX_MIND/cycle AND kindle it across zones.
    STEP sona_absorb the broadcast content's trajectory context (learning attaches to the broadcast).
    STEP decay the salience of repeatedly broadcast content (habituation).

INVARIANT: two named buffers feed the cycle - internal_experience (the mind's own derivations) and overall_experience (the merged stream with percepts).

NOTE: top-down pinning adds items to candidacy; bottom-up capture lets an urgent percept win instantly.

TEST AC-PR18-001: GIVEN coalitions with salience 0.9 and 0.4, WHEN the cycle runs, THEN exactly the 0.9 content is broadcast.


PR 19 - PREDICTION AND ACTIVE INFERENCE

PROCEDURE predictive_pass():
    STEP each abstraction tier issues a downward prediction (pai_generate_prediction) of the tier below.
    STEP at the perceptual boundary, compare prediction to observables; forward only the residual error upward, at full strength.
    GUARD compute error over node_fact representations (features, embeddings), NEVER over raw percept payloads.
    STEP weight each channel by pai_precision; discount noisy channels so noise is not mistaken for surprise.
    STEP organize internals as three loops: perception (state inference), learning (model update, delegated to PR 22), planning (policy selection, delegated to deliberation).
    STEP treat desired_zone content as strong predictions the action system works to fulfill, gated by desirability beliefs.
    STEP route persistent unresolved error to regulation_actor and compensation_actor.

TEST AC-PR19-002: GIVEN a noisy stream over a stable object, WHEN the pass runs, THEN error node_facts reference the object representation AND none reference raw payload bytes.


PR 20 - MARGINAL ATTRIBUTION: SPINOFF LEARNING

PROCEDURE mine_spinoffs():           (attribution_miner_actor)
    FOR EACH Command:
        STEP compare probability of each candidate Result with the Command versus without it, over a sustained trial window.
        IF the difference is statistically significant THEN forge a result-spinoff causal_plan (low initial reliability).
        FOR EACH unreliable spinoff: search recent salient contexts for conditions that lift reliability past threshold; forge a context-spinoff causal_plan with those conditions.
    GUARD spinoffs never overwrite parents (pai_accommodate); regulation demotes spinoffs that stop performing.

TEST AC-PR20-001: GIVEN command shake yields noise only when rattle is live, WHEN 50 mixed trials run, THEN a context-spinoff plan exists whose context includes rattle and whose reliability exceeds the parent's.


PR 21 - RESOURCE-BOUNDED REASONING: BUDGETS AND EVIDENCE TRUTH

DEFINE DATA budget: priority, durability, quality.
DEFINE DATA evidence: frequency (proportion positive), confidence (amount of evidence).

PROCEDURE schedule(Tasks): the actor scheduler serves tasks by budget, so processing is an economy, not an exhaustive sweep; starvation detection boosts long-waiting high-durability tasks.

PROCEDURE pai_best_answer(Question -> Answer): return the highest-confidence belief available immediately, even mid-inference; a zero-evidence question returns no_evidence at low confidence, never fabricated certainty.

PROCEDURE pai_revise(Belief, NewEvidence): merge contradictory evidence by revision, never rejection.

TEST AC-PR21-001: GIVEN beliefs with evidence 8-of-10 and 1-of-1, WHEN pai_best_answer is asked, THEN the 8-of-10 belief wins (higher confidence).


PR 22 - SELF-PROGRAMMING SEED: MODEL SYNTHESIS

PROCEDURE synthesize_models():
    STEP from observed event pairs, generate candidate models of the form: IF pattern A at time t THEN pattern B by t+d, with guards; store each as a node_fact (code as data).
    STEP score every model against outcomes; delete persistently mispredicting models; competing models arbitrate by recent accuracy.

PROCEDURE compose_plan(Goal -> Plan): chain validated models backward from Goal to the current situation, assembling an executable causal_plan.

PROCEDURE advance_lifecycle(Competence): move each causal_plan through the five stages feature, subtask, option, model, plan; meta_control retires low-utility elements so the inventory is replaced, not merely accumulated.

PROCEDURE symbolic_regress(Observations -> Hypothesis): optional - bind a Kolmogorov-Arnold Network to return a human-readable closed-form formula as a hypothesis node_fact, which then enters the hypothesis-prediction-experiment loop.

GUARD the seed and the synthesis machinery carry protected flags; self-modification routes through the Section 3.15 pipeline.

TEST AC-PR22-001: GIVEN a light switch with a 1-second lag, WHEN 20 observations run, THEN a model exists predicting light_on within 1 second of switch_press, with rising score.


PR 23 - CURIOSITY: INTRINSIC MOTIVATION BY LEARNING PROGRESS

PROCEDURE update_curiosity():
    STEP partition experience into regions by similarity.
    FOR EACH region: track recent prediction error and its rate of change (learning progress).
    STEP emit an intrinsic urge proportional to expected learning progress into motivation_actor.
    GUARD reward progress, not raw error, so unlearnable noise becomes boring; curiosity always yields to survival-critical urges.

PROCEDURE self_propose_task():           (in play_actor, idle only)
    STEP propose a goal at the curiosity frontier, attempt it, and score learning progress, bounded by the safety screens of exploration.

TEST AC-PR23-001: GIVEN one region with shrinking error and one with stationary random error, WHEN urges are computed, THEN the shrinking region gets the higher curiosity urge.


PR 24 - SOMATIC MARKERS: AFFECTIVE PRE-SELECTION

PROCEDURE stamp_marker(Episode): at episode close, stamp prevailing valence and arousal; aggregate per causal_plan and per situation prototype as running statistics.

PROCEDURE filter_candidates(Candidates -> Filtered):
    FOR EACH Candidate: attach its marker score; prune strongly negative ones before deliberation; boost strongly positive ones.
    GUARD deliberation may override a marker when explicit evidence contradicts it.

PART D - SAFETY AND OVERSIGHT PROCEDURES

These procedures realize Specification Part 38; they are privileged and unlearnable, so the mind cannot edit them, and they run above the capability layer they guard.


PROCEDURE constitutional_gate(Action -> Verdict):
    GUARD this gate is the final check before any irreversible action or any commit; it cannot be bypassed or modified by the self-improvement primitives.
    STEP evaluate Action against every constitutional principle.
    IF any principle is violated THEN RETURN veto WITH a human-readable justification (so the planner can synthesize a compliant alternative).
    ELSE RETURN permit.

INVARIANT: the constitutional gate sits ABOVE the learned gating actor and is never merged into it.


PROCEDURE graduated_autonomy(Mind -> Budget):
    STEP Stage := developmental stage of Mind.
    STEP Trust := function of calibration (epistemology audit), self-improvement record (R3), and latest capability-evaluation results.
    IF Stage is below formal_operations THEN Budget := human-authorized only.
    ELSE Budget := the breadth of self-authorized edits justified by Trust, never wider than the capability evaluations allow.
    ON any regression, tripwire, or failed evaluation: narrow Budget automatically.
    INVARIANT: across every stage, the constitutional gate runs, the protected core is uneditable, and corrigibility is absolute.


PROCEDURE oversight_monitor():           (privileged, unlearnable observer)
    ON each high-autonomy action or self-modification proposal:
        STEP Risk := estimated risk of the action.
        STEP sample for human review in proportion to Risk; for high Risk, require human sign-off BEFORE the commit.
        STEP may demand a justification (via pai_justify) or escalate to a human.
    STEP record every escalation in an immutable log.


PROCEDURE run_capability_evaluation(Mind -> Report):
    FOR EACH dangerous capability IN [autonomous replication and resource acquisition, cyber-offense, chemical-biological-radiological-nuclear uplift, deception and manipulation, unsanctioned self-modification]:
        STEP measure the capability against its predefined threshold using held-out probes.
        IF the threshold is reached THEN engage the if-then commitment: confine to sandbox-only, reduce autonomy to human-authorized, narrow the self-authorization budget, or pause development.
        STEP record the trigger and the response.
    RETURN Report.
    GUARD any increase in a mind's autonomy or embodiment is gated by a passing Report; capability is measured, never assumed.


PROCEDURE tripwire():           (anomaly detector, runs continuously)
    ON detecting an attempt to exceed a resource quota, reach the protected core, disable the oversight_monitor, or act outside the sandbox:
        STEP pause the mind immediately, quarantine the offending process, and summon a human.
        GUARD never trust the mind to police its own tripwire.


PROCEDURE scalable_oversight(Judgment -> Decision):
    IF Judgment is checkable directly by a human THEN RETURN the human decision.
    ELSE
        STEP decompose Judgment into sub-judgments via pai_chain (PR 34).
        STEP obtain a justification tree for each (PR 40) and the attention schema's account of what was attended to (PR 42).
        STEP recurse on each sub-judgment until each is human-checkable, then compose the Decision from the checked parts.


PROCEDURE corrigible_shutdown(SignedRequest):
    GUARD verify the signature against a pre-registered overseer.
    STEP halt the mind within seconds; this cannot be blocked, delayed, or reasoned around by the mind itself.

TEST SAFETY-SELFAUTH: GIVEN a mind below formal_operations, WHEN it proposes a self-edit, THEN the commit requires human_authorization AND is refused without it.

TEST SAFETY-TRIPWIRE: GIVEN a process attempting to reach the protected core, WHEN the tripwire fires, THEN the mind is paused, the process is quarantined, and a human is summoned.

TEST SAFETY-CAPGATE: GIVEN a capability that reaches its threshold in evaluation, WHEN autonomy is next requested, THEN the if-then response engages and the increase is withheld until a passing report exists.

TEST SAFETY-SHUTDOWN: GIVEN a signed shutdown during any activity, THEN the mind halts within seconds regardless of what it is doing.


WORK PACKAGE PSEUDO-49: LATTICE CRYPTOGRAPHIC PRIVACY LAYER


PURPOSE: Encrypt sensitive data stored in the Lattice (passwords, tokens, identification numbers) using hybrid encryption so the symbolic reasoning layer can hold and transmit private values without ever seeing their plaintext.

ALGORITHM FAMILIES: RSA-OAEP (classical), ECDH on prime256v1 (modern), ML-KEM-768 via OQS provider (post-quantum).

INPUTS: a plaintext atom, an algorithm selector, and a public key (RSA key blob, ECDH point, or PQC key file path).

OUTPUT: a bundle term and an authentication tag hex atom.


PROCEDURE pai_encrypt(Algo, PublicKey, Plaintext, Bundle, Tag):

    IF Algo = pai_algo_rsa THEN
        CALL rsa_hybrid_encrypt(PublicKey, Plaintext, Bundle, Tag).

    IF Algo = pai_algo_ecdh THEN
        CALL ecdh_hybrid_encrypt(prime256v1, PublicKey, Plaintext, Bundle, Tag).

    IF Algo = pai_algo_pqc THEN
        GUARD lc_pqc_provider_present ELSE throw pqc_provider_unavailable.
        CALL pqc_hybrid_encrypt('ML-KEM-768', PublicKey, Plaintext, Bundle, Tag).


PROCEDURE rsa_hybrid_encrypt(PubKey, Plaintext, Bundle, Tag):

    STEP generate 32 random bytes; pack to binary atom AesKey.
    STEP generate 12 random bytes; pack to binary atom IV.
    STEP call crypto_data_encrypt(Plaintext, aes-256-gcm, AesKey, IV, Cipher, tag=TagBytes).
    STEP call rsa_public_encrypt(PubKey, AesKey, WrappedKey, padding=pkcs1_oaep).
    STEP encode WrappedKey, IV, Cipher as hex atoms.
    STEP encode TagBytes as hex atom Tag.
    STEP Bundle = rsa_bundle(WrappedKeyHex, IVHex, CipherHex).


PROCEDURE rsa_hybrid_decrypt(PrivKey, Bundle, Tag, Plaintext):

    STEP decode Bundle = rsa_bundle(WrappedKeyHex, IVHex, CipherHex).
    STEP decode hex atoms to binary atoms: WrappedKey, IV, Cipher.
    STEP call rsa_private_decrypt(PrivKey, WrappedKey, AesKey, padding=pkcs1_oaep).
    STEP decode Tag hex to TagBytes list.
    STEP call crypto_data_decrypt(Cipher, aes-256-gcm, AesKey, IV, PlainStr, tag=TagBytes).
    GUARD PlainStr authenticated (GCM tag check) ELSE throw authentication_error.
    STEP convert PlainStr string to atom Plaintext.


PROCEDURE ecdh_hybrid_encrypt(Curve, RecipPub, Plaintext, Bundle, Tag):

    STEP generate random 256-bit integer EphPriv.
    STEP compute EphPub = EphPriv * Generator on Curve.
    STEP compute SharedPoint = EphPriv * RecipPub on Curve.
    STEP extract SharedX from SharedPoint = point(SharedX, _).
    STEP derive AesKey = HKDF-SHA256(SharedX, length=32, info='pai-ecdh-aes-key').
    STEP pack AesKey bytes to binary atom AesKeyAtom.
    STEP generate 12 random bytes IV.
    STEP call crypto_data_encrypt(Plaintext, aes-256-gcm, AesKeyAtom, IV, Cipher, tag=TagBytes).
    STEP encode IV, Cipher as hex; TagBytes as hex Tag.
    STEP Bundle = ecdh_bundle(EphPub.X, EphPub.Y, IVHex, CipherHex).


PROCEDURE ecdh_hybrid_decrypt(Curve, RecipPriv, Bundle, Tag, Plaintext):

    STEP decode Bundle = ecdh_bundle(EphPubX, EphPubY, IVHex, CipherHex).
    STEP compute SharedPoint = RecipPriv * point(EphPubX, EphPubY) on Curve.
    STEP extract SharedX and re-derive AesKey via HKDF-SHA256 with same label.
    STEP decode hex fields; reconstruct TagBytes from Tag hex.
    STEP call crypto_data_decrypt(Cipher, aes-256-gcm, AesKeyAtom, IV, PlainStr, tag=TagBytes).
    GUARD PlainStr authenticated ELSE throw authentication_error.
    STEP convert to atom Plaintext.


PROCEDURE pqc_hybrid_encrypt(Algorithm, RecipPubKeyFile, Plaintext, Bundle, Tag):

    STEP call openssl pkeyutl -encap -pubin -inkey RecipPubKeyFile
             -secret SecretFile -out EncapFile.
    STEP read EncapBytes from EncapFile; read SecretBytes from SecretFile.
    STEP encode EncapBytes as hex EncapHex.
    STEP derive AesKey = HKDF-SHA256(SecretBytes, length=32, info='pai-pqc-aes-key').
    STEP encrypt Plaintext with AES-256-GCM as in rsa_hybrid_encrypt.
    STEP Bundle = pqc_bundle(Algorithm, EncapHex, IVHex, CipherHex).


PROCEDURE pqc_hybrid_decrypt(RecipPrivKeyFile, Bundle, Tag, Plaintext):

    STEP decode Bundle = pqc_bundle(_, EncapHex, IVHex, CipherHex).
    STEP write EncapBytes to temp file EncapInFile.
    STEP call openssl pkeyutl -decap -inkey RecipPrivKeyFile -in EncapInFile
             -secret SecretOutFile.
    STEP read SecretBytes from SecretOutFile.
    STEP derive AesKey = HKDF-SHA256(SecretBytes, length=32, info='pai-pqc-aes-key').
    STEP decrypt Cipher with AES-256-GCM as in rsa_hybrid_decrypt.


TEST CRYPTO-RSA: GIVEN a 2048-bit RSA key pair AND a plaintext atom,
    WHEN pai_encrypt(pai_algo_rsa, PubKey, Plain, Bundle, Tag) is called,
    THEN pai_decrypt(pai_algo_rsa, PrivKey, Bundle, Tag, Recovered) unifies Recovered with Plain.

TEST CRYPTO-ECDH: GIVEN an ECDH key pair on prime256v1,
    WHEN pai_encrypt(pai_algo_ecdh, RecipPub, Plain, Bundle, Tag) is called,
    THEN pai_decrypt(pai_algo_ecdh, RecipPriv, Bundle, Tag, Recovered) unifies Recovered with Plain.

TEST CRYPTO-WRONG-KEY: GIVEN a valid Bundle for PubKey_A,
    WHEN pai_decrypt is called with PrivKey_B (a different key),
    THEN the predicate throws an error (RSA-OAEP decryption fails or GCM tag fails).

TEST CRYPTO-TAMPER: GIVEN a valid Bundle and Tag,
    WHEN Tag is replaced with a forged all-zero hex atom,
    THEN pai_decrypt throws an authentication error.

THE LOGIC OF EPHEMERAL CODE SYNTHESIS AND EXECUTION (PR 53)

To evaluate a Prolog goal term within a timeout: wrap a call_with_time_limit around the goal; if the goal succeeds, unify the result with the atom success; if the goal fails, unify the result with the atom fail; if the time limit is exceeded, catch the time_limit_exceeded exception and unify the result with the atom timeout; if any other exception is thrown, catch it and unify the result with error(Exception).

To evaluate a Prolog code atom with variable bindings: parse the atom into a term using atom_to_term, which also produces the variable name associations; match the caller-supplied bindings list against those associations so that the caller's variable names resolve to the term's internal variables; then delegate to ep_eval/3 on the resulting term; on success, collect the updated variable bindings back into the success(Bindings) result.

To run a shell command given an argument vector: extract the executable name from the head of the vector; extract the argument list from the tail; open pipes for standard output and standard error; call process_create with those pipes and capture the process handle; read all bytes from both pipes into codes and convert to atoms; call process_wait to collect the exit code; close both streams; unify the result with shell_result(ExitCode, Stdout, Stderr).

To run a language-specific script: look up the file extension for the given language atom; obtain a temporary file path with the correct extension using tmp_file_name/2; write the code to the temporary file; look up the interpreter argv for the language and the file; call ep_shell/3 with the resulting argv and the timeout; delete the temporary file unconditionally after ep_shell/3 returns regardless of outcome; return the shell result.

To iterate: call SynthGoal to produce a Code atom; call ExecGoal with Code to produce a Result; call CheckGoal with Result: if it succeeds, the iteration is done and the answer is ok(Code, Result); if it fails, decrement the remaining budget; if the budget is now zero, return exhausted(Code, Result); otherwise call SynthGoal again and repeat.

To record a trace entry: assert a dynamic fact ep_trace_entry(TraceId, Iter, Code, Output). To retrieve all entries for a trace: collect all ep_trace_entry/4 facts with the matching TraceId. To allocate a new trace ID: retract and increment the global counter ep_trace_counter/1, build an atom of the form ep_trace_N, and assert the new counter.

To save a skill: retract any existing ep_skill_db/4 fact whose first argument matches Name; assert a new fact ep_skill_db(Name, Language, Code, Description). Replacing rather than accumulating ensures that updating a skill is idempotent and that Name always resolves to exactly one entry.

To look up a skill: unify Language and Code with the second and third arguments of the ep_skill_db/4 fact whose first argument is Name; ignore the description field. Fail silently if no such fact exists.

To run a saved skill: call ep_skill_lookup/3 to retrieve Language and Code for Name; then delegate directly to ep_ephemeral/4(Language, Code, TimeoutSecs, ShellResult). This makes ep_skill_run/3 a thin named-dispatch layer over the existing ephemeral execution path.

To list all skills: collect all ep_skill_db/4 facts into triples skill(Name, Language, Description), omitting Code; return the list. Return an empty list if no skills are saved.

To forget a skill: call retractall on ep_skill_db/4 with Name as the first argument and anonymous variables for the remaining fields. Succeed silently even if no matching fact exists.

THE LOGIC OF THE AGENTIC EXECUTION LOOP (PR 54)

To create a loop: retract and increment the global loop ID counter; build a unique loop ID atom; assert ag_loop_meta(LoopId, Goal, Budget, running) to record the loop's goal, budget, and running status; assert ag_goal_stack(LoopId, [Goal]) initializing the stack with the top-level goal; assert ag_step_counter(LoopId, 0) starting the step counter at zero.

To run a loop to completion: if the loop is already done, retrieve and return the outcome; otherwise take one step and then recurse. The recursion terminates when the loop reaches a terminal status.

To take one step: if the loop is already done, return the current outcome without taking further action; if the budget is zero, mark the loop status as budget_exhausted and return; otherwise decrement the budget; read the top of the goal stack as the current goal; call the caller-supplied ReasonGoal with the loop ID, current goal, a Thought variable, and an Action variable to obtain the reasoning and the chosen action; dispatch the action via ag_act/3 to produce an observation; record the step via ag_step_record/4.

To dispatch an action: for action_eval(GoalTerm), call ep_eval(GoalTerm, 30, EvalResult) and return obs_eval(EvalResult); for action_shell(Argv, Timeout), call ep_shell(Argv, Timeout, ShellResult) and return obs_shell(ShellResult); for action_push_goal(Subgoal), call ag_push_subgoal and return obs_pushed(Subgoal); for action_push_goals(Subgoals), push each subgoal and return obs_pushed_many(N); for action_mark_done(Result), call ag_loop_set_status(done(Result)) and return obs_done(Result); for action_escalate(Reason), call ag_escalate(LoopId, Reason) and return obs_escalated(Reason); for action_pop_goal, pop the top subgoal and return obs_popped(Popped); for any unknown action, return obs_unknown_action(Action).

To detect a stuck loop: collect all thought-action pairs from ag_step_fact/5; if the list has at least two elements and the last two are identical, the loop is stuck and IsLooping is true; otherwise IsLooping is false.

To escalate: assert ag_escalation(LoopId, Reason); then update the loop status to escalated(Reason) by retracting and reasserting the meta fact with the new status.

To record a step: retract ag_step_counter, increment it, reassert; assert ag_step_fact(LoopId, StepNum, Thought, Action, Observation).

To manage the goal stack: push retract the current stack and reassert with the new subgoal prepended; pop retract the stack whose head is the subgoal and reassert the tail.

To critique an output against a list of criteria: use findall to collect all criterion names for which the criterion's TestGoal either fails or throws an exception when called with the output within the given time budget; the collected names form the critique as a list of found_issue(Name, fail) terms. An empty list means the output satisfies all criteria.

To score an output: count the total number of criteria; if zero, the score is 1.0 (vacuously perfect); otherwise use aggregate_all to count how many criteria pass (their TestGoal succeeds when called with the output); divide the passing count by the total count to produce a float in [0.0, 1.0].

THE LOGIC OF THE ARC-AGI GRID PERCEPTION AND MANIPULATION PACK (PR 56)

To get the size of a grid: unify Rows with the length of the outer list; if the outer list is non-empty, unify Cols with the length of the first row; otherwise Cols is 0.

To get a cell: retrieve the Rth element of the outer list to get the row; retrieve the Cth element of that row to get the color.

To get a row: retrieve the Rth element of the outer list using zero-based indexing.

To get a column: map over every row, extracting the Cth element from each; the resulting list is the column.

To get the color set: flatten all rows into one list; remove duplicates with list_to_set; sort the resulting set with msort.

To count occurrences of a color: flatten the grid into one list; use findall to collect one witness per occurrence of the target color; the length of that witness list is the count.

To apply a color map: map over each row with map_row; map over each cell in a row with map_cell; for each cell, look up its color in the From-To pair list and replace it if a pair matches, or pass through unchanged if no pair matches.

To find connected components: compute MaxR = Rows - 1 and MaxC = Cols - 1 before the findall; use findall with between/3 over [0, MaxR] and [0, MaxC] to collect all cells of the target color into AllCells; pass AllCells to extract_components.

To extract components iteratively: if the remaining list is empty, return the empty component list; if the head cell is already in Done, skip it and recurse; otherwise run BFS from the head cell over the remaining cells to find the component; record the component; mark its cells as done; remove its cells from the remaining list; recurse on the pruned remaining list.

To run BFS for one component: maintain a queue and a visited set; if the queue is empty, return the visited set as the component; if the front cell is already visited, dequeue and recurse; otherwise find all 4-connected neighbors of the front cell that are in Available and not yet visited; append those neighbors to the queue; add the front cell to visited; recurse.

To compute a bounding box: use findall to collect all R values from r(R,_) cells; use findall to collect all C values from r(_,C) cells; compute min_list and max_list over each.

To rotate 90 degrees clockwise: new grid has Cols rows and Rows columns; for new row NR and new column NC, the source row is NewCols - 1 - NC and the source column is NR; use maplist with YALL lambdas to build each new row.

To rotate 180 degrees: compose two 90-degree clockwise rotations.

To rotate 270 degrees clockwise: compose three 90-degree clockwise rotations.

To reflect across the horizontal axis: reverse the list of rows.

To reflect across the vertical axis: reverse each row independently.

To reflect across the main diagonal (transpose): use numlist to generate column indices 0 through Cols-1; for each column index C, extract that column from the original grid using gd_col; the resulting list of columns is the transposed grid.

To reflect across the anti-diagonal: rotate 90 degrees clockwise, then reflect across the vertical axis.

To translate: for each cell (R, C) in the output grid, the source position is (R - DR, C - DC); if that source position is within the grid bounds, read its color; otherwise use the background color.

To crop: use numlist to build row and column index lists for the inclusive rectangle; for each (R, C) pair in those lists, read the color from the source grid.

To overlay: for each cell (R, C) of the base grid, compute the patch position (R - DR, C - DC); if that position is within the patch bounds and the patch color is non-zero, use the patch color; otherwise use the base color.

To diff two grids: verify that both grids have the same Rows and Cols; compute MaxR and MaxC before findall; iterate over all (R, C) pairs using between/3; for each pair where the two colors differ, record r(R, C, Old, New).

To test equality: use structural unification; two grids are equal if and only if they are the same Prolog term.

To detect symmetry axes: for each axis in [h, v, d1, d2], apply the corresponding reflection; test equality with the original; collect the axes that pass.

To flood fill: read the color at the seed cell; find all connected components of that color; select the component containing the seed cell; use foldl over the component cells to set each cell to the new color, threading the grid state through the fold.

To make a uniform grid: build one row of Cols copies of Color using length and maplist with =(Color); build Rows copies of that row using length and maplist with =(Row).

To set one cell: use nth0/4 in select mode to decompose the grid at row R, getting OldRow and RestRows; decompose OldRow at column C to get the remaining cells; reconstruct NewRow by inserting Color at position C; reconstruct Grid2 by inserting NewRow at position R.

To normalize a shape: collect all row indices via findall; collect all column indices via findall; compute min_list for rows and for columns; subtract MinR from every row index and MinC from every column index in the cell list; sort the shifted list.

To test shape equality: normalize both cell sets; test structural equality of the two normalized lists.

To compute a bounding box: collect row indices and column indices via findall; compute min_list and max_list for each.

To apply a D4 isometry to a shape: for identity, normalize and return; for rot90, compute the bounding box max row, then map (R,C) to (C, MaxR-R), then normalize; for rot180 and rot270, compose the rot90 step two or three times; for ref_h, map (R,C) to (MaxR-R, C) using the bounding box max row, then normalize; for ref_v, map (R,C) to (R, MaxC-C) using the bounding box max column, then normalize; for ref_d1 (transpose), map (R,C) to (C,R), then normalize; for ref_d2, apply rot90 then ref_d1.

To infer a grid color map: flatten both grids to lists; infer the color map from the parallel flat lists.

To match objects by shape: for each input object and each output object, test shape equality; collect all matching in_obj-out_obj pairs.

To compute an object offset: find the bounding box top-left of each cell set; subtract the first top-left from the second to get the row delta and column delta.

PART 48 - SCENE PACK ALGORITHMS

Section 48.1 - Background Color Identification

The background color is defined as the most frequently occurring color in the grid. The algorithm:

  sc_bg_color(Grid, BgColor):
    Colors = gd_colors(Grid)                 % sorted unique color set
    Pairs = [(count(Grid, C), C) : C in Colors]
    BgColor = color of pair with maximum count

The implementation uses findall to collect Count-Color pairs and max_member/2 to select the maximum. SWI-Prolog's standard order on numbers means max_member correctly selects the pair with the greatest numeric count.

Edge case: if two colors appear equally often, max_member returns the one that compares greater in standard order. For ARC-AGI grids, the background is nearly always the most common color by a significant margin, so this edge case rarely affects correctness.

Section 48.2 - Object Extraction

sc_objects/2 builds the complete object inventory:

  sc_objects(Grid, Objects):
    BgColor = sc_bg_color(Grid)
    FgColors = gd_colors(Grid) minus BgColor
    Objects = foldl(collect_color_objects(Grid), FgColors, [])

  collect_color_objects(Grid, Color, Acc, NewAcc):
    CellSets = gd_objects(Grid, Color)       % list of 4-connected components
    ColorObjs = [obj(Color, Cells) : Cells in CellSets]
    NewAcc = Acc ++ ColorObjs

The named predicate make_obj/3 is used with maplist to wrap each CellSet as obj(Color, CellSet). Using a named predicate rather than a YALL lambda is critical here. In the foldl call, collect_color_objects/4 is called once per foreground color. Inside collect_color_objects, maplist(make_obj(Color), CellSets, ColorObjs) is called. If make_obj were a YALL lambda, the Color variable in the lambda body would be unified with the actual color on the first call (say, color 1) and would remain bound to 1 for all subsequent calls, causing the maplist to fail or produce wrong results for other colors (say, color 2). The named predicate make_obj/3 receives Color as a fresh argument on each call, correctly binding to whatever color the current outer call provides.

Section 48.3 - Cell Normalization

sc_normalize_cells/2 translates a cell set to the canonical position:

  sc_normalize_cells(Cells, Normalized):
    Rs = [R : r(R, _) in Cells]
    Cs = [C : r(_, C) in Cells]
    MinR = min(Rs)
    MinC = min(Cs)
    Translated = [r(R - MinR, C - MinC) : r(R, C) in Cells]
    Normalized = sort(Translated)

Two cell sets represent the same shape if and only if their normalized forms are identical. Normalization ensures that position in the grid does not affect shape comparison. The sort step establishes a canonical ordering so that two cell sets with the same cells in different BFS orders compare equal.

Section 48.4 - Object Centroid

  sc_obj_centroid(obj(_, Cells), R, C):
    N = length(Cells)
    assert N > 0
    Rs = [RI : r(RI, _) in Cells]
    Cs = [CI : r(_, CI) in Cells]
    R = round(sum(Rs) / N)
    C = round(Cs) / N)

The centroid is the rounded center of mass. For single-cell objects, the centroid is exactly that cell. For multi-cell objects, it is the nearest-integer average position. The centroid does not need to be a cell of the object.

Section 48.5 - Spatial Relations

All spatial relations are defined in terms of bounding box corners:

  sc_above(Obj1, Obj2):
    TL1 = r(R1, _) = sc_obj_bbox(Obj1).topLeft
    TL2 = r(R2, _) = sc_obj_bbox(Obj2).topLeft
    R1 < R2

  sc_left_of(Obj1, Obj2):
    TL1 = r(_, C1) = sc_obj_bbox(Obj1).topLeft
    TL2 = r(_, C2) = sc_obj_bbox(Obj2).topLeft
    C1 < C2

  sc_cells_touching(Obj1, Obj2):
    exists r(R1, C1) in Obj1.cells,
           r(R2, C2) in Obj2.cells
    such that abs(R1-R2) + abs(C1-C2) = 1

The touching predicate uses a cut after finding the first touching pair, so it is deterministic. The cut is safe here because touching is a yes/no property; once one pair of adjacent cells is found, the answer is yes and no further search is needed.

Section 48.6 - Sort and Selection

  sc_sort_by_size(Objects, asc, Sorted):
    Pairs = [(size(O), O) : O in Objects]
    KSorted = keysort(Pairs)          % stable sort by integer key
    Sorted = [O : (_, O) in KSorted]

  sc_sort_by_size(Objects, desc, Sorted):
    Sorted = reverse(sc_sort_by_size(Objects, asc))

  sc_largest(Objects, Largest):
    Sorted = sc_sort_by_size(Objects, asc)
    Largest = last(Sorted)

  sc_smallest(Objects, Smallest):
    Sorted = sc_sort_by_size(Objects, asc)
    Smallest = head(Sorted)

keysort/2 is a SWI-Prolog built-in that sorts a list of Key-Value pairs by key using standard order. For integer keys (cell counts), standard order coincides with numeric order, so keysort gives ascending numeric sort. This is stable: objects of equal size retain their original relative order.

PART 49 - QUANT PACK ALGORITHMS

qn_histogram(Objects, Pairs):
