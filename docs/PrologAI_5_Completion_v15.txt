PART COMP-0 - WHAT COMPLETION IS


SECTION 0.1 - THE DEFINITION OF DONE

The platform is done, for a release, when four conditions hold together.

Correct and reliable: the refinement exit criteria are met, with the full acceptance-criterion catalogue green, the five test levels passing, the decision gates recorded, and the performance contract holding at the specified scales.

Packaged: a single prologai executable is produced per supported platform, bundling the engine, the compiled core, and all packs, installable without errors in under thirty minutes.

Documented: the full documentation set ships with the release, so a newcomer can install, run, understand, extend, and operate the platform.

Lawful and open: the licensing and NOTICE obligations are satisfied, and the source is published openly so anyone can build with it.

Completion is the discipline of making all four true at once and declaring it.


SECTION 0.2 - WHAT COMPLETION IS NOT

Completion of the platform is not the completion of a mind.

A finished platform is a substrate ready to grow minds; a mind such as Mentova is then grown on it through a developmental run, which is a separate, ongoing process.

Completion of one release is also not the end of the project: the SPARC cycle repeats for the next version, and the mind's own continual refinement loop never stops.

Completion means the substrate is finished enough to be trusted, shipped, and built upon, not that learning has ended.


PART COMP-1 - THE RELEASE ARTIFACT


SECTION 1.1 - THE DISTRIBUTION

The release artifact is produced from the prologai-dist repository: installers and saved states that bundle a released SWI-Prolog, the compiled prologai-core shared library, and all packs into one prologai executable per platform.

The supported platforms are Linux, macOS, and Windows, each with its own installer and its own compiled core library.

The executable preloads every pack, installs the bootstrap ontology, prints the banner with the embedded SWI-Prolog attribution line, sets the version and dialect flags, and starts the top level.


SECTION 1.2 - LICENSING AND NOTICE, FINALIZED

The licensing is finalized at completion and shipped in the distribution.

PrologAI's own original code is licensed Apache-2.0, for its explicit patent grant, its retaliation clause, and its trademark protection of the PrologAI name.

The embedded SWI-Prolog retains its Simplified BSD-2 notice; the vector backend components retain their MIT OR Apache-2.0 notices; any optional LGPL components remain separately licensed and replaceable by the end user.

A NOTICE file at the repository root credits SWI-Prolog, the vector backend, and their copyright holders, and every distribution carries it; this satisfies the notice-preservation obligations that attach because the distribution redistributes engine binaries.


SECTION 1.3 - THE RELEASE CONTENTS MANIFEST

A complete release contains, at minimum: the per-platform prologai executable; the source of all packs and the launcher and syntax components; the compiled and source prologai-core; the documentation set; the test suite and its recorded results; the decision-gate result files; the NOTICE and all license files; and a signed checksum manifest so a downloader can verify integrity.

The recorded bake-off and audit results travel with the release, so the evidence behind every closed decision is auditable by anyone who receives it.

PART COMP-3 - THE DEFINITION-OF-DONE CHECKLIST

The following checklist aggregates the conditions that must all be true to declare a release complete.

Quality: the acceptance-criterion catalogue is green across the foundation, cognitive, substrate-and-frontier, and system bands.

Quality: the five test levels pass, including the loops audit and the epistemology audit.

Quality: the five decision gates have recorded results with a selected default and retained fallbacks.

Quality: the performance contract holds at the specified scales, measured at the 99th percentile on the perception-to-action path.

Quality: the continual refinement harness is validated, with a long longitudinal run showing no unrolled regression and stable calibration.

Packaging: the per-platform executables build and install cleanly in under thirty minutes, and the checksum manifest is signed.

Packaging: the no-fork dividend is re-verified - the packs rebuild against the bundled engine version without source changes.

Documentation: the five SPARC volumes, the README, the PlDoc, the micro-step guides, the contributor guide, and the safety and governance documents all ship.

Legal: Apache-2.0 is applied to original code, upstream notices are preserved, and the NOTICE file is present and correct.

Openness: the source is published at the public repository with a license, a contributor guide, and a code of conduct.

Risk: the risk register is current, with every retired risk backed by a recorded green test and every open risk carrying a disposition.

Only when every line above is satisfied is the ship gate opened.


PART COMP-4 - THE RELEASE PROCESS AND VERSIONING


SECTION 4.1 - VERSIONING

PrologAI uses semantic versioning: a version is major.minor.patch, where major changes may break compatibility, minor changes add capability compatibly, and patch changes fix defects.

The first complete platform release is version 1.0.0, and the running flag prologai_version reports it, while the dialect flag remains swi for full engine compatibility.

The work-package series and the capability layers are the unit of minor growth: a new mind capability arrives as a new pack and a new layer, compatibly, exactly as the frontier packs were added to the foundation.


SECTION 4.2 - THE RELEASE GATING

A release is gated by the definition-of-done checklist; the ship gate does not open until every line is satisfied and recorded.

Each release carries a changelog that states what was added, changed, fixed, and deprecated, and a migration note when a major version changes anything compatibility-affecting.

Each release is tagged in version control, and the tag points at the exact sources, recorded results, and documentation that constitute that release.


SECTION 4.3 - THE NO-FORK DIVIDEND, RE-VERIFIED EVERY RELEASE

Because the engine is an unforked dependency, every release re-verifies that the packs rebuild against the bundled engine version without source changes.

When the engine publishes an upgrade within a major version, the upgrade is adopted, the test levels are re-run, and the dividend is confirmed: improvements arrive for free, and the project never carries a divergent engine copy.

This re-verification is itself a system acceptance criterion, so the dividend is a tested property of every release, not a hope.


PART COMP-5 - THE OPEN-SOURCE LAUNCH AND GOVERNANCE


SECTION 5.1 - THE PUBLIC REPOSITORY

PrologAI is published, free and open source, at https://github.com/ai-university-aiu/PrologAI.

The repository ships the Apache-2.0 license, the NOTICE file, a README that orients a newcomer, a contributor guide, and a code of conduct, so that the project is not only open but joinable.

Publishing openly is a completion deliverable, not an afterthought, because the platform's purpose is that anyone can build a mind with it.


SECTION 5.2 - GOVERNANCE

Governance is stated plainly so contributors know how decisions are made.

Changes arrive as pull requests against the monorepo, pass the test levels in continuous integration, and are reviewed against the specification and the refinement criteria before merge.

The constitutional layer, the protected core, and the kernel are governed conservatively: changes to them require explicit review and a recorded rationale, because they are the substrate's safety and identity.

The risk register is a governance instrument: open risks are visible, and a risk is retired only by a recorded green test.


SECTION 5.3 - UPSTREAM AND ECOSYSTEM DISTRIBUTION

Three general-purpose components are contributed upstream rather than kept private: the background worker (from the cyclic-actor pack), the message endpoint (from the receptor pack), and the publish-subscribe library, each stripped of cognitive-architecture framing.

The recommended sequence is to publish them first as a community pack, let them mature, and then propose core inclusion with adoption as evidence.

The general-purpose library packs are distributed through the SWI-Prolog pack registry, so the wider community benefits even if it never builds a whole mind, and improvements to the existing engine ecosystem (for example the Model Context Protocol server) are contributed back rather than duplicated.


PART COMP-6 - THE DEPLOYMENT AND OPERATIONS RUNBOOK


SECTION 6.1 - SINGLE-HOST INSTALL

To stand up a single mind: install a supported SWI-Prolog, install the Rust toolchain, obtain the prologai distribution for the platform, and run the installer, which produces the prologai executable bundling the engine, the compiled core, and all packs.

Launch the executable; confirm the banner identifies PrologAI with the engine attribution line, that the version and dialect flags report correctly, and that the lattice library loads, which together are the system acceptance criteria for a good install.


SECTION 6.2 - CONFIGURATION

The operator configures a deployment through the persistent configuration store.

Select the vector backend (the bake-off winner by default, a fallback if needed); select the embedding provider (local by default for privacy, a learned local model for genuine semantic similarity, an external service only by opt-in).

Initialize the constitutional layer at startup and register the human overseers and their authority levels, so corrigibility is live before the mind does anything.

Set the activation window, the actor intervals and thresholds, and the attention-economy money supply to suit the deployment.


SECTION 6.3 - OPERATING A RUNNING MIND

Observability is read from meta-control: usefulness, throughput, queue depth, error counts, and consciousness-indicator coverage, sampled at adaptive rates that never starve the system.

Durability is the journal: the write-ahead log replays cleanly after a crash, and lattice_verify confirms integrity by hash.

Safety in operation is corrigibility: a signed shutdown executes within seconds and cannot be blocked by the mind's own reasoning, and the constitutional veto log is immutable and reviewable.

Upgrades follow the no-fork procedure: adopt the engine upgrade, reinstall the packs, re-run the test levels, and confirm the dividend before returning the mind to service.


SECTION 6.4 - MIND SERVER OPERATION

To run many minds, deploy the Mind Server through the Model Context Protocol gateway: it registers devices, provisions mind instances, persists each mind's profile between sessions, and hosts durable agent mail.

Choose a topology per deployment: individual, unity, joint, or hive.

Each provisioned mind keeps its own Lattice, its own constitution instance, and its own overseers, and inbound tasks and mail always land in a quarantine scope before anything is trusted.

PART COMP-11 - RESPONSIBLE RELEASE AND THE OPEN-SUBSTRATE STANCE

This part states how completion is reconciled with responsibility, given that the platform is both open and aimed at the north stars of Specification Part 38.

What is released is a substrate, not a trained frontier model: PrologAI is general-purpose infrastructure - a language and platform - whose every mind inherits the constitutional layer, corrigibility, auditability, and the capability-evaluation gating by construction.

Release is capability-aware: the buildable core and the cognitive packs ship openly, while any component that materially raises a dangerous capability is disclosed only behind the if-then discipline of the Refinement volume, with the safe configuration as the documented default and the unsafe configuration requiring a deliberate, visible act.

The open-versus-closed question for powerful AI is genuinely contested, and this project records its reasoning rather than asserting a verdict: openness is judged net-positive here because an inspectable symbolic substrate is more auditable than a closed model, because many independent eyes harden the safety machinery, and because the constitutional safety travels with the substrate; the honest counterweight is that an open substrate can be misconfigured, which the protected-core discipline, the safe defaults, and this responsible-release posture are designed to mitigate.

Responsible release is therefore a standing commitment, re-evaluated at every version as capabilities grow: the more capable the substrate becomes, the more its disclosure is gated by measured evaluation rather than by default openness, and corrigibility and the constitution are never among the things a release makes optional.


PART COMP-12 - SHIPPING TOOLS AND CONNECTORS

A tool-equipped mind ships with more than the engine and the packs; it ships with a tool registry, the connectors that populate it, and the discipline that governs them.

The distribution includes the tool-use faculty (PR 44) and the computer-use body (PR 45), and a starter registry populated from the affordance ontology of Specification Part 39, with each tool carrying its tool_card and risk class.

Connectors are distributed the way the wider ecosystem already distributes them: Model Context Protocol servers for data sources and applications, Agent2Agent skills for inter-agent collaboration, native foreign predicates for performance-critical tools, and body capabilities for devices and computer use; adding a connector is registration, not a new release of the core.

Tool disclosure follows the responsible-release posture of the previous part: low-risk tools ship enabled, moderate-risk tools ship with the constitutional check engaged, and high-risk tool classes (payments, privileged shell, real-world computer and robotic control, and tool self-modification) ship disabled by default and are enabled only behind capability evaluation, oversight, and staged autonomy.

Operating a tool-equipped mind adds three runbook duties to the deployment guide: curate the registry to the tools a deployment actually needs, scope each connector's authorization to the minimum it requires, and keep computer use sandboxed until its reliability record justifies wider autonomy.

SPARC DOCUMENTATION CHANGE RULE - COMPLETION OBLIGATION

The standing SPARC documentation rule (Specification Section 41.2) is a completion obligation: no code change is complete until the corresponding documentation change is committed. This obligation applies to every future pull request on both PrologAI and Mentova.

The completion checklist for any code change now includes:

  Code: the work package code is committed and tested on a feature branch.
  Documentation: the relevant SPARC volumes (1 through 5) are updated with a description of the change and the reason for it.
  Version: each updated SPARC volume has its minor version incremented.
  CLAUDE.md: the project CLAUDE.md references the new document versions.
  PR: the code change and documentation change are in the same pull request or in an explicitly linked documentation PR.

Reason: documentation that trails the code is documentation that cannot be trusted. The standing rule ensures the five SPARC volumes remain the authoritative description of the platform at every point in its development history, not only at the moment of initial specification.

TUTORIAL READINESS AS A COMPLETION CRITERION

Beginning with Acc_68, tutorial readiness is a completion criterion for the Mentova platform. The platform is not complete for a given release if a new user cannot start an interactive session with a single command and immediately issue the tutorial's first query.

The console REPL launcher is the infrastructure for that criterion. Future improvements to tutorial readiness (the friendlier command interface, a built-in tutorial script, annotated example sessions) are tracked as extensions of this criterion.

SECTION 19.3 - WHAT CHANGED AND WHY

What changed: a new pack (pack lattice_crypto) introduces hybrid encryption as a cross-cutting service. Three algorithm families are supported: RSA-OAEP, ECDH on prime256v1, and ML-KEM-768 (PQC). All encrypt a payload with AES-256-GCM and wrap the session key asymmetrically.

Why: the glass-box transparency of the Lattice is a strength for reasoning, but it creates a risk for sensitive data. Any process that can read the Lattice journal can read stored plaintext values. The privacy layer closes that gap: sensitive values are encrypted before entering the Lattice and decrypted only by a holder of the private key. This also enables inter-mind confidential payloads over the four interoperability protocols (MCP, A2A, ACP, ANP).

Why PQC now: NIST finalized FIPS 203 (ML-KEM) in August 2024. Organizations with long-lived secrets (stored today, possibly readable in a decade when quantum computers mature) benefit from encrypting those secrets with ML-KEM now. The pack makes PQC a drop-in choice alongside RSA and ECDH.

Why hybrid: direct asymmetric encryption of large payloads is slow and limited by key size. Hybrid encryption is the universally recommended approach: the fast AES cipher handles the payload; the asymmetric algorithm handles only the small session key. AES-256-GCM adds authenticated integrity, catching both decryption with the wrong key and active tampering.

What was done: Wave 18 of the ARC-AGI-1 named-rule library added five new rules, fixed a critical operator precedence bug that was causing the benchmark to hang, and applied six performance guards to previously unbounded rules.

The critical bug fix: scale_shape_into_frame had a malformed if-then-else condition in its output builder. The condition was written:
( AR =:= FMinR ; AR =:= FMaxR ; AC =:= FMinC ; AC =:= FMaxC -> OV = FrameV ; interior )

Due to Prolog operator precedence (;/2 at 1100 outer-most, ->/2 at 1050 inner), this parses as:
( AR =:= FMinR ; (AR =:= FMaxR ; (AC =:= FMinC ; (AC =:= FMaxC -> OV = FrameV ; interior))) )

The first three disjuncts succeed without binding the output variable OV. OV remained free, meaning arc_transform generated infinitely many solutions when backtracked into. arc_fits_all, the training-pair verifier, backtracks into arc_transform whenever a computed output does not match the expected output. With scale_shape_into_frame generating infinite solutions, arc_fits_all never returned for any task where this rule was tried and failed. This blocked evaluation of all subsequent rules for those tasks and blocked the benchmark from advancing to later tasks, preventing discovery of 63 correct solutions.

The fix: add parentheses around the compound condition and add a determinism cut (!) before the output builder maplist. The corrected condition is:
( (AR =:= FMinR ; AR =:= FMaxR ; AC =:= FMinC ; AC =:= FMaxC) -> OV = FrameV ; interior )

The six performance fixes (size guards added):
read_color_grid_from_bands: duplicate four-way definition caused 16 rule attempts per task; non-deterministic helper predicates caused backtracking. Fixed with sort() deduplication, once() wrappers, and a ! cut.
fill_max_zero_rect_6: O(NR^2 * NC^2) rectangle enumeration. Guard: NR =< 15, NC =< 15.
fill_component_bbox_with_7: exponential DFS for disconnected cell pairs. Guard: NNZ =< 15.
crop_to_colored_rect_border: O(NR * NC) per-cell iteration on large grids. Guard: NR =< 20, NC =< 20.
scale_up_by_color_count: large output dimensions on large inputs. Guard: NR =< 15, NC =< 15.
draw_cross_with_intersection: no size guard previously. Guard: NR =< 20, NC =< 20.

The five new named rules added in wave 18 (did not solve any task in this run but are in the library):
intersect_halves_at_5, recolor_1_to_2_extend_row_period, mark_l_shape_missing_corner, fill_hole_in_repeating_pattern, fill_span_row_col_with_2.

Design principle established: every arc_transform clause must be deterministic. Non-determinism in a named rule causes arc_fits_all to hang whenever the rule fails a training pair. Determinism is enforced by:
1. Ending every arc_transform clause with a ! cut before or at the maplist output builder.
2. Wrapping non-deterministic sub-goals in once().
3. Parenthesizing all compound conditions in if-then-else to prevent operator precedence misparse.
