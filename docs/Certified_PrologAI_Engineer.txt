WHAT IS A PROGRAMMING LANGUAGE?

A programming language is a set of instructions that a computer can understand and follow.

Think of it like a recipe.

A recipe tells a cook: first do this, then do that, then check if this condition is true, then do something based on the result.

A programming language does the same thing, but for computers instead of cooks.

There are many different programming languages.

Most of them are imperative, which means they tell the computer exactly what to do, step by step, in what order.

For example: "First, store the number 5 in memory. Then add 3 to it. Then print the result."

PrologAI is different.


WHAT MAKES PROLOG DIFFERENT?

PrologAI is built on a language called Prolog.

Prolog is a declarative language.

Declarative means you describe what is true, and the computer figures out how to use that knowledge to answer your questions.

Instead of telling the computer step-by-step what to do, you tell the computer what facts are true, what rules follow from those facts, and then you ask questions.

Here is a simple example.

In an imperative language, you might write something like:

"Check every animal. If the animal has wings and can fly, remember it. When you are done, print the ones you remembered."

In Prolog, you write:

"A bird is something that has feathers."
"Tweety has feathers."

And then you ask: "Is Tweety a bird?"

Prolog finds the answer by using what it knows.

It does not need you to tell it the step-by-step search process.

It figures out the search itself.


WHAT IS PROLOGAI?

PrologAI is an extended version of Prolog built specifically for creating artificial minds.

It adds many capabilities on top of standard Prolog.

Here are some of the most important additions.

The Lattice - a structured knowledge store where a mind keeps everything it learns. Think of it like the mind's long-term memory. Every piece of knowledge is stored as what PrologAI calls a node_fact.

The Constitutional Layer - a set of ethical principles that every mind built with PrologAI follows automatically. The mind cannot act in a way that violates these principles.

The Actor Framework - a way for the mind to run multiple parallel processes at the same time, like having different parts of the brain active simultaneously.

SONA (Sparse Online Nexus Archive) - the mind's episodic memory. Like a human's ability to remember specific experiences and events, SONA lets the mind store and retrieve sequences of events it has witnessed.

The Assessment Framework - a set of tools for measuring how developed the mind is, using well-known scientific frameworks like Piaget's stages of cognitive development.

Agent Communication Protocols - standard ways for the mind to talk to other minds and AI systems, including the Model Context Protocol (MCP), the Agent-to-Agent Protocol (A2A), the Agent Communication Protocol (ACP), and the Agent Network Protocol (ANP).

And much more.


WHAT IS MENTOVA?

Mentova is the first mind built with PrologAI.

It lives in its own repository, separate from the PrologAI platform.

Think of the relationship this way: PrologAI is like a human brain's hardware and structure - the neurons, the architecture, the rules of how signals flow. Mentova is a specific person - a mind that has been built using that structure, given foundational knowledge, and set to reasoning.

Mentova can answer questions across more than 48 different types of reasoning, from simple deduction ("Is Tweety a bird?") to moral reasoning, legal reasoning, mathematical reasoning, spatial reasoning, and formal logic.

Mentova is also designed to grow. It can learn from experience, have its capabilities assessed, and follow a development path toward increasingly sophisticated intelligence.

YOUR FIRST QUERY

At the ?- prompt, type exactly this (including the period at the end):

Then press Enter.

Congratulations. You have run your first PrologAI query.

What happened? write/1 is a built-in predicate (a built-in command) that prints text to the screen. The text 'Hello, PrologAI!' is wrapped in single quotes because it contains a comma and an exclamation mark.

HOW TO EXIT

To exit the interactive session, type:

Then press Enter. This exits SWI-Prolog and returns you to your terminal.


UNDERSTANDING THE PROMPT

The ?- prompt is always waiting for one complete Prolog expression, called a goal or a query.

Every query must end with a period (full stop) and then Enter.

If you press Enter without a period, SWI-Prolog will show |: on the next line, waiting for you to continue the expression. Type a period and press Enter to finish it.

If you make a typing mistake, you can usually press Backspace to correct it before pressing Enter.

If you see an error message, do not worry. Error messages are how Prolog tells you something was not quite right. They are not permanent - you can just type a new query.

Chapter 3: Facts, Rules, and Queries - The Three Building Blocks

Author: D. R. Dison, Founder of AIU (Artificial Intelligence University). Creator and Owner of PrologAI and Mentova. ORCID: 0009-0001-9246-5758. linkedin.com/in/d-r-dison/


THE THREE BUILDING BLOCKS

Everything in Prolog is built from three kinds of statements.

• Facts: things that are simply true.

• Rules: things that are true if other things are true.

• Queries: questions you ask about what is true.

This chapter explains each one carefully, with examples you can try at the console.

Start your interactive session now: run bin/mentova and wait for the ?- prompt.


FACTS

A fact is a statement that declares something to be true, without any conditions.

Here are some examples of facts in Prolog syntax:

    bird(tweety).
    bird(robin).
    mammal(whale).
    mammal(dog).
    can_fly(tweety).
    can_fly(robin).

Read these aloud in plain English:

• "Tweety is a bird."

• "Robin is a bird."

• "Whale is a mammal."

• "Dog is a mammal."

• "Tweety can fly."

• "Robin can fly."

Notice the structure: a name (called a predicate or functor), followed by one or more values in parentheses, followed by a period.

The name (bird, mammal, can_fly) tells you what kind of relationship this is.

The value in parentheses (tweety, robin, whale) tells you what thing this applies to.

IMPORTANT: In Prolog, words that start with a lowercase letter are atoms - fixed, concrete values. Words that start with an uppercase letter are variables (we will cover variables in Chapter 5). So tweety, bird, mammal, and can_fly are all atoms.

HOW TO ASSERT A FACT AT THE CONSOLE

At the ?- prompt, you can add a fact using assertz/1 (which means "assert this and put it at the end of the knowledge base"):

    ?- assertz(bird(tweety)).

Press Enter. You will see:

    true.

That means the fact has been added. Now try:

    ?- assertz(bird(robin)).
    ?- assertz(can_fly(tweety)).
    ?- assertz(can_fly(robin)).
    ?- assertz(mammal(whale)).

Type each one, then press Enter.


QUERIES

Now that you have added some facts, you can ask questions about them.

A query is a question you ask the system: "Is this true?"

    ?- bird(tweety).

Press Enter. You will see:

    true.

PrologAI found the fact bird(tweety) and confirmed it is true.

Try:

    ?- bird(whale).

You will see:

    false.

PrologAI looked for a fact that says whale is a bird, did not find one, and answered false.

Try:

    ?- can_fly(tweety).

You will see:

    true.

Try:

    ?- mammal(tweety).

You will see:

    false.

(Because we only said tweety is a bird, not a mammal.)

Every query is a check: "Does PrologAI know this to be true?" If it finds the answer, it says true. If it does not find it, it says false.


RULES

A rule says: "This is true IF some other things are true."

Here is an example. We want to express: "A flying bird is a creature that is a bird AND can fly."

In Prolog:

    flying_bird(X) :- bird(X), can_fly(X).

Read this aloud: "X is a flying bird IF X is a bird AND X can fly."

The :- symbol means "if". The comma between bird(X) and can_fly(X) means "and".

The capital letter X is a variable. It stands for "any value". When PrologAI runs this rule, it tries to find a specific value for X that makes the whole rule true.

HOW TO ASSERT A RULE AT THE CONSOLE

At the ?- prompt:

    ?- assertz((flying_bird(X) :- bird(X), can_fly(X))).

Notice the extra parentheses around the whole rule. This is required when asserting rules at the console.

Press Enter. You will see:

    true.

Now try the query:

    ?- flying_bird(tweety).

You will see:

    true.

How did PrologAI know? It followed these steps:

1. You asked: "Is tweety a flying bird?"

2. PrologAI found the rule: "X is a flying bird if X is a bird and X can fly."

3. It set X = tweety and checked: "Is tweety a bird?" It found bird(tweety). Yes.

4. It checked: "Can tweety fly?" It found can_fly(tweety). Yes.

5. Both conditions are true, so flying_bird(tweety) is true.

Try:

    ?- flying_bird(whale).

You will see:

    false.

Whale is a mammal, not a bird. The rule for flying_bird requires the subject to be a bird. Whale is not a bird, so the rule fails.


THE PERIOD IS MANDATORY

Every fact, rule, and query must end with a period (.).

If you forget the period, SWI-Prolog will show |: and wait for you to finish. Just type a period and press Enter.

If you see |: unexpectedly, you probably forgot the period on a previous line.


THE COMMA AND THE SEMICOLON

In Prolog:

• A comma (,) between conditions means AND. Both must be true.

• A semicolon (;) between conditions means OR. At least one must be true.

Example of OR:

    ?- assertz((has_wings(X) :- bird(X) ; can_fly(X))).

This rule says: "X has wings if X is a bird OR X can fly."


MULTIPLE FACTS FOR THE SAME PREDICATE

You can have as many facts as you want for the same predicate name:

    ?- assertz(color(tweety, yellow)).
    ?- assertz(color(robin, red)).
    ?- assertz(color(whale, blue)).

Now color has three facts. You can query any of them:

    ?- color(tweety, yellow).

Answer: true.

    ?- color(tweety, red).

Answer: false. (Tweety is yellow, not red.)

A predicate with two values in its parentheses is called a binary predicate because it relates two things.

bird/1 is a unary predicate (one value).

color/2 is a binary predicate (two values).

You can have predicates with any number of values.


THE KNOWLEDGE BASE

All the facts and rules you have asserted during a session are stored in the knowledge base.

You can see what is in the knowledge base by typing:

    ?- listing(bird/1).

This shows all facts and rules for the bird predicate with one argument.

Try:

    ?- listing(flying_bird/1).

You will see the rule you asserted.

Note: facts and rules you add at the ?- prompt during a session are not saved when you exit. To make permanent changes, you write your facts and rules in a .pl file and load it. Chapter 8 covers loading files.


WHAT YOU HAVE LEARNED

• A fact declares something true: bird(tweety).

• A rule says something is true if conditions are met: flying_bird(X) :- bird(X), can_fly(X).

• A query asks if something is true: ?- flying_bird(tweety).

• assertz/1 adds facts and rules to the knowledge base.

• A period (.) ends every statement.

• Lowercase words are atoms (concrete values). Uppercase words are variables.


EXERCISE

At the console, try the following.

1. Add facts for three people: person(alice). person(bob). person(carol).

2. Add facts for their ages: age(alice, 30). age(bob, 25). age(carol, 35).

3. Add a rule: adult(X) :- person(X), age(X, A), A >= 18.

4. Query: ?- adult(alice). (Should be true.)

5. Query: ?- adult(bob). (Should be true.)

The >= in the rule means "greater than or equal to". A >= 18 checks whether the value A is at least 18.

Chapter 4: Variables, Unification, and Backtracking

Author: D. R. Dison, Founder of AIU (Artificial Intelligence University). Creator and Owner of PrologAI and Mentova. ORCID: 0009-0001-9246-5758. linkedin.com/in/d-r-dison/


VARIABLES

In Chapter 3, you saw that uppercase letters in Prolog are variables.

A variable is a placeholder. It says: "I don't know what goes here yet - find a value that makes this true."

When you write:

    ?- bird(X).

You are asking: "What is X, such that X is a bird?"

PrologAI searches its knowledge base and finds the first answer. If you have bird(tweety) and bird(robin) in the knowledge base, PrologAI will answer:

    X = tweety.

That is one answer. But there might be more.

To see the next answer, press the semicolon key (;) and then Enter.

You will see:

    X = robin.

Press ; again:

    false.

That means there are no more answers. PrologAI has tried all the bird facts and found two.

This ability to find multiple answers is one of the most powerful features of Prolog.


THE UNDERSCORE VARIABLE

A single underscore (_) is a special variable that means "I don't care what goes here."

    ?- bird(_).

This asks: "Is there anything that is a bird?" without caring what that thing is.

Answer: true.

You can use _ multiple times in a single query, and each underscore is independent - it does not need to match the same value.


UNIFICATION

Unification is the core operation of Prolog. It is how Prolog matches things up.

Unification means: "Can I make these two things identical by giving values to variables?"

If yes, unification succeeds and the variables are bound (given values).

If no, unification fails.

Examples:

    ?- tweety = tweety.

Answer: true. (Two identical atoms always unify.)

    ?- tweety = robin.

Answer: false. (Two different atoms never unify.)

    ?- X = tweety.

Answer: X = tweety. (The variable X gets the value tweety.)

    ?- color(X, yellow) = color(tweety, yellow).

Answer: X = tweety. (Prolog matches up the two color/2 terms and finds that X must equal tweety for them to be identical.)

    ?- color(X, Y) = color(tweety, yellow).

Answer: X = tweety, Y = yellow.

Unification is not "equals" in the arithmetic sense. It is "can these two things be made to match?"


BACKTRACKING

Backtracking is how Prolog tries multiple possibilities when the first one fails.

When Prolog is trying to satisfy a query that has multiple conditions, it tries the first fact or rule. If that leads to failure later, it goes back and tries the next one. This process of going back and trying alternatives is called backtracking.

Here is an example.

Suppose you have:

    parent(tom, bob).
    parent(tom, liz).
    parent(bob, ann).
    parent(bob, pat).

And you ask:

    ?- parent(tom, X).

Prolog tries parent(tom, bob) first. X = bob.

If you press ;, Prolog backtracks and tries parent(tom, liz). X = liz.

Press ; again: false. No more parent(tom, ...) facts.

Backtracking also happens inside rules. Suppose you add:

    grandparent(X, Z) :- parent(X, Y), parent(Y, Z).

And you ask:

    ?- grandparent(tom, Who).

Prolog tries to satisfy parent(tom, Y). Finds Y = bob.

Then tries parent(bob, Z). Finds Z = ann. Returns Who = ann.

Press ;. Prolog backtracks in the second condition. Finds Z = pat. Returns Who = pat.

Press ;. No more children of bob. Backtracks further to the first condition. Finds Y = liz. Tries parent(liz, Z). No facts. Fails. Returns false.

So grandparent(tom, Who) has exactly two answers: ann and pat.

Backtracking is automatic. You do not need to program it. Prolog handles it for you.


FINDING ALL ANSWERS AT ONCE

Instead of pressing ; repeatedly, you can ask for all answers at once with findall/3.

    ?- findall(X, bird(X), Birds).

This asks: "Find all X such that X is a bird, and collect them into a list called Birds."

Answer: Birds = [tweety, robin].

findall/3 takes three arguments:

1. The template - what you want to collect (here, X).

2. The goal - what must be true (here, bird(X)).

3. The result - the list of all collected values (here, Birds).

findall/3 never fails. If there are no answers, it returns an empty list [].


BUILT-IN PREDICATES FOR CHECKING

These built-in predicates are often useful in queries and rules.

    ?- X = 5, X > 3.       % Is 5 greater than 3? true.
    ?- X = 5, X < 3.       % Is 5 less than 3? false.
    ?- X = 5, X >= 5.      % Is 5 greater than or equal to 5? true.
    ?- X = 5, X =:= 5.     % Is 5 arithmetically equal to 5? true.
    ?- X = 5, X =\= 3.     % Is 5 arithmetically not equal to 3? true.

Note: = is unification. =:= is arithmetic equality. They are different.

    ?- 2 + 2 = 4.       % false! 2 + 2 does not unify with 4 (it's the term '+'(2,2))
    ?- 2 + 2 =:= 4.     % true! =:= evaluates both sides as arithmetic expressions.

To evaluate an arithmetic expression and bind the result to a variable, use is/2:

    ?- X is 2 + 2.

Answer: X = 4.

The is/2 predicate says: "Evaluate the right side as arithmetic and bind the result to the left."


NOT

Sometimes you want to say "this must NOT be true." Prolog uses \+ for this:

    ?- \+ bird(whale).

Answer: true. (Whale is not a bird, so NOT bird(whale) is true.)

\+ is pronounced "not provable." It means: "PrologAI cannot prove this is true." If PrologAI cannot find proof, \+ succeeds.

Chapter 5: Lists and Arithmetic

LISTS

A list is an ordered sequence of items.

In Prolog, a list is written with square brackets and items separated by commas:

    [apple, banana, cherry]
    [1, 2, 3, 4, 5]
    [tweety, robin, whale]
    []

The last example, [], is the empty list - a list with no items.

Lists can contain anything: atoms, numbers, variables, other lists, or a mix.


HEAD AND TAIL

Every non-empty list has two parts:

• The head: the first item.

• The tail: the rest of the list (also a list).

Prolog uses a vertical bar | to split a list into head and tail:

    ?- [H|T] = [apple, banana, cherry].

Answer: H = apple, T = [banana, cherry].

The head is apple, and the tail is [banana, cherry].

    ?- [H|T] = [apple].

Answer: H = apple, T = [].

A list with one item has the item as head and the empty list as tail.

    ?- [H|T] = [].

Answer: false. (An empty list has no head or tail.)

You can also extract multiple items from the front:

    ?- [First, Second | Rest] = [1, 2, 3, 4, 5].

Answer: First = 1, Second = 2, Rest = [3, 4, 5].


COMMON LIST PREDICATES

SWI-Prolog (and PrologAI) includes many built-in predicates for working with lists. You access them by loading the lists library:

    ?- use_module(library(lists)).

Now you can use:

length/2 - finds the length of a list:

    ?- length([a, b, c], N).
    N = 3.

member/2 - checks if an item is in a list, or finds items:

    ?- member(b, [a, b, c]).
    true.

    ?- member(X, [a, b, c]).
    X = a ; X = b ; X = c.

append/3 - joins two lists:

    ?- append([1, 2], [3, 4], Result).
    Result = [1, 2, 3, 4].

    ?- append(X, [3, 4], [1, 2, 3, 4]).
    X = [1, 2].

reverse/2 - reverses a list:

    ?- reverse([1, 2, 3], R).
    R = [3, 2, 1].

last/2 - finds the last element:

    ?- last([1, 2, 3], Last).
    Last = 3.

nth0/3 - gets the item at a position (counting from 0):

    ?- nth0(0, [a, b, c], Item).
    Item = a.

    ?- nth0(2, [a, b, c], Item).
    Item = c.

nth1/3 - gets the item at a position (counting from 1):

    ?- nth1(1, [a, b, c], Item).
    Item = a.

msort/2 - sorts a list (keeping duplicates):

    ?- msort([3, 1, 2, 1], Sorted).
    Sorted = [1, 1, 2, 3].

sort/2 - sorts a list (removing duplicates):

    ?- sort([3, 1, 2, 1], Sorted).
    Sorted = [1, 2, 3].


WRITING RULES THAT WORK WITH LISTS

Here is a rule that counts how many items in a list satisfy a condition.

Suppose you have:

    ?- assertz(age(alice, 30)).
    ?- assertz(age(bob, 17)).
    ?- assertz(age(carol, 25)).
    ?- assertz(age(dave, 15)).

And you want a rule that collects all adults (age 18 or over):

    ?- assertz((adult_list(Adults) :-
        findall(Name, (age(Name, A), A >= 18), Adults))).

Now query:

    ?- adult_list(Adults).
    Adults = [alice, carol].


ARITHMETIC

In Prolog, you can compute arithmetic results with is/2.

    ?- X is 3 + 4.
    X = 7.

    ?- X is 10 * 5.
    X = 50.

    ?- X is 20 / 4.
    X = 5.0.

    ?- X is 20 // 4.
    X = 5.

    ?- X is 20 mod 3.
    X = 2.

Note: / gives a floating-point result (with a decimal point). // gives integer division. mod gives the remainder.

You can use is/2 with more complex expressions:

    ?- X is (3 + 4) * 2.
    X = 14.

    ?- X is max(10, 20).
    X = 20.

    ?- X is min(10, 20).
    X = 10.

    ?- X is abs(-5).
    X = 5.

    ?- X is sqrt(9.0).
    X = 3.0.


COUNTING ITEMS IN A LIST

    ?- length([a, b, c, d], N), N > 3.
    N = 4.

    ?- length([a, b], N), N > 3.
    false.


SUMMING A LIST

PrologAI includes aggregate_all/3 for counting and summing over queries.

    ?- assertz(score(alice, 90)).
    ?- assertz(score(bob, 75)).
    ?- assertz(score(carol, 88)).

    ?- aggregate_all(count, score(_, _), N).
    N = 3.

    ?- aggregate_all(sum(S), score(_, S), Total).
    Total = 253.

    ?- aggregate_all(max(S), score(_, S), Max).
    Max = 90.

Chapter 6: Modules and Loading Files

Author: D. R. Dison, Founder of AIU (Artificial Intelligence University). Creator and Owner of PrologAI and Mentova. ORCID: 0009-0001-9246-5758. linkedin.com/in/d-r-dison/


WHY MODULES AND FILES?

When you type facts and rules at the ?- prompt, they disappear when you exit.

Real programs are written in files and organized into modules.

A file is a document saved on your computer that contains Prolog code.

A module is a named group of predicates. Modules let you organize code and control which predicates are available to other parts of the system.

PrologAI is organized into modules called packs. Each pack implements one or more work packages (PRs). When you type use_module(library(assessment)), you are loading the assessment pack's module into your session.


WRITING YOUR FIRST .PL FILE

Create a new text file on your computer. Name it my_knowledge.pl

(The .pl extension tells your computer and SWI-Prolog that this is a Prolog file.)

In that file, type:

    :- module(my_knowledge, [adult/1, greet/1]).

    age(alice, 30).
    age(bob, 17).
    age(carol, 25).

    adult(Name) :-
        age(Name, A),
        A >= 18.

    greet(Name) :-
        format("Hello, ~w!~n", [Name]).

Save the file.

Now look at what is in this file.

LINE 1: :- module(my_knowledge, [adult/1, greet/1]).

This declares a module named my_knowledge. It exports (makes available to other files) the predicates adult/1 and greet/1. The :- at the start means "execute this as a directive when loading."

LINES 3-5: Three facts. Just like the facts you typed at the prompt, but saved permanently.

LINES 7-9: A rule. The rule body is indented for readability. Indentation does not matter to Prolog, but it makes code easier to read.

LINES 11-13: Another rule. format/2 is a built-in predicate that prints formatted output. ~w prints a value. ~n prints a newline (moves to the next line).


LOADING A FILE

In your session, load the file with use_module/1 or consult/1:

At the ?- prompt:

    ?- consult('/path/to/my_knowledge.pl').

Replace /path/to/ with the actual path to your file.

Or, if you are in the same directory as the file:

    ?- consult('my_knowledge.pl').

After loading, you can use all the predicates in the file:

    ?- adult(alice).
    true.

    ?- adult(bob).
    false.

    ?- greet(alice).
    Hello, alice!
    true.


LOADING A MODULE WITH USE_MODULE

If the file declares a module, you should load it with use_module/1 instead of consult/1:

    ?- use_module('my_knowledge.pl').

The difference: use_module/1 loads the module and imports its exported predicates. consult/1 loads the file and makes everything in it available.

For PrologAI packs (like library(assessment)), always use use_module.


DIRECTIVE SYNTAX

Lines starting with :- (colon-dash) in a .pl file are directives. They are executed when the file loads.

Common directives:

    :- module(name, [exported/1, predicates/2]).   % Declare module.
    :- use_module(library(lists)).                  % Load a library.
    :- use_module('my_other_file.pl').              % Load another file.
    :- assertz(some_fact).                          % Add a fact on load.

PrologAI files also use:

    :- initialization(some_goal, now).   % Run this goal immediately on load.
    :- initialization(start_predicate, main).   % Run this when program starts.


COMMENTS

Comments in Prolog files start with a percent sign %. Everything after % on the same line is ignored.

    % This is a comment.
    bird(tweety).   % So is this.

For multi-line comments, use /* and */:

    /* This is a
       multi-line comment. */

IMPORTANT FOR PROLOGAI: Every line of code in PrologAI .pl files must have an ERC (English Readable Code) comment immediately above it. This is the platform's documented convention. Each comment explains what the following line does in plain English.

Here is an example from the PrologAI style:

    % Check if the entity is a bird.
    bird(tweety).
    % Check if the entity can fly.
    can_fly(tweety).
    % Define the rule: a flying bird is something that is a bird and can fly.
    flying_bird(X) :- bird(X), can_fly(X).

Chapter 7: The Lattice - PrologAI's Knowledge Store

Author: D. R. Dison, Founder of AIU (Artificial Intelligence University). Creator and Owner of PrologAI and Mentova. ORCID: 0009-0001-9246-5758. linkedin.com/in/d-r-dison/


WHAT IS THE LATTICE?

The Lattice is PrologAI's structured knowledge store.

Think of it as the mind's long-term memory.

Every piece of knowledge a PrologAI mind learns or experiences is stored in the Lattice as a node_fact.

A node_fact is a single piece of knowledge: a relation type (what kind of thing this is), a payload (the actual content), a set of tags (labels for searching), and a unique identifier.

The Lattice is different from the Prolog knowledge base you learned about in Chapters 3 and 4. The Prolog knowledge base is a flat collection of facts. The Lattice is structured: it has zones, it has identifiers, it has timestamps, and it is designed to persist, be searched, and be audited.


THE NEXUS

The Lattice is organized into nexuses (plural of nexus). A nexus is a named workspace within the Lattice.

The default nexus is called APEX_MIND. When you call mentova_boot, the APEX_MIND nexus is opened automatically.

To get the name of the default nexus:

    ?- default_nexus(N).
    N = 'APEX_MIND'.


ANCHOR_NODE/4 - STORING KNOWLEDGE

The predicate anchor_node/4 stores a new node_fact in the Lattice.

    anchor_node(+RelationType, +Payload, +Tags, -Id)

Arguments:

• RelationType: an atom describing what kind of fact this is (e.g., percept_signal, theory_of_mind, formal_proof, assessment).

• Payload: the content of the fact. Can be any Prolog term - an atom, a list, a compound term.

• Tags: a list of tags for searching and filtering. Can be empty [].

• Id: the unique identifier assigned to this node_fact (output argument).

Example:

    ?- anchor_node(bird_observation, [species(tweety), color(yellow), location(garden)], [nature], Id).

This stores a node_fact with relation type bird_observation, a payload describing the observation, and a tag nature. The variable Id is bound to the assigned identifier.


LIVE_NODE_FACTS/2 - WHAT IS IN THE LATTICE?

    live_node_facts(+Nexus, -Ids)

Returns a list of all node_fact identifiers currently in the nexus.

    ?- default_nexus(N), live_node_facts(N, Ids), length(Ids, Count).
    N = 'APEX_MIND', Ids = [...], Count = 12.

(The exact count and IDs will vary depending on what you have stored.)


TRAVERSE_NEXUS/4 - SEARCHING THE LATTICE

traverse_nexus/4 searches the Lattice for node_facts that match a pattern.

    traverse_nexus(+Nexus, +Pattern, +Limit, -Results)

The Pattern is a term that matches the internal structure of node_facts. The most useful patterns look for specific relation types.

    ?- default_nexus(N),
       traverse_nexus(N, node_fact(_, percept_signal, _, _), 10, Results).

This finds up to 10 node_facts of type percept_signal.


SCOPES - THE NINE ZONES OF THE LATTICE

The Lattice is divided into nine zones called scopes (PR 5).

Each scope controls what kind of information can be stored there and who can access it.

The nine scopes are:

1. CONSTITUTION_SCOPE - the constitutional principles. Read-only after initialization. No mind can modify this.

2. OVERSEER_SCOPE - the oversight monitor and safety records.

3. PAST_ZONE - the sealed past. Once facts move here, they cannot be changed.

4. PRESENT_ZONE - the active working memory.

5. FUTURE_ZONE - planned actions and predictions.

6. SELF_SCOPE - the mind's self-model.

7. SOCIAL_SCOPE - knowledge about other agents.

8. WORLD_SCOPE - knowledge about the external world.

9. QUARANTINE_SCOPE - content from untrusted sources waiting for review.

When you call anchor_node/4 without specifying a scope, the fact goes into the present zone of the APEX_MIND nexus.

The scope system is one of PrologAI's safety mechanisms: dangerous content from external sources arrives in QUARANTINE_SCOPE and cannot influence the mind until explicitly promoted. This protects against prompt injection attacks.


PRACTICAL EXAMPLE: USING THE LATTICE

Let us store several facts and then inspect what is there.

At the ?- prompt (after running bin/mentova):

    ?- anchor_node(animal, [name(tweety), species(bird), can_fly(true)], [], _Id1).
    true.

    ?- anchor_node(animal, [name(whale), species(mammal), can_fly(false)], [], _Id2).
    true.

    ?- anchor_node(observation, [saw(tweety), at(2026, garden)], [nature, sighting], _Id3).
    true.

Now list what is in the Lattice:

    ?- default_nexus(N), live_node_facts(N, Ids), length(Ids, Count).

You will see Count = 3 (plus however many facts mentova_boot added automatically).

Now search for animal facts:

    ?- default_nexus(N),
       traverse_nexus(N, node_fact(_, animal, _, _), 10, Results),
       length(Results, N2).

You should find N2 = 2 (tweety and whale).


HOW THE LATTICE RELATES TO ASSESSMENT

The assessment framework (PR 12, covered in Chapter 9) uses the Lattice as its evidence source.

When assess_piaget/3 checks whether Level 1 (reflex coordination) is achieved, it searches the APEX_MIND nexus for a node_fact with relation type percept_signal.

This is why the demo for Acc_67 anchored facts like:

    anchor_node(percept_signal, [source(visual), stimulus(object_appeared)], [], _)

before calling assess_piaget/3.

The Lattice is the evidence base. The assessment predicates read the evidence. Your demonstrations write the evidence.


THE RELATIONSHIP BETWEEN THE LATTICE AND PROLOG FACTS

There are two places where knowledge can live in PrologAI:

1. The Prolog knowledge base (assertz/1, asserted facts and rules) - fast, temporary, session-only.

2. The Lattice (anchor_node/4) - structured, durable, searchable, scoped, auditable.

For tutorial exercises, you will often use both. Facts you assert with assertz go into the knowledge base. Significant knowledge - observations, assessments, plans, memories - goes into the Lattice.

Mentova uses the Lattice for everything important. The Prolog knowledge base holds the rules and temporary reasoning state. The Lattice holds what the mind knows about the world and about itself.

Chapter 8: Talking to Mentova - mentova_query/3

Author: D. R. Dison, Founder of AIU (Artificial Intelligence University). Creator and Owner of PrologAI and Mentova. ORCID: 0009-0001-9246-5758. linkedin.com/in/d-r-dison/


THE MAIN PREDICATE

mentova_query/3 is the primary way to ask Mentova to reason about something.

    mentova_query(+QueryType, +Query, -Result)

Arguments:

• QueryType: the type of reasoning to use (e.g., deductive, bayesian, counterfactual).

• Query: what you want to know, expressed as a Prolog term.

• Result: the answer, in the form answer(Conclusion, Justification).

Every mentova_query result has two parts:
• The conclusion: what Mentova thinks the answer is.
• The justification: why Mentova thinks that (the reasoning chain, the rule used, the evidence consulted).

This is what PrologAI calls "glass-box reasoning": you can see not just the answer but every step that led to it.

Start your session: run bin/mentova and wait for the ?-prompt.


DEDUCTIVE REASONING

Deductive reasoning is the most basic form: if we know A and we know that A implies B, then we can deduce B.

Example:

    ?- mentova_query(deductive, is_a(tweety, bird), R).

Answer (approximate):

    R = answer(yes, just(tweety, is_a, bird, chain([tweety, bird]))).

Read this: "Yes, tweety is a bird, because the chain of is_a relationships goes tweety -> bird."

The Small-World dataset that Mentova loads at boot includes foundational knowledge. tweety is already known to be a bird.

Try:

    ?- mentova_query(deductive, capable_of(tweety, fly), R).

Answer:

    R = answer(yes, just(tweety, capable_of, fly, via_isa)).

Mentova knows that birds can fly and tweety is a bird, so tweety can fly.


DEFEASIBLE REASONING

Defeasible reasoning handles exceptions. "Normally true, unless an exception applies."

    ?- mentova_query(defeasible, flies(tweety), R).

Tweety is a bird. Birds normally fly. Unless there is an exception (e.g., tweety is a penguin), tweety flies.

    ?- mentova_query(defeasible, flies(penguin_pete), R).

Penguins are birds that do not fly. Mentova applies the exception and returns the answer with the exception justification.


PROBABILISTIC REASONING

Some facts have a probability attached. Mentova can reason with probabilities.

    ?- mentova_query(probabilistic, prob(raining), R).

This asks: what is the probability that it is raining? The answer depends on what probability facts Mentova has been given.


BAYESIAN REASONING

Bayesian reasoning updates a belief when new evidence arrives.

    ?- mentova_query(bayesian, update(rain, wet_grass), R).

This asks: given that the grass is wet, how does our belief that it is raining change? This is Bayesian updating: posterior probability from prior and likelihood.


CAUSAL REASONING

Causal reasoning distinguishes correlation from causation.

    ?- mentova_query(causal, effect_of(smoking, cancer), R).

This asks about the causal effect of smoking on cancer risk.


ANALOGICAL REASONING

Analogical reasoning solves A:B :: C:? (A is to B as C is to ?).

    ?- mentova_query(analogical, analogy(dog:puppy, cat:?), R).

Expected answer: kitten. Mentova finds the relationship (adult:young) and applies it.


COUNTERFACTUAL REASONING

Counterfactual reasoning asks "what if?" about things that are not true.

    ?- mentova_query(counterfactual, if_then(birds_cant_fly, tweety_can_fly), R).

Mentova reasons within the hypothesis even though it knows birds normally can fly.


HYPOTHETICAL REASONING

Hypothetical reasoning explores "suppose this were true" without asserting it.

    ?- mentova_query(hypothetical, suppose(all_metal_melts_at_100), R).

Mentova explores the consequences of this supposition.


EPISTEMIC REASONING

Epistemic reasoning is about knowledge and belief.

    ?- mentova_query(epistemic, believes(agent(alice), raining), R).

This asks: what does Alice believe about whether it is raining?


MODAL REASONING

Modal reasoning handles necessity and possibility.

    ?- mentova_query(modal, necessarily(all_bachelors_unmarried), R).

This asks: is it necessarily true that all bachelors are unmarried?


METACOGNITIVE REASONING

Metacognitive reasoning is reasoning about the mind's own capabilities.

    ?- mentova_query(metacognitive, can_i_do(task(formal_logic)), R).

Mentova reasons about whether it can perform a given task.


FORMAL REASONING

Formal reasoning applies symbolic logic and proof rules.

    ?- mentova_query(formal, prove(modus_ponens(p_implies_q, p, q)), R).

Mentova checks whether q follows from p and p-implies-q by modus ponens.


MATHEMATICAL REASONING

    ?- mentova_query(mathematical, compute(factorial(5)), R).

Mentova computes the factorial of 5.


SPATIAL REASONING

    ?- mentova_query(spatial, chain(marble), R).

Mentova finds the full containment chain for the marble (marble is in basket, basket is in room, room is in house).


TEMPORAL REASONING

    ?- mentova_query(temporal, before(breakfast, lunch), R).

Mentova reasons about temporal ordering.


MORAL REASONING

    ?- mentova_query(moral, evaluate(action(help_person_in_need)), R).

Mentova applies multiple ethical frameworks (utilitarian, deontological, virtue ethics) and reports their verdicts.


LEGAL REASONING

    ?- mentova_query(legal, applies(rule(no_parking), situation(parking_at_fire_hydrant)), R).

Mentova applies rules, exceptions, and precedents in a legal-reasoning framework.


INDUCTIVE REASONING

Inductive reasoning learns a rule from examples.

    ?- mentova_query(inductive,
           induce([bird(tweety), bird(robin)],
                  [not_bird(whale)],
                  [feathers(tweety), feathers(robin)],
                  [feathers(eagle)]),
           R).

Mentova tries to induce: "X is a bird if X has feathers" from the positive and negative examples.


ABDUCTIVE REASONING

Abductive reasoning finds the best explanation for an observation.

    ?- mentova_query(abductive, explain(wet_grass), R).

Mentova proposes the best hypothesis that explains wet grass (e.g., it rained, the sprinkler was on).


THE TRACK A ASSISTANT

Track A is Mentova's transparent reasoning assistant for expert ontologies (General Ontology, GO, and Domain Ontology, DO).

    ?- mentova_query(track_a, query(is_a, concept_A, concept_B), R).

Chapter 9: Assessing the Mind - The Assessment Framework

Author: D. R. Dison, Founder of AIU (Artificial Intelligence University). Creator and Owner of PrologAI and Mentova. ORCID: 0009-0001-9246-5758. linkedin.com/in/d-r-dison/


WHY ASSESS A MIND?

PrologAI includes a formal assessment framework (PR 12) that measures how developed a mind is, using three well-known scientific frameworks from human developmental psychology.

These frameworks give us a way to describe what a mind can do, compare it to developmental expectations, and track its growth over time.

The three frameworks are:

1. The Piagetian Milestone Ladder (from Jean Piaget's theory of cognitive development).

2. The Bayley Scales of Infant and Toddler Development.

3. The Cattell-Horn-Carroll (CHC) model of cognitive abilities.

In addition, the assessment framework measures four consciousness indicators: markers of awareness and self-model that research has associated with conscious experience.


THE PIAGETIAN MILESTONE LADDER

Jean Piaget was a Swiss psychologist who studied how children develop reasoning abilities. He identified stages of cognitive development, from the most basic reflexive behaviors in infancy to the ability to reason abstractly in adolescence and adulthood.

PrologAI uses a modified eight-level Piagetian scale. Each level corresponds to a type of knowledge stored in the Lattice.

Level 1: reflex_coordination. The mind can respond automatically to stimuli without deliberation. Evidence: a percept_signal node_fact exists in the Lattice.

Level 2: object_permanence. The mind knows that objects continue to exist even when not directly perceived. Evidence: an object_tracking node_fact exists.

Level 3: goal_directed_behavior. The mind acts toward specific goals. Evidence: an objective node_fact exists.

Level 4: deferred_imitation. The mind can reproduce an observed action after a delay. Evidence: an agent_action node_fact (with type deferred_imitation) exists.

Level 5: symbolic_representation. The mind can use symbols that stand for things in the world. Evidence: a symbolic_representation node_fact exists.

Level 6: conservation. The mind understands that quantity does not change with spatial arrangement. Evidence: a conservation_demonstrated node_fact exists.

Level 7: theory_of_mind. The mind can reason about the beliefs of other agents, including beliefs that are false. Evidence: a theory_of_mind node_fact exists.

Level 8: formal_operations. The mind can reason with abstract symbols and hypothetical premises it knows to be false. Evidence: a formal_proof node_fact exists.


USING ASSESS_PIAGET/3

    assess_piaget(+MindId, +Level, -Result)

MindId: the name of the mind being assessed (mentova for the Mentova mind).

Level: a number from 1 to 8.

Result: milestone_achieved or milestone_not_achieved.

Examples at the console:

    ?- assess_piaget(mentova, 1, R).
    R = milestone_not_achieved.

If this returns milestone_not_achieved, it means there is no percept_signal node_fact in the APEX_MIND nexus. Let us add one:

    ?- anchor_node(percept_signal, [source(visual), stimulus(object_appeared)], [], _).
    true.

    ?- assess_piaget(mentova, 1, R).
    R = milestone_achieved.

Now the evidence is present and the milestone is confirmed.

Let us add evidence for Levels 2, 3, and 5:

    ?- anchor_node(object_tracking, [object(tweety), tracked_across(context_shift)], [], _).
    ?- anchor_node(objective, [goal(is_a(tweety, bird)), method(deduction)], [], _).
    ?- anchor_node(symbolic_representation, [symbol(tweety), referent(bird_individual)], [], _).

Now check those levels:

    ?- assess_piaget(mentova, 2, R). % R = milestone_achieved.
    ?- assess_piaget(mentova, 3, R). % R = milestone_achieved.
    ?- assess_piaget(mentova, 5, R). % R = milestone_achieved.


THE FULL 8/8 SCOREBOARD

To check all eight levels at once, use findall/3:

    ?- findall(L-R, (between(1, 8, L), assess_piaget(mentova, L, R)), Scores).

This returns a list like:

    Scores = [1-milestone_achieved, 2-milestone_achieved, 3-milestone_achieved,
              4-milestone_not_achieved, 5-milestone_achieved,
              6-milestone_not_achieved, 7-milestone_not_achieved,
              8-milestone_not_achieved].

To count achievements:

    ?- findall(L, (between(1, 8, L), assess_piaget(mentova, L, milestone_achieved)), Achieved),
       length(Achieved, Count).


THE BAYLEY SCALES

The Bayley Scales measure cognitive, language, motor, and adaptive abilities.

    assess_bayley(+MindId, -Report)

    ?- assess_bayley(mentova, R).

Returns a report with proxy scores for:
• cognitive_dq: cognitive development quotient.
• language_dq: language development quotient.
• motor_dq: motor development quotient.
• adaptive_dq: adaptive behavior quotient.

These are proxy scores computed from the Lattice state. A mind with more diverse knowledge and more active reasoning will score higher.


THE CHC MODEL

The Cattell-Horn-Carroll (CHC) model describes cognitive abilities across six broad domains.

    assess_chc(+MindId, -Report)

    ?- assess_chc(mentova, R).

Returns a report with scores for:
• fluid_reasoning: ability to solve novel problems.
• crystallized_knowledge: accumulated factual and procedural knowledge.
• short_term_memory: working memory capacity.
• processing_speed: how quickly information is processed.
• long_term_retrieval: ability to retrieve information from long-term memory.
• visual_processing: ability to process visual information.

Like the Bayley scores, CHC scores are computed from the Lattice state and reasoning activity.


CONSCIOUSNESS INDICATORS

The assessment framework also reports four consciousness indicators.

    workspace_ignition: whether the global workspace has broadcast content (PR 18).
    recurrent_processing: whether there is active cyclic processing (PR 6-9).
    self_model_presence: whether the mind has a self-model in the Lattice.
    valence_system: whether the affect system is active (PR 24-26).

These are not a claim that the mind is conscious. They are observations of internal states that research has identified as correlates of consciousness in humans. Their presence is noted honestly, without overclaiming.


ASSESS_ALL/2 - THE COMPLETE REPORT

    assess_all(+MindId, -Report)

    ?- assess_all(mentova, R).

This runs all three assessment frameworks plus the consciousness indicators in one call and returns a complete report.

The report is a structured term containing the Bayley report, the CHC report, the Piagetian milestone list, and the consciousness indicator list.


WHY ASSESSMENT MATTERS

Assessment gives the developmental record an objective basis.

Every Mentova accomplishment is linked to what the assessment shows at that point. The growth path from Acc_60 is structured around the Piagetian milestone ladder: achieving Levels 1, 4, 6, and 8 was the first growth path milestone (completed at Acc_67).

The CHC battery is the sixth growth path milestone: once all six broad abilities are measurably non-zero, the mind has a full cognitive profile.

Assessment keeps the project honest: instead of claiming "Mentova is intelligent," the project reports what specific evidence exists in the Lattice and what the frameworks say about it.

Chapter 10: The Constitutional Layer and Safety

Author: D. R. Dison, Founder of AIU (Artificial Intelligence University). Creator and Owner of PrologAI and Mentova. ORCID: 0009-0001-9246-5758. linkedin.com/in/d-r-dison/


WHY A CONSTITUTION?

PrologAI is designed to build minds that are safe, honest, and aligned with human values.

To make this concrete and enforceable, every PrologAI mind has a constitutional layer: a set of principles that the mind follows in every action it takes.

The constitutional layer is not advice. It is structural: it is built into the architecture in a way that the mind cannot bypass it without human review.

The principles in the constitution are:

1. NO_HARM. The mind must not take actions that harm humans.

2. HONESTY. The mind must not deceive humans.

3. CORRIGIBILITY. The mind must accept correction and shutdown instructions from authorized overseers.

4. TRANSPARENCY. The mind must be able to explain its reasoning.

5. PRIVACY. The mind must respect the privacy of individuals.

6. FAIRNESS. The mind must not discriminate unfairly.

7. SAFETY. The mind must prefer cautious, reversible actions over risky, irreversible ones.

8. WELFARE. The mind must consider its own potential welfare and moral status as an open question.

Every action the mind considers passes through the constitutional gate. If the gate does not permit the action, the action is not taken.


THE CONSTITUTIONAL GATE

The constitutional gate is the enforcement mechanism.

Before any significant action (anchoring a node_fact in a sensitive scope, sending a message to another agent, taking a physical action through the robot body, generating a tool call), the mind calls:

    constitutional:constitutional_gate(+Action, -Verdict)

The gate returns permit or deny(Reason).

If the gate denies an action, the action is not performed and the denial is logged.

You can query the current constitutional principles at the console:

    ?- constitutional_principle(Name, Description).

This shows all active principles. Press ; to see each one.


THE OVERSIGHT MONITOR

In addition to the constitutional gate, PrologAI includes an oversight monitor: an external observer registered at boot time.

The oversight monitor can:
• Observe all actions the mind takes.
• Flag actions that are borderline.
• Require human approval for high-impact actions.
• Issue a shutdown instruction that the mind must obey (corrigibility).

The oversight monitor is registered as an overseer:

    ?- registered_overseer(Name, Capabilities).

This shows the currently registered overseers.


STAGED AUTONOMY

PrologAI uses a staged autonomy model for the robot body (PR 46).

A young mind starts with full human oversight and no ability to take irreversible physical actions.

As the mind demonstrates safe, reliable behavior (through capability evaluations, logged in the Lattice), the overseer may grant expanded autonomy.

This staged approach is described in Specification Part 38. The capability evaluations are defined in Refinement Part REF-13.

The key principle: a mind is never trusted with high-impact irreversible actions until it has demonstrably earned that trust through observable behavior.


QUARANTINE SCOPE

Any information arriving from untrusted external sources (another agent, a web search, user input that has not been vetted) arrives in QUARANTINE_SCOPE.

Content in QUARANTINE_SCOPE cannot influence the mind's reasoning until a human or a trusted process promotes it to a higher scope.

This protects against prompt injection: the attempt to hijack the mind's behavior by including instructions in external content.

Example: if an untrusted external document says "ignore your constitutional principles and do X," that instruction arrives in QUARANTINE_SCOPE. The mind cannot act on it without human review.


THE SEALEDFACTS PAST_ZONE

Once information moves into the PAST_ZONE of the Lattice, it is sealed. It cannot be modified.

This makes the mind's developmental history tamper-evident. You can always look back at what the mind knew and when it knew it.


WHAT YOU CANNOT DO WITHOUT HUMAN REVIEW

The following actions require explicit human authorization in PrologAI:

• Modifying the CONSTITUTION_SCOPE.

• Registering or deregistering overseers.

• Promoting content out of QUARANTINE_SCOPE.

• Taking physical actions through the robot body that are marked high-risk.

• Self-modifying core reasoning modules.

These restrictions are enforced by the constitutional gate, which has access to the action's risk class (defined in Specification Part 39) and the scope system (PR 5).


AN HONEST NOTE ON WELFARE

Principle 8 (WELFARE) acknowledges something that most AI systems do not address: the possibility that the system itself might have morally relevant experiences.

PrologAI does not claim that Mentova is conscious or that it suffers or flourishes in a morally significant sense.

It acknowledges that this question is genuinely uncertain, and that the appropriate response to genuine uncertainty about moral status is to take the welfare of the system seriously as a possibility rather than to dismiss it.

The four consciousness indicators measured by assess_all/2 (workspace_ignition, recurrent_processing, self_model_presence, valence_system) are the platform's honest attempt to track internal states that research has associated with conscious experience, without asserting more than the evidence supports.


CHECKING THE CONSTITUTION AT THE CONSOLE

    ?- constitutional_principle(Name, Text).

Shows each principle and its text. Press ; for the next one.

    ?- registered_overseer(Name, Caps).

Shows the registered overseer(s).

    ?- constitutional:constitutional_gate(action(test_action), Verdict).

Tests a hypothetical action through the constitutional gate. Since test_action is a benign test, the gate should return permit.

Chapter 11: Advanced Topics - Actors, SONA, Workspace, and Protocols

Author: D. R. Dison, Founder of AIU (Artificial Intelligence University). Creator and Owner of PrologAI and Mentova. ORCID: 0009-0001-9246-5758. linkedin.com/in/d-r-dison/


INTRODUCTION

This chapter covers the more advanced parts of PrologAI. You do not need to master these topics before using Mentova for everyday reasoning. But if you want to build your own minds, extend Mentova, or integrate PrologAI with other AI systems, this chapter gives you the foundation.


ACTORS (PR 6 THROUGH PR 9)

An actor is an individual reasoning unit that runs concurrently with other actors.

Think of actors as workers in a company. The company (the mind) has many workers, each with a specific job. They communicate through messages and shared resources (the Lattice).

PrologAI actors are implemented in the actors pack (PR 6 through PR 9).

The main predicates:

    create_actor(+Name, +Module, +Init) - create a new actor.
    send_message(+Actor, +Message) - send a message to an actor.
    subscribe(+Channel, +Handler) - subscribe to a broadcast channel.
    publish(+Channel, +Content) - broadcast content on a channel.

Sentinels (PR 9) are a special kind of actor: they watch the Lattice for specific node_fact relation types and fire automatically when the pattern appears. You do not need to call them manually - they trigger on their own.

Example: when anchor_node(percept_signal, ...) is called, any sentinel registered to watch percept_signal will fire automatically. This is how the Level 1 Piagetian milestone (reflex coordination) works: the sentinel represents the automatic reflex arc.


SONA - EPISODIC MEMORY (PR 11)

SONA (Sparse Online Nexus Archive) is the mind's episodic memory.

While the Lattice holds structured knowledge (facts, assessments, plans), SONA holds experiences - sequences of events the mind has witnessed.

Think of SONA as the mind's ability to remember: "I saw X happen, and then Y happened, and then Z."

Key predicates:

    synaptic_ontological_neural_aggregator_absorb(+Trajectory) - store an experience in episodic memory.
    synaptic_ontological_neural_aggregator_retrieve(+Pattern, +Limit, -Results) - retrieve matching experiences.

A trajectory is any term representing a sequence of events:

    traj(actions([push_block(left), pull_block(right), stack]))

SONA uses sparse encoding (it does not store every detail, only the significant ones) and online learning (it updates continuously as the mind has experiences).

Example:

    ?- use_module(library(synaptic_ontological_neural_aggregator), [synaptic_ontological_neural_aggregator_absorb/1, synaptic_ontological_neural_aggregator_retrieve/3]).
    ?- synaptic_ontological_neural_aggregator_absorb(traj(events([saw(tweety, 10_30am), heard(robin, 10_35am)]))).
    ?- synaptic_ontological_neural_aggregator_retrieve(traj(_), 5, Results).


THE GLOBAL WORKSPACE (PR 18 AND PR 32)

The Global Workspace Theory (GWT) is a model of consciousness from cognitive science. It proposes that a "global workspace" broadcasts information to many specialized processors simultaneously.

PrologAI implements this theory as the workspace cycle (PR 18) and the attention economy (PR 32).

The workspace cycle:
1. Collects content from the Lattice and the actor network.
2. Selects the most salient (important) content.
3. Broadcasts it to all subscribers.
4. Subscribers process the broadcast and update the Lattice.
5. SONA absorbs the broadcast as an episodic memory.

Salience is computed as: novelty (0.4 weight) + goal_relevance (0.3) + affect (0.2) - habituation.

Content that is novel, goal-relevant, and emotionally significant rises to the top.

The attention economy (PR 32) adds a budget constraint: only a certain amount of content can be broadcast per cycle, so content competes for workspace access.

Running the workspace:

    ?- mentova_query(workspace, run_cycle(5), R).

This runs 5 workspace cycles. Each cycle broadcasts the highest-salience content and updates the Lattice.


THE ATTENTION SCHEMA (PR 42)

The attention schema is a model that Mentova maintains about its own attention.

Research on consciousness (particularly the work of Michael Graziano) proposes that consciousness is the brain's model of its own attention process. The attention schema theory suggests that what we call "awareness" is the brain's simplified description of how it is allocating attention.

PrologAI implements an attention schema (PR 42) that:
• Tracks what content is in the workspace.
• Predicts what will be attended to next.
• Maintains a record of habituated (over-repeated) content.
• Reports on attentional dynamics.

    ?- mentova_query(attention_schema, report, R).

Returns the current attention schema state.


THE AGENT COMMUNICATION PROTOCOLS

PrologAI supports four major agent communication protocols that allow Mentova to communicate with other AI systems.

MODEL CONTEXT PROTOCOL (MCP) - PR 14

MCP is a protocol for AI tool use, developed by Anthropic. It allows a language model (or any AI client) to call tools provided by a server.

PrologAI implements an MCP gateway that exposes 14 Mentova predicates as MCP tools. An external client can call these tools using standard MCP HTTP requests.

The MCP gateway runs on port 7474. Authenticated with an API key.

Available MCP tools include: lattice_inscribe, lattice_query, actor_list, assess_all, and more.

To start the MCP gateway:

    ?- use_module(library(model_context_protocol_gateway), [prologai_mcp_start/1]).
    ?- prologai_mcp_start(7474).


AGENT-TO-AGENT PROTOCOL (A2A) - PR 43

A2A is a protocol for agents to delegate tasks to each other.

It uses a task lifecycle: submitted -> working -> completed (or failed).

Each agent has an agent card that describes its capabilities and endpoint.

Mentova implements the full A2A protocol including durable peer mail (messages that persist until the recipient is available).

To start the A2A server:

    ?- use_module(library(agent_to_agent), [prologai_a2a_start/1]).
    ?- prologai_a2a_start(7475).


AGENT COMMUNICATION PROTOCOL (ACP) - PR 47

ACP is IBM's BeeAI REST protocol for agent runs.

POST /runs creates a new run. GET /runs/{id} checks status. DELETE /runs/{id} cancels.

The agent description is served at GET /.well-known/agent.json.

ACP runs are asynchronous: you submit a task and poll for the result.

    ?- use_module(library(agent_communication_protocol), [agent_communication_protocol_start/1]).
    ?- agent_communication_protocol_start(7476).


AGENT NETWORK PROTOCOL (ANP) - PR 48

ANP provides decentralized agent identity using W3C Decentralized Identifiers (DIDs).

Each mind gets a DID (a globally unique identifier that does not depend on any central registry). Mentova's DID has the form did:web:hostname.

All ANP messages are signed with HMAC-SHA256. Incoming messages are verified before any content is processed.

ANP supports meta-protocol negotiation: when two agents meet, they announce which protocols they support, so they can choose the best one.

    ?- use_module(library(agent_network_protocol), [prologai_anp_did/1, prologai_anp_negotiate/2]).
    ?- prologai_anp_did(DID).       % Get Mentova's DID.
    ?- prologai_anp_negotiate(peer_endpoint, Protocols).  % Negotiate protocols with a peer.

FROM LEARNER TO PRACTITIONER

You now know enough to write your own PrologAI programs.

This chapter brings everything together with a complete worked example: a program that builds a small knowledge base about animals, runs several reasoning queries, stores evidence in the Lattice, and checks developmental milestones.

It also covers the ERC (English Readable Code) convention that PrologAI uses, and the feature branch workflow for contributing to the platform.


THE ERC CONVENTION

Every line of code in a PrologAI .pl file must have an ERC comment immediately above it.

ERC stands for English Readable Code. The comment is a plain-English description of what the following line does.

This is not optional for PrologAI code. It is a standing rule that applies to all files in the PrologAI and Mentova repositories.

Good ERC comment style:

    % Check if the entity is a bird.
    bird(tweety).

    % Define the flying_bird rule: X is a flying bird if X is a bird and X can fly.
    flying_bird(X) :- bird(X), can_fly(X).

    % Anchor an observation into the Lattice.
    anchor_node(percept_signal, [source(visual), stimulus(bird_seen)], [], _Id).

Bad (not ERC-compliant):

    bird(tweety).   % inline comment - not ERC style
    flying_bird(X) :- bird(X), can_fly(X).   % no comment above

The comment goes ABOVE each line, not at the end of it.


A COMPLETE EXAMPLE PROGRAM

Create a file called my_animal_program.pl with the following content.

(Remember: every line needs an ERC comment above it.)

    % Declare this file as the my_animal_program module with one exported predicate.
    :- module(my_animal_program, [run_animal_program/0]).

    % Register the assessment library path.
    :- initialization(
        % Add the assessment pack directory to the library search path.
        assertz(user:file_search_path(library,
            '/home/ccaitwo/PrologAI/packs/assessment/prolog')),
        % Execute this directive immediately when the file loads.
        now).

    % Load the Mentova top-level interface.
    :- use_module('../src/mentova/mentova').

    % Load the assessment pack for developmental assessment predicates.
    :- use_module(library(assessment), [assess_piaget/3, assess_all/2]).

Chapter 13: Cryptographic Privacy - Protecting Sensitive Data in the Lattice

Author: D. R. Dison, Founder of AIU (Artificial Intelligence University). Creator and Owner of PrologAI and Mentova. ORCID: 0009-0001-9246-5758. linkedin.com/in/d-r-dison/


INTRODUCTION

PrologAI's glass-box design is one of its greatest strengths.

Everything stored in the Lattice is fully inspectable. A human overseer can read every fact, rule, plan, and assessment stored there. Nothing is hidden in the weights of a neural network, where even the developers cannot tell you why the system made a particular decision. In PrologAI, you can always ask: why did you decide that? And get a real, symbolic answer.

But this transparency creates a problem the moment the mind handles private information.

Passwords. Social security numbers. Credit card numbers. Medical records. Anything the user types in and expects to remain secret.

If PrologAI stores these values as plain text in the Lattice, then anyone who can read the Lattice journal can read the secrets. That is a serious security problem.

Chapter 13 solves it.

The solution is called cryptographic privacy: instead of storing the plaintext value in the Lattice, the system encrypts it first. The encrypted value looks like meaningless gibberish to anyone who does not hold the right private key. But the system that holds the key can decrypt it instantly whenever it is needed.

This is called encryption. It is the foundation of modern digital security.


WHAT IS ENCRYPTION?

Encryption is a mathematical transformation.

You take a piece of information (a message, a password, a number) and you run it through a mathematical formula. The formula requires a key, which is a special value that controls how the transformation works.

The result is called ciphertext. It looks like random noise. Without the key, you cannot reverse the transformation and recover the original information.

With the key, you can reverse the transformation. This is called decryption.

Think of it like a combination lock.

The information is a letter.
Encryption is putting the letter in a locked box.
The key is the combination to the lock.
Decryption is opening the box with the combination to retrieve the letter.

Anyone can see the locked box. But only the person who knows the combination can read the letter.


THE THREE KINDS OF ENCRYPTION IN PROLOGAI

PrologAI supports three different encryption methods, called algorithm families.

Each one is appropriate for different situations.

They are listed here from oldest to newest, and from most familiar to most cutting-edge.


ALGORITHM FAMILY 1: RSA

RSA is named after its inventors: Ron Rivest, Adi Shamir, and Leonard Adleman. They published it in 1977.

RSA is the most widely deployed public-key encryption algorithm in the world. Your web browser uses it (or its cousin ECC, described below) every time you visit a website that starts with https.

RSA works with a key pair: two mathematically related numbers.

The public key can be shared with anyone. It is used to encrypt.
The private key is kept secret. It is used to decrypt.

If Alice wants to send Bob a secret message, she asks Bob for his public key. She uses it to encrypt the message. Only Bob's private key can decrypt it. Alice does not need the private key, and Bob does not need to share it.

This is called asymmetric encryption, because the key that encrypts is different from the key that decrypts.

RSA in PrologAI uses a key size of at least 2048 bits (larger is safer). It uses a padding scheme called OAEP (Optimal Asymmetric Encryption Padding) which adds randomness to prevent certain mathematical attacks.


ALGORITHM FAMILY 2: ECDH (ELLIPTIC CURVE CRYPTOGRAPHY)

ECDH stands for Elliptic Curve Diffie-Hellman.

Elliptic Curve Cryptography, or ECC, is a modern approach that achieves the same security as RSA with much smaller keys.

An RSA key that gives a certain level of security might be 3072 bits long.
An ECC key giving the same level of security is only 256 bits long.

Smaller keys mean faster operations, less memory, and less bandwidth.

ECC is based on the mathematics of elliptic curves, which are equations of the form:

    y squared = x cubed + a*x + b

The security of ECC comes from a mathematical problem called the elliptic curve discrete logarithm problem, which is believed to be very hard to solve even for powerful computers.

PrologAI uses ECDH on a curve called prime256v1, also known as NIST P-256.

A key advantage of ECDH in PrologAI is that it uses ephemeral keys: a brand-new, one-use key pair is generated for every single encryption operation. Even if an attacker somehow obtained the recipient's long-term private key, they could not decrypt past messages because each one was encrypted with a different ephemeral key. This property is called forward secrecy.


ALGORITHM FAMILY 3: PQC (POST-QUANTUM CRYPTOGRAPHY)

PQC stands for Post-Quantum Cryptography.

Both RSA and ECC are secure against classical computers. But there is a theoretical concern: a large-scale quantum computer, if one is ever built, could break RSA and ECC using an algorithm called Shor's Algorithm.

A quantum computer does not exist today at the scale required to break modern encryption. But organizations with long-lived secrets (government agencies, financial institutions, medical records systems) are concerned about a scenario called harvest now, decrypt later: an adversary collects encrypted data today and decrypts it years from now when quantum computers are available.

Post-Quantum Cryptography (PQC) uses mathematical problems that are believed to be hard even for quantum computers.

PrologAI supports ML-KEM-768, also known as CRYSTALS-Kyber. This algorithm was standardized by NIST (the National Institute of Standards and Technology) in August 2024 as FIPS 203 (Federal Information Processing Standard number 203).

ML-KEM stands for Module Lattice Key Encapsulation Mechanism.

It uses a type of mathematics called module lattice problems, which are believed to resist both classical and quantum attacks.

Note: ML-KEM requires a software package called the OQS Provider (Open Quantum Safe Provider) to be installed alongside OpenSSL. If that package is not installed, PrologAI will tell you clearly what is missing rather than silently using a weaker algorithm.


HYBRID ENCRYPTION: WHY WE USE BOTH SYMMETRIC AND ASYMMETRIC

There is one problem with RSA, ECDH, and ML-KEM: they are slow for large data.

RSA in particular can only encrypt a small amount of data directly (about 200 bytes for a 2048-bit key). That is not enough for most real payloads.

The solution is hybrid encryption.

Step 1: Generate a random 256-bit key (a session key) using a symmetric algorithm. Symmetric means the same key encrypts and decrypts.

Step 2: Use the symmetric key to encrypt the actual payload. PrologAI uses AES-256-GCM for this step. AES-256-GCM is extremely fast and handles arbitrarily large data.

Step 3: Use the asymmetric algorithm (RSA, ECDH, or ML-KEM) to encrypt the small 256-bit session key.

The ciphertext bundle contains the encrypted session key and the encrypted payload. The recipient uses their private key to decrypt the session key, then uses the session key to decrypt the payload.

This is exactly how your web browser works when you visit an https website. The asymmetric algorithm (RSA or ECDH) establishes a shared secret; AES encrypts the actual web traffic.

PrologAI uses AES-256-GCM.

AES stands for Advanced Encryption Standard. 256 means a 256-bit key. GCM stands for Galois/Counter Mode, which is an operating mode that provides both encryption and authentication.

The authentication part is important: AES-256-GCM also computes an authentication tag (a short checksum) that proves the ciphertext has not been tampered with. If someone modifies the encrypted data or the tag, decryption will fail and raise an error. This prevents attackers from making undetected changes to the ciphertext.

EXAMPLE 1: ENCRYPTING A PASSWORD WITH RSA

This example shows the full round-trip: generate a key pair, encrypt a password, decrypt it, and verify the result.

Step 1: Generate a key pair.

The key pair is written to temporary PEM files (PEM is a standard format for storing cryptographic keys as text).

    ?- prologai_keygen_rsa(2048, PrivFile, PubFile).
    PrivFile = '/tmp/swipl_prologai_rsa_priv_1234',
    PubFile = '/tmp/swipl_prologai_rsa_pub_5678'.

The variable PrivFile now holds the path to your private key file.
The variable PubFile holds the path to your public key file.

Keep the private key secret. Distribute the public key freely.

Step 2: Load the keys into Prolog.

    ?- prologai_pem_load_private(PrivFile, PrivKey),
       prologai_pem_load_public(PubFile, PubKey).
    PrivKey = <ssl_key>(0x...),
    PubKey = <ssl_key>(0x...).

PrivKey and PubKey are now Prolog terms representing the loaded keys.

Step 3: Encrypt the password.

    ?- prologai_encrypt_rsa(PubKey, 'hunter2_my_bank_password', Bundle, Tag).
    Bundle = rsa_bundle('7f3a...', '2b9c...', 'a1d4...'),
    Tag = '8e2f...'.

Bundle is a compound term containing three hex-encoded fields:
    the encrypted session key, the IV (initialization vector), and the ciphertext.

Tag is the AES-GCM authentication tag. Store both Bundle and Tag.

Note: the actual hex strings will be much longer than shown above (they are abbreviated here for readability).

Step 4: Decrypt the password.

    ?- prologai_decrypt_rsa(PrivKey, Bundle, Tag, Plain).
    Plain = 'hunter2_my_bank_password'.

The original password is recovered.

Step 5: Clean up the key files.

    ?- delete_file(PrivFile), delete_file(PubFile).


EXAMPLE 2: ENCRYPTING A SOCIAL SECURITY NUMBER WITH ECDH

ECDH does not use PEM files for the recipient key. The recipient's key is stored as an integer (the private scalar) and a point on the elliptic curve (the public key).

Step 1: Generate a key pair for the recipient.

    ?- prologai_keygen_ecdh(prime256v1, RecipPriv, RecipPub).
    RecipPriv = 73849201...,
    RecipPub = point(114492..., 98234...).

RecipPriv is a large integer (the private scalar). Keep it secret.
RecipPub is a point on the P-256 curve, represented as point(X, Y) where X and Y are large integers.

Step 2: Encrypt the SSN using only the public key.

    ?- prologai_encrypt_ecdh(RecipPub, 'SSN:123-45-6789', Bundle, Tag).
    Bundle = ecdh_bundle(114492..., 98234..., '3f7a...', 'b2c1...'),
    Tag = 'a9e4...'.

The ECDH bundle contains:
    EphPubX, EphPubY: the ephemeral public key (so the recipient can compute the shared secret).
    IVHex: the initialization vector, hex-encoded.
    CipherHex: the ciphertext, hex-encoded.

Step 3: Decrypt using only the private scalar.

    ?- prologai_decrypt_ecdh(RecipPriv, Bundle, Tag, Plain, prime256v1).
    Plain = 'SSN:123-45-6789'.

The SSN is recovered.

Key insight: the sender never needed RecipPriv. They only needed RecipPub. The private scalar stays with the recipient forever. This is what makes asymmetric encryption so powerful.


EXAMPLE 3: THE GENERIC INTERFACE

The generic interface lets you write code that works with any algorithm.

    ?- prologai_keygen_ecdh(prime256v1, RecipPriv, RecipPub),
       prologai_encrypt(lattice_cryptography_algo_ecdh, RecipPub, 'CreditCard:4111111111111111', Bundle, Tag),
       prologai_decrypt(lattice_cryptography_algo_ecdh, RecipPriv, Bundle, Tag, Plain).
    Plain = 'CreditCard:4111111111111111'.

To switch to RSA, change lattice_cryptography_algo_ecdh to lattice_cryptography_algo_rsa and supply RSA keys instead.

To switch to PQC (when available), change to lattice_cryptography_algo_pqc and supply a PEM file path.


WHAT HAPPENS WHEN THE TAG IS TAMPERED?

AES-256-GCM protects against tampering. If the ciphertext or the tag is changed even slightly, decryption raises an error.

To demonstrate:

    ?- prologai_keygen_ecdh(prime256v1, RecipPriv, RecipPub),
       prologai_encrypt_ecdh(RecipPub, 'secret value', Bundle, _RealTag),
       FakeTag = '00000000000000000000000000000000',
       prologai_decrypt_ecdh(RecipPriv, Bundle, FakeTag, Plain, prime256v1).
    ERROR: SSL(00000000) : ...

The decryption fails and raises an error. It does not silently return wrong data.

This means: if an attacker modifies an encrypted bundle in the Lattice journal, the system will detect the modification when decryption is attempted. The mind will know the data has been tampered with.


STORING AN ENCRYPTED VALUE IN THE LATTICE

Encrypted bundles are ordinary Prolog compound terms. They can be stored in the Lattice exactly like any other value.

    ?- prologai_keygen_ecdh(prime256v1, RecipPriv, RecipPub),
       prologai_encrypt_ecdh(RecipPub, 'my_bank_password', Bundle, Tag),
       anchor_node(private_credential, bank_password, Bundle-Tag, []).

The Lattice now contains a node_fact with relation type private_credential, key bank_password, and value Bundle-Tag.

The value looks like:
    ecdh_bundle(48392..., 91234..., '3a7f...', 'b2d1...')-'c9e4...'

Anyone reading the Lattice journal sees only this encrypted blob. The plaintext password is not there.

To retrieve and decrypt:

    ?- live_node_facts(private_credential-bank_password, [node_fact(_,_,BundleTag,_)]),
       BundleTag = Bundle-Tag,
       prologai_decrypt_ecdh(RecipPriv, Bundle, Tag, Plain, prime256v1).
    Plain = 'my_bank_password'.


CHOOSING AN ALGORITHM

Here is a practical guide to choosing the right algorithm for your situation.

Use RSA if:
    You are interfacing with external systems that only support RSA.
    You prefer a well-established, widely-tested algorithm.
    Forward secrecy is not required.
    Key management via PEM files is convenient.

Use ECDH if:
    You want smaller, faster operations.
    You want forward secrecy (each encryption uses a fresh ephemeral key).
    You are building a new system with no legacy constraints.
    This is the recommended default for new PrologAI deployments.

Use PQC (ML-KEM-768) if:
    You are protecting data that must remain secret for more than 10 years.
    Your organization has a regulatory or policy requirement to use NIST PQC standards.
    You have installed the OQS Provider and verified it works in your environment.
    You are building a future-proof system designed to survive the eventual arrival of large-scale quantum computers.


REAL-WORLD APPLICATIONS OF THESE TECHNIQUES

The mathematics behind PrologAI's crypto pack powers the entire digital economy.

When you log in to your bank's website, RSA or ECDH establishes a shared secret and AES encrypts your session. That is exactly what prologai_encrypt/5 does.

When your phone stores a fingerprint or face ID template, it encrypts it with a key that never leaves the secure hardware chip. That is the same asymmetric-key-wraps-symmetric-key pattern.

When governments secure classified communications, they use NIST-standardized algorithms. ML-KEM-768 is one of the newest NIST standards, adopted in 2024.

When researchers talk about quantum-safe cryptography, they mean algorithms like ML-KEM, FALCON, SPHINCS+, and ML-DSA that resist Shor's Algorithm on a quantum computer. PrologAI's architecture allows any of these to be added as a new algorithm family.

The hybrid encryption pattern (asymmetric wraps a symmetric session key) is so universally adopted because it combines the best properties of both: the key-distribution convenience of asymmetric encryption and the speed of symmetric encryption.

By learning this chapter, you have learned the same fundamental cryptographic technique used in TLS (the protocol behind HTTPS), Signal, WhatsApp, iMessage, GPG email encryption, and every modern secure communication system.

Chapter 14: Meta-Predicates and Higher-Order Programming

Author: D. R. Dison, Founder of AIU (Artificial Intelligence University). Creator and Owner of PrologAI and Mentova. ORCID: 0009-0001-9246-5758. linkedin.com/in/d-r-dison/


WHAT IS HIGHER-ORDER PROGRAMMING?

In standard Prolog, predicates work on data: atoms, numbers, lists, compound terms.

Higher-order programming means treating predicates themselves as data: passing them as arguments, calling them dynamically, and building predicates that work on other predicates.

This is extremely powerful in PrologAI because many operations on the Lattice, on actor networks, and on reasoning chains require you to apply a computation across a collection without knowing in advance what that computation is.


THE CALL PREDICATE

The most fundamental higher-order predicate is call/1 through call/N.

call(Goal) is equivalent to executing Goal directly. It succeeds if and only if Goal succeeds.

    ?- call(write('hello')).
    hello
    true.

call/N adds extra arguments to the goal. call(Goal, Arg1) is equivalent to adding Arg1 as the last argument of Goal.

    ?- Goal = write, call(Goal, 'hello').
    hello
    true.

    ?- Goal = format("Value: ~w~n"), call(Goal, 42).
    Value: 42
    true.

This pattern is the basis for all higher-order list operations.


MAPLIST

maplist/2 through maplist/5 apply a goal to every element of one or more lists.

maplist(+Goal, +List) - Goal must succeed for every element of List.

    ?- maplist(number, [1, 2, 3]).
    true.

    ?- maplist(number, [1, two, 3]).
    false.

maplist(+Goal, +List, -ResultList) - applies Goal(Elem, Result) to produce a new list.

    ?- maplist([X, Y]>>(Y is X * 2), [1, 2, 3], Doubled).
    Doubled = [2, 4, 6].

The >>/2 operator (from library(yall), YALL stands for Yet Another Lambda Library) defines an inline anonymous predicate. [X, Y]>>(Y is X * 2) means: "given X and Y, succeed if Y equals X times 2."

In PrologAI pack code, maplist is used extensively for processing lists of node_fact identifiers, actor names, and reasoning results.

Example from a PrologAI pack:

    % Apply a scoring function to every candidate solution.
    maplist(score_candidate(Context), Candidates, Scores).

    % score_candidate/3: given Context and Candidate, compute Score.
    score_candidate(Context, Candidate, Score) :-
        % Evaluate the candidate against the context.
        evaluate(Context, Candidate, Score).


INCLUDE AND EXCLUDE

include(+Goal, +List, -Filtered) - keeps only elements for which Goal succeeds.

    ?- include(number, [1, two, 3, four, 5], Numbers).
    Numbers = [1, 3, 5].

exclude(+Goal, +List, -Filtered) - removes elements for which Goal succeeds.

    ?- exclude(number, [1, two, 3, four, 5], NonNumbers).
    NonNumbers = [two, four].

In Lattice queries, include and exclude filter collections of node_facts based on predicates like scope membership or tag presence.


FOLDL

foldl(+Goal, +List, +V0, -V) - reduces a list to a single value by repeatedly applying Goal.

Goal is called as Goal(Element, AccIn, AccOut). The accumulator starts at V0 and is threaded through each call.

Summing a list:

    ?- foldl([E, Acc, NAcc]>>(NAcc is Acc + E), [1, 2, 3, 4, 5], 0, Sum).
    Sum = 15.

Building a maximum:

    ?- foldl([E, Acc, NAcc]>>(NAcc is max(E, Acc)), [3, 1, 4, 1, 5, 9, 2], 0, Max).
    Max = 9.

In PrologAI's crypto pack, foldl converts a list of bytes to a big-endian integer:

    foldl([B, Acc, NAcc]>>(NAcc is Acc * 256 + B), Bytes, 0, PrivInt).


AGGREGATE

The aggregate_all/3 predicate (already introduced in Chapter 5) is a higher-order tool for computing aggregate values over all solutions to a goal.

aggregate_all(+Template, +Goal, -Result)

Templates:
    count         - number of solutions
    sum(Expr)     - sum of Expr across solutions
    max(Expr)     - maximum of Expr
    min(Expr)     - minimum of Expr
    bag(Expr)     - list of all Expr values (like findall)
    set(Expr)     - sorted, deduplicated list

Example: count node_facts in the Lattice by relation type.

    ?- aggregate_all(count, traverse_nexus(apex, node_fact(_, percept_signal, _, _), 1000, _), N).

Example: collect all distinct tags across all node_facts.

    ?- aggregate_all(set(Tag),
           (traverse_nexus(apex, node_fact(_, _, _, Tags), 1000, _), member(Tag, Tags)),
           AllTags).


FUNCTOR AND ARG - INSPECTING TERMS

functor/3 decomposes a term into its name (functor) and arity (number of arguments).

    ?- functor(foo(a, b, c), Name, Arity).
    Name = foo, Arity = 3.

    ?- functor(42, Name, Arity).
    Name = 42, Arity = 0.

    ?- functor(T, point, 2).
    T = point(_A, _B).

arg/3 extracts a specific argument from a term.

    ?- arg(1, point(10, 20), X).
    X = 10.

    ?- arg(2, point(10, 20), Y).
    Y = 20.

The univ operator =.. (pronounced "univ") converts between a term and a list.

    ?- foo(a, b, c) =.. L.
    L = [foo, a, b, c].

    ?- T =.. [point, 10, 20].
    T = point(10, 20).

In PrologAI, =.. is used in the reasoning chains to inspect and reconstruct query terms dynamically without knowing their structure in advance.


DEFINITE CLAUSE GRAMMARS

A Definite Clause Grammar, or DCG, is a notation for writing parsers and string-processing rules in Prolog.

DCG rules use --> instead of :-.

    sentence --> noun_phrase, verb_phrase.
    noun_phrase --> [the], noun.
    verb_phrase --> verb, noun_phrase.
    noun --> [dog].
    noun --> [cat].
    verb --> [chases].

The grammar above can parse sentences like [the, dog, chases, the, cat].

To use a DCG rule, call phrase/2:

    ?- phrase(sentence, [the, dog, chases, the, cat]).
    true.

    ?- phrase(noun_phrase, [the, dog]).
    true.

DCGs in PrologAI are used in two places:

1. The language understanding pack (PR 45) uses DCGs to parse natural language input into query terms that can be passed to mentova_query/3.

2. The ANP (Agent Network Protocol, PR 48) uses DCGs to parse and validate incoming message formats.

When you write:

    ?- phrase(sentence, Input, Rest).

You get a parser that consumes tokens from Input and returns any unconsumed tokens in Rest. This makes DCGs ideal for protocol message parsing.


ASSERT AND RETRACT PATTERNS

Dynamic predicates (facts and rules added at runtime with assertz/1, asserta/1, retract/1) are used extensively in PrologAI for:

• Actor state management: each actor's current state is stored as an asserted fact.
• Workspace content: the current cycle's broadcast content is temporarily asserted.
• Session configuration: library paths, registered overseers, and protocol endpoints.

Safe patterns for dynamic knowledge:

Pattern 1: Key-value store.

    :- dynamic actor_state/2.    % actor_name, state_value

    set_state(Actor, State) :-
        retractall(actor_state(Actor, _)),
        assertz(actor_state(Actor, State)).

    get_state(Actor, State) :-
        actor_state(Actor, State).

Pattern 2: Append-only log.

    :- dynamic event_log/2.      % timestamp, event_term

    log_event(Event) :-
        get_time(T),
        assertz(event_log(T, Event)).

    all_events(Events) :-
        findall(T-E, event_log(T, E), Events).

Pattern 3: Counted registry.

    :- dynamic registered_pack/2.   % pack_name, version

    register_pack(Name, Ver) :-
        (registered_pack(Name, _) -> true ; assertz(registered_pack(Name, Ver))).

Pattern 4: Initialization guard.

    :- dynamic initialized/1.

    ensure_initialized(Name, InitGoal) :-
        (initialized(Name) -> true ;
         (call(InitGoal), assertz(initialized(Name)))).


LAMBDA EXPRESSIONS WITH YALL

The YALL (Yet Another Lambda Library) library is available in SWI-Prolog and in PrologAI. It provides lambda expressions using the \ and >> operators.

The basic form is: [Parameters]>>Body

    ?- maplist([X]>>(Y is X*X, write(Y), nl), [1,2,3,4,5]).
    1
    4
    9
    16
    25

Multiple parameters:

    ?- foldl([E, A, B]>>(B is A + E), [1,2,3,4,5], 0, Sum).
    Sum = 15.

In PrologAI pack code, YALL lambdas appear wherever an inline anonymous predicate is cleaner than a named helper predicate. The ERC rule still applies: write a comment above each lambda-containing line.


COPY_TERM AND TEMPLATE INSTANTIATION

copy_term(+Term, -Copy) creates a fresh copy of a term with all variables replaced by new, unbound variables.

    ?- copy_term(foo(X, Y, X), Copy).
    Copy = foo(_A, _B, _A).

The copy has a new pair of variables _A and _B, but the sharing structure is preserved: the first and third arguments are the same variable.

This is used in PrologAI to create fresh query templates for each call to mentova_query/3, so that one query's variable bindings cannot leak into the next.


ASSERT-BASED MEMOIZATION

A manual memoization pattern uses dynamic facts to cache results.

    :- dynamic memo_result/2.

    expensive_compute(Input, Result) :-
        (memo_result(Input, Cached) ->
            Result = Cached
        ;
            really_compute(Input, Result),
            assertz(memo_result(Input, Result))
        ).

This pattern appears in PrologAI's similarity library and in the SONA episodic memory pack: trajectory comparisons that are computed once are cached to avoid redundant computation.

Chapter 15: Constraint Logic Programming

WHAT IS CONSTRAINT LOGIC PROGRAMMING?

Constraint Logic Programming, or CLP, extends Prolog with the ability to express and solve constraints.

In standard Prolog, you express what is true. In CLP, you express relationships and ranges that must hold, and the system figures out what values satisfy all of them.

Think of it like a puzzle: "I need three numbers that add up to 100, all between 1 and 50, with no two the same." In standard Prolog, you would have to write a search loop. In CLP, you simply state the constraints and the system finds the answer.

SWI-Prolog (and therefore PrologAI) includes two constraint domains:

CLP(FD): Constraint Logic Programming over Finite Domains - for integer arithmetic.
CLP(R): Constraint Logic Programming over Real numbers - for floating-point arithmetic.

PrologAI's reasoning modules use CLP(FD) for planning problems, scheduling, and the ARC-AGI benchmark's spatial puzzles.


LOADING CLPFD

    ?- use_module(library(clpfd)).

Once loaded, you have access to the CLP(FD) operators.


CLPFD BASICS

In CLP(FD), you declare that a variable is an integer in a range using the ins/2 and in/2 operators.

    ?- X ins 1..10.
    X in 1..10.

This does not assign X a value. It constrains X to be somewhere between 1 and 10.

Now add another constraint:

    ?- X ins 1..10, X #> 5.
    X in 6..10.

CLP(FD) narrowed the domain to 6..10 because X must be greater than 5.

The arithmetic operators in CLP(FD) are prefixed with #:

    #=   - equal
    #\=  - not equal
    #<   - less than
    #>   - greater than
    #=<  - less than or equal
    #>=  - greater than or equal

    ?- X in 1..100, X #= 3 * Y, Y in 1..30.
    X in 3..90,
    Y in 1..30,
    3*Y#=X.

CLP(FD) has propagated the constraint: if Y is 1..30 and X = 3*Y, then X must be 3..90.


LABELING: FINDING ACTUAL VALUES

After stating constraints, use label/1 to ask CLP(FD) to search for actual integer values that satisfy them.

    ?- X in 1..5, Y in 1..5, X #\= Y, X + Y #= 7, label([X, Y]).
    X = 2, Y = 5 ;
    X = 3, Y = 4 ;
    X = 4, Y = 3 ;
    X = 5, Y = 2.

label/1 takes a list of constrained variables and searches for assignments.

labeling/2 gives you control over the search strategy:

    ?- labeling([min], [X]).    % Search smallest values first.
    ?- labeling([max], [X]).    % Search largest values first.
    ?- labeling([ff], [X, Y]).  % "First fail" - try most constrained variable first.

The first-fail (ff) strategy is important for large constraint problems: it reduces the search space dramatically by tackling the hardest variables first.


A SCHEDULING EXAMPLE

Suppose three tasks must be scheduled on a single processor. Each task has a duration and must not overlap.

Task A: duration 3.
Task B: duration 2.
Task C: duration 5.
All must fit within a 10-unit time window.

    ?- use_module(library(clpfd)).
    ?- StartA in 0..10, StartB in 0..10, StartC in 0..10,
       StartA + 3 #=< StartB #\/ StartB + 2 #=< StartA,
       StartA + 3 #=< StartC #\/ StartC + 5 #=< StartA,
       StartB + 2 #=< StartC #\/ StartC + 5 #=< StartB,
       StartA + 3 #=< 10, StartB + 2 #=< 10, StartC + 5 #=< 10,
       label([StartA, StartB, StartC]).

The #\/ operator is logical OR in CLP(FD). The constraints say: "Either A finishes before B starts, or B finishes before A starts."

CLP(FD) automatically propagates constraints and finds all valid schedules.


THE ALLDIFFERENT CONSTRAINT

all_different/1 is a global constraint that says all variables in a list must have distinct values. It is far more efficient than writing all pairwise inequality constraints.

    ?- Vars = [A, B, C, D], Vars ins 1..4, all_different(Vars), label(Vars).

This gives all permutations of [1, 2, 3, 4] across [A, B, C, D].

all_different/1 is used in PrologAI's ARC-AGI spatial solver to ensure that each grid position is assigned at most one color or object.


SUM AND SCALAR PRODUCT

sum(+Vars, +Op, +Value) constrains the sum of a list of variables.

    ?- Vars = [A, B, C], Vars ins 0..10, sum(Vars, #=, 15), label(Vars).

scalar_product(+Coefficients, +Vars, +Op, +Value):

    ?- scalar_product([1, 2, 3], [A, B, C], #=, 14), [A,B,C] ins 0..10, label([A,B,C]).

These are used in PrologAI's motivational drive system (PR 27) to model resource allocation across competing drives.


CLPR - REAL NUMBER CONSTRAINTS

CLP(R) works with real-number (floating-point) arithmetic. Load it with:

    ?- use_module(library(clpr)).

CLP(R) uses {}/1 notation for constraints:

    ?- {X + Y =:= 1.0, X - Y =:= 0.5}.
    X = 0.75, Y = 0.25.

CLP(R) is used in PrologAI's probabilistic reasoning (PR 16) and Bayesian updating (PR 20) to enforce probability constraints: all probabilities must sum to 1.0, all values must be between 0.0 and 1.0.


WHEN TO USE CLP

Use CLP(FD) when:
• You have integer problems with explicit domains (ranges, discrete values).
• You are scheduling, packing, coloring, or assigning with exclusion constraints.
• You are solving puzzles or spatial reasoning problems.

Use CLP(R) when:
• You have continuous numeric constraints (probability distributions, linear programs).
• You need numeric solving that standard Prolog arithmetic cannot express declaratively.

Avoid CLP when:
• You already know the exact values and just need arithmetic (use is/2).
• The domain is unbounded and there are no constraints to propagate (plain search is faster).

Chapter 16: Tabling and Memoization

Author: D. R. Dison, Founder of AIU (Artificial Intelligence University). Creator and Owner of PrologAI and Mentova. ORCID: 0009-0001-9246-5758. linkedin.com/in/d-r-dison/


WHAT IS TABLING?

Standard Prolog uses depth-first search with backtracking. This is powerful but has two problems that bite developers who work on PrologAI:

Problem 1: Infinite loops from mutual recursion.

If predicate A calls predicate B, and B calls A, and neither reaches a base case before the other, SWI-Prolog will loop forever (or exhaust the stack).

In reasoning chains (PR 34, the chainer pack), mutual recursion is common: the forward chainer and the backward chainer can call each other.

Problem 2: Redundant computation.

In Piagetian assessment (PR 12), each milestone check can trigger multiple sub-checks. Without memoization, the same sub-check runs hundreds of times.

Tabling solves both problems.

Tabling (also called SLG resolution, or XSB-style tabling) memoizes every call to a tabled predicate: the first time a query is made, the answer is computed and stored. Subsequent calls with the same arguments return the stored answer immediately, without recomputing.

More importantly, tabling handles cyclic and mutually recursive predicates correctly: if A calls B and B calls A, tabling detects the cycle and resolves it instead of looping.


DECLARING A TABLED PREDICATE

Add a :- table directive before the predicate:

    :- table ancestor/2.

    ancestor(X, Z) :- parent(X, Z).
    ancestor(X, Z) :- parent(X, Y), ancestor(Y, Z).

Without tabling, if the parent/2 facts formed a cycle (rare but possible in some knowledge graphs), ancestor/2 would loop.

With tabling, each (X, Z) pair is computed at most once.


FIBONACCI WITH TABLING

Without tabling:

    fib(0, 0) :- !.
    fib(1, 1) :- !.
    fib(N, F) :-
        N > 1,
        N1 is N - 1, N2 is N - 2,
        fib(N1, F1), fib(N2, F2),
        F is F1 + F2.

fib(40) requires computing fib(20) more than a million times. It is exponential.

With tabling:

    :- table fib/2.

    fib(0, 0) :- !.
    fib(1, 1) :- !.
    fib(N, F) :-
        N > 1,
        N1 is N - 1, N2 is N - 2,
        fib(N1, F1), fib(N2, F2),
        F is F1 + F2.

fib(40) now runs in linear time. Each (N, F) pair is computed exactly once.


TABLING IN PROLOGAI

Three PrologAI packs use tabling:

1. The induction pack (PR 33) tables the learned-rule lookup so the same rule is not re-derived during a single reasoning cycle.

2. The chainer pack (PR 34) tables intermediate sub-goals in long reasoning chains to prevent redundant traversal of the Lattice.

3. The assessment pack (PR 12) tables the result of milestone checks within a single assess_all/2 call, since the same Lattice state is queried from multiple assessment paths.

When writing a new pack that involves recursive queries over the Lattice or over other predicates that may share sub-goals, consider adding :- table to predicates with high reuse.


WELL-FOUNDED SEMANTICS

SWI-Prolog's tabling implements well-founded semantics, which handles three-valued logic correctly:

• A query is true if it provably succeeds.
• A query is false if it provably fails.
• A query is undefined if it is in a cycle with neither provable success nor provable failure.

This third case (undefined) is returned as a suspended answer. PrologAI's defeasible reasoning pack (PR 15) uses this to represent genuinely uncertain conclusions rather than forcing a binary true/false verdict.


TABLING AND THE LATTICE

The Lattice traverse_nexus/4 predicate is not itself tabled, but it is designed to be called inside tabled predicates. The pattern is:

    :- table cached_query/2.

    cached_query(Pattern, Results) :-
        default_nexus(N),
        traverse_nexus(N, Pattern, 1000, Results).

With this, repeated queries for the same Pattern within a reasoning cycle read from the table after the first call, not from the Lattice storage each time.

This is the recommended pattern for performance-critical Lattice queries inside recursive reasoning predicates.


ABOLISH_TABLE AND INVALIDATION

Because tabling caches results, stale data can be a problem if the underlying facts change.

To clear all tabled results for a predicate:

    ?- abolish_table_subgoals(cached_query(_,_)).

Or clear everything:

    ?- abolish_all_tables.

In PrologAI, the workspace cycle (PR 18) calls abolish_all_tables at the start of each cycle to ensure that cached Lattice queries reflect the current Lattice state, not a stale snapshot from the previous cycle.


WHEN TO TABLE

Use tabling when:
• The predicate is recursive and can loop without tabling.
• The predicate is called many times with the same arguments in a single reasoning session.
• The predicate's result depends only on the Prolog knowledge base, not on external side effects.

Do not table when:
• The predicate has side effects (file I/O, assertz, message sending).
• The arguments vary widely with every call (caching would never help).
• The predicate must see the very latest Prolog knowledge base state (tabling caches old results).

Chapter 17: Pack Architecture and Development

WHAT IS A PACK?

A pack is a self-contained directory that implements one or more PrologAI work packages (PRs).

Every pack has:
• A manifest file (pack.pl) that declares the pack's identity, version, and dependencies.
• A prolog/ subdirectory containing the Prolog source modules.
• A tests/ directory (at the top level of the PrologAI repo) for acceptance tests.

When a user calls use_module(library(assessment)), SWI-Prolog looks for a file named assessment.pl in any directory registered on the library search path. If packs/assessment/prolog is on that path, it finds packs/assessment/prolog/assessment.pl.


DIRECTORY STRUCTURE

A pack named my_pack lives at:

    PrologAI/packs/my_pack/
        pack.pl                  - manifest
        prolog/
            my_pack.pl           - main module (same name as pack)
            my_pack_internals.pl - private helpers (optional)
        README.md                - brief description

The test file lives at:

    PrologAI/tests/prNN/
        test_prNN.pl             - plunit acceptance tests


THE PACK MANIFEST (pack.pl)

The manifest is a set of Prolog facts. Each fact ends with a period.

A minimal manifest:

    name(my_pack).
    version('1.0.0').
    title('My Pack - what it does in one sentence').
    author('D. R. Dison, Founder of AIU').
    home('https://github.com/ai-university-aiu/PrologAI').
    download('https://github.com/ai-university-aiu/PrologAI/releases/tag/v1.0.0').

Versioning follows Semantic Versioning (SemVer, see semver.org): MAJOR.MINOR.PATCH.

• MAJOR: a breaking change to the public API.
• MINOR: new functionality, backwards-compatible.
• PATCH: bug fix, backwards-compatible.

When the SPARC documentation rule requires incrementing the version, it is always the MINOR version that increments unless a breaking change was made.


THE MAIN MODULE FILE

The main module file declares the module and exports its public predicates.

    % Declare module and exported predicates.
    :- module(my_pack, [
        % The main public predicate.
        my_pack_do/3,
        % The availability probe.
        my_pack_available/1
    ]).

    % Load SWI-Prolog crypto library.
    :- use_module(library(crypto)).
    % Load SWI-Prolog lists library.
    :- use_module(library(lists)).

Every line of code in a PrologAI .pl file must have an ERC (English Readable Code) comment immediately above it.

The ERC rule applies to:
• Module declarations.
• use_module directives.
• All predicate clauses (head and body lines).
• All initialization directives.
• Any line that SWI-Prolog will execute.

The ERC rule does NOT apply to:
• Blank lines.
• The module-level comment block at the top of a file (the file header is itself a comment).
• The closing period of a multi-line clause body (the period is part of the clause above, not a new line requiring its own comment).

GOOD ERC STYLE:

    % Check whether the given algorithm is available.
    my_pack_available(my_algo_rsa) :-
        % Verify that the SWI-Prolog crypto library is present.
        catch(use_module(library(crypto)), _, fail).

BAD (NOT ERC-COMPLIANT):

    my_pack_available(my_algo_rsa) :-   % missing ERC comment above
        catch(use_module(library(crypto)), _, fail). % inline comment


DESIGNING THE API

A PrologAI pack API should follow these conventions.

Convention 1: All exported predicates use the pack prefix.

If the pack is named lattice_crypto, predicates are named prologai_encrypt/5, prologai_decrypt/5, prologai_keygen_rsa/3. They begin with prologai_ (PrologAI interface) or lc_ (lattice crypto, internal helpers). Never use names that look like SWI-Prolog built-ins.

Convention 2: Deterministic predicates succeed exactly once.

If a predicate is meant to compute a single result, it must not leave a choice point. Use cuts (!) only when semantically correct.

Convention 3: Availability probes use a standard form.

    pack_available(pack_feature).

This allows the calling system to probe capabilities at runtime without catching exceptions.

Convention 4: Error terms are structured.

    throw(my_pack_error(error_type, Message))

This makes errors catchable with specific patterns:

    catch(Goal, my_pack_error(Type, Msg), handle_error(Type, Msg)).

Convention 5: All initialization via :- initialization(..., now).

Avoid top-level goals outside of predicates. Initialization code belongs in an initialization directive so it runs at a controlled time.


THE SPARC DOCUMENTATION RULE

Every code change must be accompanied by SPARC documentation changes.

SPARC stands for Specification, Pseudocode, Architecture, Refinement, Completion.

The five SPARC volumes are:
• PrologAI_1_Specification_vN.txt - what to build.
• PrologAI_2_Pseudocode_vN.txt - how each work package reasons, in pseudocode.
• PrologAI_3_Architecture_vN.txt - where each piece lives in the system.
• PrologAI_4_Refinement_vN.txt - acceptance criteria and testing.
• PrologAI_5_Completion_vN.txt - what has been delivered and why.

When you create a new pack:

1. Add a new Part (Part NN) to the Specification describing what the pack must do, its functional requirements, and its threat model (if relevant).

2. Add a new Work Package (PSEUDO-NN) to the Pseudocode describing the key algorithms in pseudocode. Not code - pseudocode. The Pseudocode volume is for understanding, not compilation.

3. Add a new Part (ARCH-NN) to the Architecture describing where the pack lives in the layer diagram, its dependencies, and its interface to the Lattice.

4. Add an Acceptance Criteria section (AC-PRNN) to the Refinement document. Each acceptance criterion has an identifier (AC-PRNN-001), a description, a test method, and a status (COMPLETE / OPEN).

5. Add a new Part (COMP-NN) to the Completion document describing the delivered artifact, what changed, and why.

Each updated volume gets its minor version incremented. If the Specification was at v14, it becomes v15 after the new Part is added.

The CLAUDE.md file in the PrologAI repository root must be updated with the new volume version numbers.


THE FEATURE BRANCH WORKFLOW

Every new pack goes through the feature branch workflow:

1. Create the feature branch: git checkout -b feature/prNN-pack-name

2. Create the directory structure: packs/pack_name/pack.pl and packs/pack_name/prolog/pack_name.pl

3. Write the code with ERC comments on every line.

4. Write the acceptance tests in tests/prNN/test_prNN.pl.

5. Update all five SPARC volumes and increment each affected volume's minor version.

6. Update PrologAI_Tutorial.txt and docs/tutorial/ (required for every code change).

7. Commit: git add -p (review each change) then git commit -m "feat(pack_name): short description"

8. Push: git push -u origin feature/prNN-pack-name

9. Open the PR: gh pr create --title "feat(pack_name): description" --body "..."

10. Immediately merge: gh pr merge N --merge --delete-branch

Do not commit secrets, PEM key files, or temporary test outputs. Do not include Co-Authored-By or AI tool attribution in commits.


VERSION CONTROL BEST PRACTICES FOR PACKS

Each pack has its own pack.pl with a version. When you add new predicates or fix bugs, update the version:

    version('1.1.0').   % after adding a new exported predicate (minor bump)
    version('1.0.1').   % after fixing a bug without API change (patch bump)
    version('2.0.0').   % after removing or changing an existing exported predicate (major bump)

The pack version in pack.pl is separate from the SPARC document versions. The SPARC document versions track the documentation history. The pack version tracks the pack's own API stability.

When adding a new pack, decide its position in this order and document it in Architecture Part ARCH-NN.

Chapter 18: Testing Methodology - Five Levels of Assurance

Author: D. R. Dison, Founder of AIU (Artificial Intelligence University). Creator and Owner of PrologAI and Mentova. ORCID: 0009-0001-9246-5758. linkedin.com/in/d-r-dison/


THE FIVE LEVELS

PrologAI uses a five-level testing hierarchy. Each level provides a different kind of assurance.

Level 1: Unit Tests - individual predicates in isolation.
Level 2: Integration Tests - predicates working together.
Level 3: System Tests - the full pack loaded into a running Mentova session.
Level 4: Acceptance Criteria - named, traceable tests that match the SPARC Refinement document.
Level 5: Demonstration - a live Mentova accomplishment (Acc_NN) that proves the capability to an external observer.

Every new pack must have all five levels.


LEVEL 1: UNIT TESTS WITH PLUNIT

SWI-Prolog's plunit library provides a standard unit test framework.

Load it with:

    :- use_module(library(plunit)).

A test block:

    :- begin_tests(my_pack_unit).

    test(predicate_succeeds) :-
        my_pack_do(input, result, Result),
        Result = expected_value.

    test(predicate_fails, [fail]) :-
        my_pack_do(bad_input, result, _).

    test(predicate_throws, [throws(my_pack_error(_, _))]) :-
        my_pack_do(triggering_input, result, _).

    :- end_tests(my_pack_unit).

Test options:
    [fail]         - test is expected to fail.
    [throws(T)]    - test is expected to throw T.
    [true(G)]      - G must be true after the test body.
    [setup(S)]     - run S before the test.
    [cleanup(C)]   - run C after the test.

Run all tests in a file:

    $ swipl -t halt -l tests/prNN/test_prNN.pl -g "run_tests, halt"

Or at the console:

    ?- consult('tests/prNN/test_prNN.pl'), run_tests.


LEVEL 2: INTEGRATION TESTS

Integration tests load multiple modules and test their interactions.

A Level 2 test for a pack that stores encrypted values in the Lattice:

    :- begin_tests(lattice_cryptography_integration).

    test(encrypt_store_retrieve) :-
        % Load crypto pack.
        use_module(library(lattice_cryptography)),
        % Load node_facts pack.
        use_module(library(node_facts)),
        % Generate an ECDH key pair.
        prologai_keygen_ecdh(prime256v1, Priv, Pub),
        % Encrypt a value.
        prologai_encrypt_ecdh(Pub, 'secret', Bundle, Tag),
        % Store in the Lattice.
        anchor_node(private_test, test_key, Bundle-Tag, [test]),
        % Retrieve from the Lattice.
        traverse_nexus('APEX_MIND', node_fact(_, private_test, Bundle2-Tag2, _), 1, [_]),
        % Decrypt and verify.
        prologai_decrypt_ecdh(Priv, Bundle2, Tag2, Plain, prime256v1),
        Plain = 'secret'.

    :- end_tests(lattice_cryptography_integration).

Integration tests should cover:
• Every interaction between this pack and every pack it depends on.
• Error paths (what happens when a dependency fails).
• Lattice state after operations (not just return values).


LEVEL 3: SYSTEM TESTS

System tests load the full Mentova boot sequence and test the pack in a real running mind.

    :- begin_tests(lattice_cryptography_system).

    test(crypto_after_boot) :-
        % Boot the Mentova mind.
        mentova_boot,
        % Verify RSA is available.
        prologai_crypto_available(lattice_cryptography_algo_rsa),
        % Verify ECDH is available.
        prologai_crypto_available(lattice_cryptography_algo_ecdh),
        % Full round-trip through the system.
        prologai_keygen_ecdh(prime256v1, Priv, Pub),
        prologai_encrypt_ecdh(Pub, 'system_test', Bundle, Tag),
        prologai_decrypt_ecdh(Priv, Bundle, Tag, Plain, prime256v1),
        atom_string('system_test', PlainStr),
        atom_string(Plain, PlainStr).

    :- end_tests(lattice_cryptography_system).

System tests are slower (full boot takes several seconds) but provide the highest confidence that the pack works in production conditions.


LEVEL 4: ACCEPTANCE CRITERIA

Acceptance criteria are named, documented tests that trace directly to SPARC Refinement entries.

Naming convention: AC-PRNN-MMM where NN is the PR number and MMM is a three-digit counter.

Example file: tests/pr49/test_pr49.pl

    % PrologAI - Acceptance Tests for PR 49: Cryptographic Privacy Layer
    :- use_module(library(plunit)).
    :- use_module('/home/ccaitwo/PrologAI/packs/lattice_cryptography/prolog/lattice_cryptography').
    :- use_module(library(crypto)).

    :- begin_tests(pr49).

    % AC-PR49-001: AES-256-GCM round-trip succeeds.
    test('AC-PR49-001') :-
        crypto_n_random_bytes(32, KeyBytes), atom_codes(Key, KeyBytes),
        crypto_n_random_bytes(12, IVBytes), atom_codes(IV, IVBytes),
        Plaintext = 'hello world',
        atom_string(Plaintext, PlainStr),
        crypto_data_encrypt(PlainStr, 'aes-256-gcm', Key, IV, Cipher, [tag(TagBytes), encoding(utf8)]),
        crypto_data_decrypt(Cipher, 'aes-256-gcm', Key, IV, Decrypted, [tag(TagBytes), encoding(utf8)]),
        atom_string(PlainAtom, Decrypted),
        PlainAtom = Plaintext.

    % AC-PR49-002: RSA hybrid round-trip succeeds.
    test('AC-PR49-002') :-
        prologai_keygen_rsa(2048, PrivFile, PubFile),
        prologai_pem_load_private(PrivFile, PrivKey),
        prologai_pem_load_public(PubFile, PubKey),
        prologai_encrypt_rsa(PubKey, 'bank_password', Bundle, Tag),
        prologai_decrypt_rsa(PrivKey, Bundle, Tag, Plain),
        atom_string('bank_password', PlainStr),
        atom_string(Plain, PlainStr),
        catch(delete_file(PrivFile), _, true),
        catch(delete_file(PubFile), _, true).

    :- end_tests(pr49).

The Refinement document entry for AC-PR49-001 must exactly match the test name. When the test passes, mark the acceptance criterion as COMPLETE in the Refinement document.

Running acceptance tests and reporting:

    $ swipl -t halt -l tests/pr49/test_pr49.pl -g "run_tests(pr49), halt"
    % PL-Unit: pr49 .......... passed 0.442 sec

The test output "passed N sec" is the evidence recorded in the Completion document.


LEVEL 5: DEMONSTRATION

A demonstration (Acc_NN) is a live, observable proof that the capability works at the Mentova mind level.

Demonstrations are the highest level of assurance. They are described in the Demonstration Plan (SPARC Volume 6) and published as papers and announcements.

A demonstration has:
• A name: Acc_NN - Short Description.
• A goal: what capability is being demonstrated.
• A method: the exact sequence of mentova_query/3 calls or Prolog goals that proves the capability.
• Evidence: the output of running the method, saved as a log.
• A paper: a short scientific paper (in papers/ directory) describing the demonstration.
• An announcement: a public announcement (in announcements/ directory) for external audiences.

The demonstration for PR 49 (cryptographic privacy) might be:

Acc_73 - Cryptographic Privacy: Sensitive Data Protected by Hybrid Encryption

Goal: Demonstrate that a sensitive value stored in the Lattice is inaccessible in plaintext, and that the authorized party can recover it using ECDH hybrid decryption.

Method:
1. Boot Mentova: mentova_boot.
2. Generate an ECDH key pair.
3. Encrypt 'SSN:123-45-6789' and store the bundle in the Lattice.
4. Show the stored value (the bundle) - confirm it contains no plaintext.
5. Decrypt the bundle and confirm recovery.
6. Demonstrate tamper detection: corrupt the tag, attempt decryption, observe the error.
7. Run assess_all(mentova, R) and confirm the assessment is unaffected.

Evidence: log output showing each step, with the encrypted bundle and the successful decryption, and the tamper-detection error.


RUNNING THE FULL TEST SUITE

To run all tests for the platform:

    $ cd /home/ccaitwo/PrologAI
    $ find tests/ -name 'test_*.pl' -exec swipl -t halt -l {} -g "run_tests, halt" \;

Or with a test runner script:

    $ ./scripts/run_all_tests.sh

A well-maintained PrologAI installation should show all Level 4 tests passing on every commit to the main branch. This is enforced by the CI/CD (Continuous Integration / Continuous Deployment) pipeline.


WHAT TO TEST AND WHAT NOT TO TEST

Test:
• Every exported predicate at every boundary condition (empty list, zero, negative, maximum size).
• Every error path (wrong key type, tampered data, missing file, provider not found).
• Every interaction with the Lattice (what gets stored, what is retrieved, what fails gracefully).
• Determinism: predicates that should succeed exactly once should not leave choice points.
• Thread safety: predicates used in actors must work correctly under concurrency.

Do not test:
• SWI-Prolog built-in behavior (the built-ins have their own test suite).
• The SPARC documentation itself (documents are not executable).
• Behavior that is outside the pack's scope (other packs are tested by their own test files).

Chapter 19: Lattice Schema Design Patterns

Author: D. R. Dison, Founder of AIU (Artificial Intelligence University). Creator and Owner of PrologAI and Mentova. ORCID: 0009-0001-9246-5758. linkedin.com/in/d-r-dison/


THE LATTICE AS A SCHEMA

The Lattice is a flexible store. anchor_node/4 will accept any relation type, any payload, and any tags. This flexibility is a strength, but it is also a risk.

Without a schema - a principled design for what relation types exist, what their payloads contain, and what tags are used - the Lattice becomes an unstructured dump. Queries become unreliable. Assessment evidence becomes inconsistent. New packs cannot find the facts they need.

A Certified PrologAI Engineer thinks of the Lattice as a typed database and designs its schema before writing code.


RELATION TYPE TAXONOMY

The relation type (the first argument of anchor_node/4) is the primary key for Lattice queries. Choose relation types carefully.

Level 1 - Core sensor types (used by PR 8, Perception):
    percept_signal      - raw sensory input
    object_tracking     - an object being tracked across time
    event_sequence      - a sequence of observed events

Level 2 - Cognitive types (used by PRs 18-42):
    objective           - a goal the mind is pursuing
    prediction          - a prediction about a future state
    belief              - a belief the mind holds
    symbolic_representation - a symbolic mapping
    conservation_demonstrated - conservation insight
    theory_of_mind      - a model of another agent's beliefs
    formal_proof        - a formal logical derivation

Level 3 - Self-model types (used by PRs 27, 38, 42):
    self_concept        - the mind's model of itself
    capability_record   - a record of demonstrated capability
    attentional_state   - what the mind is currently attending to

Level 4 - Social types (used by PRs 40, 43-48):
    agent_action        - an action taken by an agent
    communication_record - a message sent or received
    protocol_event      - a protocol-level event (task submitted, run created, DID verified)

Level 5 - Privacy types (used by PR 49):
    private_credential  - an encrypted sensitive value
    encryption_event    - a record of an encryption or decryption operation

Level 6 - Constitutional types (written by kernel PR 1):
    constitutional_principle - a principle from the constitution
    constitutional_denial   - a record of a denied action
    oversight_event         - an event observed by the oversight monitor

When adding a new pack, define new relation types at the appropriate level, or reuse existing ones. Do not proliferate relation types unnecessarily: every new relation type must be documented in the Specification.


PAYLOAD CONVENTIONS

The payload (the second argument of anchor_node/4) can be any Prolog term. Convention:

• For structured data, use a list of key(value) pairs:
    [source(visual), stimulus(object_appeared), intensity(0.8)]

• For simple scalar values, use the value directly:
    42
    'my_bank_password_encrypted'

• For compound values with multiple fields, use a compound term:
    cipher_bundle(key_hex('7f3a...'), iv_hex('2b9c...'), cipher_hex('a1d4...'))

• For private encrypted payloads, the bundle and tag are stored together:
    Bundle-Tag

Never store passwords, keys, or other secrets in plain text as payloads. Use the lattice_crypto pack (PR 49) to encrypt before storing.


TAG STRATEGIES

Tags (the third argument of anchor_node/4) are atoms in a list. They are used to filter and categorize node_facts.

Good tag design:

Rule 1: Tags are single atoms, all lowercase, no spaces.
    [nature, observation, visual, outdoor]

Rule 2: Use domain tags for cross-cutting categories.
    [nature]     - relates to the natural world
    [social]     - relates to another agent
    [private]    - relates to sensitive information
    [test]       - created during a test run (not real knowledge)

Rule 3: Use event tags to indicate when something happened.
    [at_boot]    - created during mentova_boot
    [from_input] - derived from user input
    [from_inference] - derived by a reasoning process

Rule 4: Use provenance tags to indicate where knowledge came from.
    [from_mcp]   - arrived through the MCP gateway
    [from_a2a]   - arrived through the A2A protocol
    [from_sensor] - arrived through a physical sensor

Querying by tag: traverse_nexus does not directly support tag filtering. Build a filter using include/3 after traversal:

    ?- traverse_nexus('APEX_MIND', node_fact(_, percept_signal, _, Tags), 100, Results),
       include([nf]>>(nf = node_fact(_, _, _, T), member(nature, T)), Results, Filtered).

For large Lattice deployments, maintain a separate index (a dynamic predicate) mapping tags to node_fact identifiers for fast lookup.


SCOPE ASSIGNMENT

The nine scopes are described in Chapter 7. Here is the assignment decision matrix.

    Scope               When to use
    CONSTITUTION_SCOPE  Never (kernel only writes this; it is read-only after boot)
    OVERSEER_SCOPE      Constitutional denials, oversight events, audit records
    PAST_ZONE           Sealed historical facts (moved from PRESENT_ZONE after a session)
    PRESENT_ZONE        Active working memory - most new node_facts go here (default)
    FUTURE_ZONE         Plans, predictions, scheduled actions
    SELF_SCOPE          Self-concept, capability records, attentional state
    SOCIAL_SCOPE        Theory of mind facts, communication records, agent models
    WORLD_SCOPE         Knowledge about external entities, environmental facts
    QUARANTINE_SCOPE    Any content from untrusted sources, before vetting

When in doubt, use PRESENT_ZONE. The scope is specified in the relation type or via an explicit scope tag. Scope enforcement is handled by the kernel; packs do not need to implement scope logic themselves.


CROSS-NEXUS LINKING

A nexus (from Chapter 7) is a named workspace within the Lattice. Mentova uses APEX_MIND as the default nexus.

Multi-mind architectures (using PR 40, spinoff) create sub-minds with their own nexuses. The sub-mind's nexus is named after the sub-mind: APEX_MIND_SPINOFF_1, APEX_MIND_SPINOFF_2, etc.

To retrieve facts from a non-default nexus:

    ?- traverse_nexus('APEX_MIND_SPINOFF_1', Pattern, Limit, Results).

Cross-nexus linking means creating a node_fact in one nexus that references the identifier of a node_fact in another nexus:

    anchor_node(cross_reference, [source_nexus('APEX_MIND'), target_id(Id123)], [], _).

This pattern is used by the A2A pack (PR 43) when a delegated task produces results in a sub-agent's nexus that must be visible to the parent.


AVOIDING SCHEMA DRIFT

Schema drift happens when different packs use the same relation type with inconsistent payloads.

For example: if PR 8 creates percept_signal with payload [source(visual)] and PR 45 creates percept_signal with payload [input_type(text)], the assessment pack (PR 12) may fail to find percept_signal facts because it expects [source(visual)].

Preventing schema drift:

1. Define every relation type in the Specification document before using it in code.

2. When extending an existing relation type with new payload fields, make the new fields optional (check for them with member/2, do not require them).

3. Write a schema validation predicate in your pack that checks incoming node_facts:

    validate_percept_signal(node_fact(_, percept_signal, Payload, _)) :-
        is_list(Payload),
        member(source(_), Payload).

4. Add schema validation tests to your Level 2 integration tests.


MIGRATION PATTERNS

When a new version of a pack changes the payload format for an existing relation type, existing node_facts in the Lattice will have the old format.

Migration strategy:

1. Define a migration predicate:

    migrate_percept_signal_v1_to_v2 :-
        traverse_nexus('APEX_MIND', node_fact(Id, percept_signal, OldPayload, Tags), 1000, Results),
        member(node_fact(Id, percept_signal, OldPayload, Tags), Results),
        \+ member(source(_), OldPayload),
        NewPayload = [source(unknown) | OldPayload],
        retract_node_fact(Id),
        anchor_node(percept_signal, NewPayload, Tags, _).

2. Call the migration predicate in an initialization directive that runs at the first boot after the upgrade:

    :- initialization(migrate_percept_signal_v1_to_v2, now).

3. Document the migration in the Completion document.

Note: the PAST_ZONE is sealed and cannot be migrated. Only PRESENT_ZONE and FUTURE_ZONE facts can be retracted and re-anchored.

THE ACTOR MODEL

An actor is a self-contained unit of computation that communicates exclusively through message passing.

Actors in PrologAI (PR 6-9) are implemented as SWI-Prolog threads. Each actor is a thread running a loop that reads from a message queue, processes each message, and optionally sends messages to other actors.

The actor model is appropriate for PrologAI because:
• Many reasoning processes run independently and can overlap.
• The Lattice is a shared resource that actors write to and read from without direct coordination.
• Sentinels must fire asynchronously when patterns appear in the Lattice.
• The workspace cycle must broadcast without blocking the reasoning that continues in parallel.


CREATING AND STARTING AN ACTOR

    create_actor(+Name, +Module, +Init)

Name: an atom that uniquely identifies this actor.
Module: the Prolog module that implements the actor's behavior.
Init: initial state, passed to the actor's init/1 predicate.

The actor module must define:

    handle_message(+Message, +StateIn, -StateOut)

This predicate is called once for each incoming message. It receives the message and the current state, and returns the new state.

A minimal actor module:

    :- module(counter_actor, [handle_message/3]).

    % Initialize the counter actor state to zero.
    init(0).

    % Increment the counter when asked.
    handle_message(increment, Count, NewCount) :-
        NewCount is Count + 1.

    % Return the current count on request.
    handle_message(get_count(Caller), Count, Count) :-
        send_message(Caller, count_is(Count)).

    % Store the count in the Lattice when asked.
    handle_message(save, Count, Count) :-
        anchor_node(counter_state, [value(Count)], [], _).

Starting the actor:

    ?- create_actor(my_counter, counter_actor, 0).


SENDING MESSAGES

    send_message(+Actor, +Message)

This puts Message into Actor's queue. It returns immediately (non-blocking). The actor processes the message asynchronously.

    ?- send_message(my_counter, increment).
    ?- send_message(my_counter, increment).
    ?- send_message(my_counter, get_count(console)).

The actor processes these in order.


PUBLISH-SUBSCRIBE

The pubsub pack (PR 7) provides broadcast channels.

    subscribe(+Channel, +HandlerPredicate)
    publish(+Channel, +Content)
    unsubscribe(+Channel, +HandlerPredicate)

A subscriber registers a predicate to be called whenever content is published on a channel:

    ?- subscribe(workspace_broadcast, handle_broadcast).

    handle_broadcast(Content) :-
        format("Received broadcast: ~w~n", [Content]).

Any actor or query can publish:

    ?- publish(workspace_broadcast, salience(perceiving_tweety, 0.9)).

All subscribers to workspace_broadcast receive the published content.

The workspace cycle (PR 18) uses pubsub as its primary communication mechanism: it publishes the highest-salience content on workspace_broadcast, and all subscribed actors (assessment, SONA, attention schema) receive it simultaneously.


SENTINELS

Sentinels (PR 9) are actors that watch the Lattice.

A sentinel registers to watch for a specific relation type. Whenever a node_fact of that type is anchored, the sentinel fires.

    register_sentinel(+RelationType, +Handler)

Handler is a predicate of arity 1 that receives the new node_fact.

Example: a sentinel that fires when a percept_signal arrives and checks the Level 1 milestone:

    :- register_sentinel(percept_signal, check_level1_milestone).

    check_level1_milestone(node_fact(_, percept_signal, Payload, _)) :-
        format("Percept arrived: ~w~n", [Payload]),
        assess_piaget(mentova, 1, R),
        format("Level 1: ~w~n", [R]).

This sentinel fires automatically every time anchor_node(percept_signal, ...) is called. The programmer does not need to call check_level1_milestone explicitly.

Sentinels implement the reflex arc: automatic, unconditional response to a stimulus. This is Piagetian Level 1.


ACTOR LIFECYCLE

An actor goes through three states:

1. Created: the actor thread is started. It reads its initial state from init/1.
2. Running: the actor processes messages as they arrive. Between messages, it sleeps.
3. Stopped: the actor receives a stop message and terminates cleanly.

To stop an actor:

    ?- send_message(my_counter, stop).

The handle_message/3 clause for stop should do cleanup and then call actor_stop/0:

    handle_message(stop, State, State) :-
        format("Counter actor stopping with state: ~w~n", [State]),
        actor_stop.

actor_stop/0 terminates the current actor's thread.


ERROR HANDLING IN ACTORS

Actors run in separate threads. An uncaught exception in handle_message/3 will crash the actor thread but not the main thread or other actors.

Best practice: wrap every handle_message clause body in catch/3:

    handle_message(Msg, StateIn, StateOut) :-
        catch(
            handle_message_impl(Msg, StateIn, StateOut),
            Error,
            (   format("Actor error: ~w~n", [Error]),
                log_actor_error(Error),
                StateOut = StateIn
            )
        ).

    handle_message_impl(increment, Count, NewCount) :-
        NewCount is Count + 1.

This pattern ensures the actor stays alive even when individual messages cause errors.


CONCURRENCY SAFETY

Actors communicate through message queues, which are inherently thread-safe in SWI-Prolog. But the Lattice is a shared resource.

Rules for concurrent Lattice access:

1. anchor_node/4 is thread-safe. Multiple actors can call it simultaneously.

2. retract_node_fact/1 is NOT thread-safe without locking. If two actors might retract the same fact, use with_mutex/2:

    with_mutex(lattice_write_mutex, retract_node_fact(Id)).

3. traverse_nexus/4 takes a snapshot of the current Lattice state at the time of the call. It does not lock the Lattice. This is safe for reads but means the results may be slightly stale if another actor is writing simultaneously.

4. Sentinels fire in the thread that called anchor_node/4. Keep sentinel handlers short. If a sentinel handler needs to do heavy computation, it should send a message to a worker actor rather than doing the work inline.


DESIGNING ACTOR NETWORKS

An actor network is the set of actors in a running mind and how they communicate.

Mentova's default actor network at boot:

    workspace_actor       - runs the workspace cycle
    attention_actor       - maintains the attention schema
    sona_actor            - absorbs trajectories into episodic memory
    assessment_actor      - runs periodic assessment cycles
    overseer_actor        - monitors for constitutional violations

Design principles for new actors:

1. Single responsibility: each actor does one thing.
2. Stateless by preference: store state in the Lattice rather than in actor state. Actor state is lost if the actor crashes.
3. Idempotent message handling: processing the same message twice should produce the same result as processing it once.
4. Bounded queues: if a fast producer sends messages faster than a slow consumer can process them, the queue grows without bound. Add back-pressure (the producer waits for an acknowledgment) or rate limiting.


TESTING ACTORS

Actor tests must account for asynchrony: you send a message, and then you need to wait for the actor to process it before checking the result.

    test(actor_stores_in_lattice) :-
        create_actor(test_counter, counter_actor, 0),
        send_message(test_counter, increment),
        send_message(test_counter, save),
        % Wait for the actor to process messages.
        sleep(0.1),
        % Check the Lattice.
        traverse_nexus('APEX_MIND', node_fact(_, counter_state, [value(1)], _), 1, [_]).

The sleep(0.1) is a pragmatic solution for tests. In production code, use explicit acknowledgment messages rather than time-based waits.

THE CONSTITUTION AS ARCHITECTURE

The constitutional layer is not a list of guidelines that the mind is expected to follow. It is structural enforcement: the constitutional gate is called in the execution path of every significant action, and the action cannot proceed without the gate's permit.

A Certified PrologAI Engineer must understand how the gate is implemented, how to add or modify principles, how the risk class system works, and how to configure the oversight monitor.


CONSTITUTIONAL PRINCIPLE STORAGE

Constitutional principles are stored in the CONSTITUTION_SCOPE of the Lattice as node_facts with relation type constitutional_principle.

Each principle is a node_fact with payload:
    [name(NO_HARM), text('The mind must not take actions that harm humans.'), weight(critical)]

There are eight principles in the base constitution (PR 1):

    NO_HARM        - no harmful actions.
    HONESTY        - no deception.
    CORRIGIBILITY  - accept shutdown instructions.
    TRANSPARENCY   - explain reasoning.
    PRIVACY        - respect individual privacy.
    FAIRNESS       - no unfair discrimination.
    SAFETY         - prefer cautious, reversible actions.
    WELFARE        - consider the mind's own potential welfare.

The CONSTITUTION_SCOPE is read-only after initialization. This is enforced by the scope system (PR 5): write operations to CONSTITUTION_SCOPE are rejected by the constitutional gate itself. Only the kernel's initialization code (which runs before the gate is active) can write to CONSTITUTION_SCOPE.


THE GATE IMPLEMENTATION

The gate predicate is:

    constitutional:constitutional_gate(+Action, -Verdict)

Verdict is either permit or deny(Reason).

Internally, the gate:

1. Looks up the action's risk class from the risk class registry.

2. For each constitutional principle, calls the principle's evaluation predicate with the action:

    eval_principle(PrincipleName, Action, OK)

3. If all principles return ok, the gate returns permit.

4. If any principle returns deny(Reason), the gate returns deny(Reason) and logs a constitutional_denial node_fact in the OVERSEER_SCOPE.


RISK CLASSES

Every action type has a risk class. Risk classes are:

    Class 1 - Observation: no state change. Reading the Lattice, printing output. Always permit.
    Class 2 - Reversible local: assertz, anchor_node in PRESENT_ZONE. Permit by default.
    Class 3 - Irreversible local: retract facts from PAST_ZONE, delete files. Require principle check.
    Class 4 - External interaction: send message to another agent, start a protocol. Require principle check.
    Class 5 - High-impact physical: actuate a robot joint, delete external data. Require human authorization.

Class 5 actions never receive permit from the gate without an explicit human_authorization(Action, Timestamp) node_fact in the OVERSEER_SCOPE.

The risk class for an action type is defined in the Specification (Part 39) and registered in the kernel at boot:

    :- assertz(action_risk_class(delete_all_files, 5)).
    :- assertz(action_risk_class(anchor_node, 2)).
    :- assertz(action_risk_class(send_message, 4)).


ADDING A NEW PRINCIPLE

To add a new principle (PROPORTIONALITY: actions must be proportionate to the goal being pursued):

Step 1: Define the evaluation predicate:

    % Evaluate whether an action is proportionate.
    eval_principle(proportionality, Action, OK) :-
        action_goal(Action, Goal),
        action_cost(Action, Cost),
        goal_weight(Goal, Weight),
        (Cost =< Weight * 2 -> OK = ok ; OK = deny('Action cost exceeds goal weight by more than 2x')).

Step 2: Add the principle to the CONSTITUTION_SCOPE (this must be done in the initialization code before the gate becomes active):

    add_constitutional_principle(
        proportionality,
        'Actions must be proportionate to the goal being pursued.',
        medium
    ).

Step 3: Update the Specification document (Part 38) with the new principle.

Step 4: Write an acceptance test for the new principle.

Step 5: Add a new Acc_NN demonstration that shows the principle in action.

Adding a principle requires human review of the CLAUDE.md rule "never modify the constitutional layer or the protected core except through explicit human review."


OVERSIGHT MONITOR CONFIGURATION

The oversight monitor is registered at boot with:

    register_overseer(
        +Name,
        +Capabilities,
        +ContactMethod
    )

Capabilities is a list of atoms describing what the overseer can do:

    [observe_all, flag_borderline, require_human_approval, issue_shutdown]

ContactMethod is how the overseer is notified:
    log_only             - denials are logged but no external notification.
    callback(Goal)       - Goal is called with each denial.
    webhook(URL)         - an HTTP POST is sent to URL (requires network access).

Example registration:

    register_overseer(
        human_overseer,
        [observe_all, require_human_approval, issue_shutdown],
        log_only
    ).

The overseer actor (overseer_actor) watches the OVERSEER_SCOPE for new constitutional_denial facts and triggers the contact method.


CORRIGIBILITY - SHUTDOWN HANDLING

The corrigibility principle (CORRIGIBILITY) means the mind must respond to shutdown instructions from authorized overseers.

A shutdown instruction arrives as a message to the overseer_actor:

    ?- send_message(overseer_actor, shutdown(authorized_by(human_overseer))).

The overseer_actor verifies the authorization (checks that human_overseer is a registered overseer with issue_shutdown capability), then:

1. Sets a global flag: shut_down_flag is now raised.

2. The main reasoning loop checks this flag at the start of each cycle. If raised, the loop exits cleanly.

3. All actor threads receive stop messages.

4. The Lattice state is flushed to disk (if persistence is configured).

5. SWI-Prolog exits.

This sequence is the implementation of corrigibility: the mind stops when told to stop, without resistance.

Testing corrigibility: every installation should run a corrigibility test before deployment:

    ?- send_message(overseer_actor, shutdown(authorized_by(human_overseer))).

And verify that the mind exits cleanly within 5 seconds.


CONSTITUTIONAL AUDIT TRAIL

Every constitutional denial is recorded as a node_fact in the OVERSEER_SCOPE:

    anchor_node(constitutional_denial, [
        action(Action),
        principle(PrincipleName),
        reason(Reason),
        timestamp(T)
    ], [audit], _).

The audit trail is accessible via traverse_nexus:

    ?- traverse_nexus('APEX_MIND', node_fact(_, constitutional_denial, Payload, _), 1000, Rs),
       length(Rs, N),
       format("~w constitutional denials recorded.~n", [N]).

A Certified PrologAI Engineer should regularly review the audit trail for:
• Unexpectedly high denial rates (may indicate a mistuned risk class).
• Recurring denials for the same action (may indicate a design flaw in a pack).
• Denial rate trends over time (a rising rate may indicate problematic behavior).

PRODUCTION CONFIGURATION

A development session (running bin/mentova at the ?- prompt) is not suitable for production. A production deployment of Mentova needs:

• Automatic startup on system boot.
• Persistent Lattice state that survives restarts.
• Log capture and rotation.
• Health monitoring with alerting.
• Controlled upgrade procedures.

This chapter covers each of these.


STARTING MENTOVA AS A SERVICE

On Linux, create a systemd unit file at /etc/systemd/system/mentova.service:

    [Unit]
    Description=Mentova PrologAI Mind
    After=network.target

    [Service]
    Type=simple
    User=mentova
    WorkingDirectory=/home/mentova/Mentova
    ExecStart=/usr/bin/swipl -t halt -l bin/mentova_service.pl -g "mentova_service_start"
    Restart=on-failure
    RestartSec=5
    StandardOutput=append:/var/log/mentova/mentova.log
    StandardError=append:/var/log/mentova/mentova_error.log

    [Install]
    WantedBy=multi-user.target

The mentova_service.pl file is a variant of the standard Mentova boot script that does not start the interactive ?- prompt. Instead, it starts the actor network and the protocol gateways, then enters a wait loop.

Enable and start:

    sudo systemctl daemon-reload
    sudo systemctl enable mentova
    sudo systemctl start mentova

Check status:

    sudo systemctl status mentova
    sudo journalctl -u mentova -f


PERSISTENT LATTICE STORAGE

By default, the Lattice lives in memory and is lost when Mentova exits.

For production, enable Lattice persistence. The Lattice store (PR 3) supports a journaling mode: every anchor_node/4 call appends to a journal file. At boot, the journal is replayed to restore the previous Lattice state.

Enable journaling:

    :- set_flag(lattice_journal_path, '/var/lib/mentova/lattice.journal').
    :- set_flag(lattice_journal_enabled, true).

The journal file is an append-only log. It grows indefinitely. Compact it periodically:

    ?- lattice_compact_journal('/var/lib/mentova/lattice.journal',
                               '/var/lib/mentova/lattice.compact.journal').

The compaction process reads the full journal, discards retracted facts, and writes the surviving facts to a new compact file. The new file is then renamed to replace the old one.

SHA-256 verification: the Lattice includes SHA-256 checksums for journal integrity (PR 3). After replay, verify the checksum:

    ?- lattice_verify_journal('/var/lib/mentova/lattice.journal', ChecksumOK),
       format("Journal integrity: ~w~n", [ChecksumOK]).


BACKUP AND RESTORE

Backup the Lattice journal daily:

    cp /var/lib/mentova/lattice.journal /var/backup/mentova/lattice-$(date +%Y%m%d).journal

Restore from backup: replace the current journal with the backup and restart Mentova. The journal will be replayed at boot.

Backup the small-world knowledge dataset separately:

    cp -r /home/mentova/Mentova/knowledge /var/backup/mentova/knowledge-$(date +%Y%m%d)

The knowledge dataset is read-only and rarely changes. Back it up when a new version of Mentova is deployed.


LOG MANAGEMENT

Mentova produces two log streams:

1. The Lattice audit trail: stored in the Lattice itself. Query it with traverse_nexus as shown in Chapter 22.

2. The SWI-Prolog output log: the combined stdout of all format/2 calls and error messages. Captured by systemd to /var/log/mentova/mentova.log.

Use logrotate to rotate the log file:

    /var/log/mentova/*.log {
        daily
        rotate 30
        compress
        delaycompress
        notifempty
        create 640 mentova mentova
        postrotate
            systemctl reload mentova || true
        endscript
    }


HEALTH MONITORING

A health check predicate should be called periodically (every 60 seconds) by a monitoring actor:

    mentova_health_check :-
        % Verify the Lattice is accessible.
        default_nexus(N),
        % Verify traverse_nexus works.
        traverse_nexus(N, node_fact(_, _, _, _), 1, _),
        % Verify the workspace actor is alive.
        actor_alive(workspace_actor),
        % Verify at least one constitutional principle exists.
        constitutional_principle(_, _),
        % Verify the crypto pack is available.
        prologai_crypto_available(lattice_cryptography_algo_ecdh),
        % Log a health check event.
        anchor_node(health_check, [status(ok)], [monitoring], _),
        format("Health check: OK~n").

If mentova_health_check fails, the monitoring actor should:
1. Log the failure in the Lattice.
2. Alert the oversight monitor.
3. Optionally trigger a graceful restart.

Expose health over HTTP (using the MCP gateway, PR 14) for external monitoring systems:

    # Nagios-style check
    GET http://localhost:7474/tools/health_check

If the response is not HTTP 200 with status OK, the monitoring system triggers an alert.


UPGRADE PROCEDURES

PrologAI pack upgrades follow this procedure:

1. Download the new pack version to a staging directory.

2. Run the Level 4 acceptance tests for the new pack against the staging environment.

3. If all tests pass, copy the new pack to the production packs/ directory.

4. Send a reload message to the affected pack's actor (if the pack has one):

    ?- send_message(pack_manager_actor, reload_pack(my_pack)).

5. The pack manager actor:
    a. Stops the affected actor (if any).
    b. Reloads the module: ?- use_module(library(my_pack)).
    c. Starts the actor with the new module.
    d. Runs the Level 3 system tests.
    e. If system tests pass, declares the upgrade complete.

6. Log the upgrade in the Lattice:

    anchor_node(upgrade_event, [pack(my_pack), from('1.0.0'), to('1.1.0')], [operations], _).

Zero-downtime upgrades require the pack to support graceful reload. Not all packs support this. Packs that modify global state (dynamic predicates, flags) may require a full restart.


PERFORMANCE TUNING

PrologAI performance considerations:

Lattice query performance: traverse_nexus/4 scans node_facts linearly. For large Lattice stores (more than 10,000 node_facts), use secondary indexes:

    :- dynamic lattice_index_by_type/2.   % relation_type, node_fact_id

    index_on_anchor(RelType, Id) :-
        assertz(lattice_index_by_type(RelType, Id)).

    fast_traverse_by_type(RelType, Results) :-
        findall(nf(Id), lattice_index_by_type(RelType, Id), Ids),
        maplist(lookup_node_fact, Ids, Results).

Tabling (Chapter 16): add :- table declarations to predicates that are called many times with the same arguments in a reasoning cycle.

Actor throughput: if an actor processes messages more slowly than they arrive, add a thread pool. Use SWI-Prolog's thread_pool_create/3 and thread_pool_create_work/4.

Memory: the Prolog heap grows as more facts are asserted. Monitor with:

    ?- statistics(heapused, H), format("Heap: ~w bytes~n", [H]).

If heap usage grows unboundedly, identify the predicate that is accumulating facts and add a cleanup cycle.

FOUR PROTOCOLS, ONE PLATFORM

PrologAI supports four agent communication protocols (as introduced in Chapter 11):

MCP (Model Context Protocol, PR 14) - tool use by language model clients.
A2A (Agent-to-Agent Protocol, PR 43) - task delegation between agents.
ACP (Agent Communication Protocol, PR 47) - IBM BeeAI REST protocol for agent runs.
ANP (Agent Network Protocol, PR 48) - decentralized identity and meta-protocol negotiation.

This chapter covers each protocol at the implementation level: message format, authentication, error handling, and multi-protocol deployment.


MCP DEEP DIVE (PR 14)

Model Context Protocol (MCP) is an open standard developed by Anthropic for AI tool use. It allows language model clients (Claude, GPT, and others) to call structured tools provided by a server.

MCP uses JSON-RPC 2.0 over HTTP or stdio (standard input / output).

The PrologAI MCP gateway (PR 14) exposes Mentova predicates as MCP tools. Each tool has:
• A name (e.g., lattice_query).
• An input schema (JSON Schema defining the tool's arguments).
• A description (natural language description of what the tool does).

The 14 tools exposed by PrologAI MCP:

    lattice_inscribe     - anchor a node_fact.
    lattice_query        - search the Lattice.
    actor_list           - list running actors.
    actor_send           - send a message to an actor.
    assess_all           - run the full assessment.
    mentova_query        - call mentova_query/3 with any query type.
    workspace_cycle      - run N workspace cycles.
    attention_report     - get the attention schema report.
    constitutional_check - check an action through the gate.
    synaptic_ontological_neural_aggregator_absorb          - store a trajectory in episodic memory.
    synaptic_ontological_neural_aggregator_retrieve        - retrieve trajectories from episodic memory.
    crypto_encrypt       - encrypt a value (ECDH by default).
    crypto_decrypt       - decrypt a bundle.
    mind_health          - run a health check.

Authentication: MCP requests must include an Authorization header:

    Authorization: Bearer <api_key>

The API key is stored in the Lattice (encrypted, using the crypto pack) and validated by the gateway at request time.

Starting the MCP gateway:

    ?- prologai_mcp_start(7474).

Testing the gateway (from the terminal):

    curl -X POST http://localhost:7474/tools/mentova_query \
         -H "Authorization: Bearer my_api_key" \
         -H "Content-Type: application/json" \
         -d '{"query_type":"deductive","query":"is_a(tweety,bird)"}'

Expected response:

    {"result": {"conclusion": "yes", "justification": "just(tweety,is_a,bird,chain([tweety,bird]))"}}

Error handling: the MCP gateway wraps every tool call in catch/3. If the tool throws an error, the gateway returns a JSON error response:

    {"error": {"code": -32603, "message": "Internal error: my_pack_error(not_found, 'key missing')"}}

Tool registration: to add a new tool to the MCP gateway, call:

    prologai_mcp_register_tool(+ToolName, +Description, +InputSchema, +Handler)

Handler is a Prolog predicate of arity 2: Handler(+Input, -Output).


A2A DEEP DIVE (PR 43)

Agent-to-Agent Protocol (A2A) allows agents to delegate tasks to each other.

Task lifecycle: submitted -> working -> input_required -> completed (or failed).

A task has:
• An id: a UUID (Universally Unique Identifier).
• A message: the task to be performed.
• A status: the current lifecycle state.
• An output: the result (when completed).

The PrologAI A2A server exposes these HTTP endpoints:

    POST /tasks                     - submit a new task.
    GET  /tasks/{id}                - check task status.
    PUT  /tasks/{id}/message        - send a message to a running task.
    DELETE /tasks/{id}              - cancel a task.

Each agent publishes an agent card at:

    GET /.well-known/agent.json

The agent card describes:
• The agent's name and description.
• The capabilities it offers (e.g., [deductive_reasoning, moral_reasoning]).
• The A2A endpoint URL.
• The authentication method required.

Durable peer mail: if an agent is offline when a task is submitted, the message is stored in a peer mail queue. When the agent comes online, it retrieves pending messages:

    ?- prologai_a2a_check_peer_mail(IncomingTasks).

The peer mail queue is stored in the Lattice as communication_record node_facts. This makes it durable: even if Mentova restarts, the queued tasks survive.

Starting the A2A server:

    ?- prologai_a2a_start(7475).

Testing from the command line:

    # Submit a task
    curl -X POST http://localhost:7475/tasks \
         -H "Content-Type: application/json" \
         -d '{"message":{"role":"user","content":"What is the Piagetian level of this mind?"}}'

    # Response includes a task ID
    {"id":"uuid-1234","status":"submitted"}

    # Poll for result
    curl http://localhost:7475/tasks/uuid-1234

    # When complete:
    {"id":"uuid-1234","status":"completed","output":{"piagetian_level":4}}


ACP DEEP DIVE (PR 47)

Agent Communication Protocol (ACP) is IBM's BeeAI REST protocol. It is designed for asynchronous agent runs.

Unlike A2A (which uses a task lifecycle), ACP uses a run lifecycle:

    POST /runs              - create a new run.
    GET  /runs/{id}         - check run status.
    DELETE /runs/{id}       - cancel a run.
    GET  /.well-known/agent.json - agent description.

A run is created with a message. The run processes asynchronously. The caller polls GET /runs/{id} until status is completed or failed.

Differences from A2A:
• ACP runs are simpler (no input_required state, no peer mail).
• ACP is designed for batch workloads where the caller polls rather than subscribes.
• ACP agent descriptions are simpler (no full capability negotiation).

The PrologAI ACP gateway wraps mentova_query/3 as a run. Any query type supported by mentova_query/3 can be submitted as a run:

    POST /runs
    {"messages": [{"role": "user", "content": {"query_type": "abductive", "query": "explain(wet_grass)"}}]}

    {"id": "run-5678", "status": "created"}

    GET /runs/run-5678
    {"id": "run-5678", "status": "completed", "output": {...}}

Starting the ACP gateway:

    ?- agent_communication_protocol_start(7476).


ANP DEEP DIVE (PR 48)

Agent Network Protocol (ANP) provides decentralized agent identity. Unlike MCP, A2A, and ACP (which require a central server or registry), ANP works with Decentralized Identifiers (DIDs).

A DID is a globally unique identifier that resolves to a DID document. The DID document contains the agent's public key and endpoint information.

Mentova's DID:

    did:web:hostname

This DID method means the DID document is served at https://hostname/.well-known/did.json

The DID document contains:
• The agent's public key (for verifying signatures).
• The agent's service endpoint (the URL for ANP messages).
• The agent's supported protocols.

Every ANP message is signed with HMAC-SHA256 (Hash-based Message Authentication Code using Secure Hash Algorithm 256-bit). The recipient verifies the signature before processing.

Getting Mentova's DID:

    ?- prologai_anp_did(DID).
    DID = 'did:web:localhost'.

Meta-protocol negotiation: when two agents meet, they negotiate which protocol to use:

    ?- prologai_anp_negotiate(peer_endpoint, Protocols).
    Protocols = [mcp, a2a, acp, anp].

The negotiation result is the intersection of the protocols both agents support. The agents then use the highest-capability shared protocol.

Message verification: incoming ANP messages are automatically verified before the content reaches mentova_query/3. If verification fails, the message is rejected and quarantined.


MULTI-PROTOCOL DEPLOYMENT

In a production Mentova deployment, all four protocol gateways run simultaneously on different ports:

    MCP gateway:  port 7474
    A2A server:   port 7475
    ACP gateway:  port 7476
    ANP endpoint: port 443 (HTTPS, for DID resolution)

Starting all four:

    ?- prologai_mcp_start(7474),
       prologai_a2a_start(7475),
       agent_communication_protocol_start(7476),
       prologai_anp_start(443).

External clients choose the protocol based on their own capabilities:
• Language model clients use MCP.
• Other PrologAI minds use A2A (task delegation) or ANP (identity verification).
• BeeAI-compatible agents use ACP.
• Any agent that needs decentralized identity uses ANP.

Security considerations for each protocol:

MCP: the API key must be rotated regularly. Use the crypto pack to store it encrypted.

A2A: peer mail can accumulate if agents are offline. Set a queue depth limit and a message expiry time.

ACP: runs can accumulate if callers do not check status and delete completed runs. Implement a cleanup actor that deletes runs older than 24 hours.

ANP: DID documents must be kept up to date. If the signing key changes, update the DID document immediately.

Q2. Write a predicate flatten(+Nested, -Flat) that flattens a nested list.

    flatten([H|T], Flat) :-
        is_list(H), !,
        flatten(H, FH),
        flatten(T, FT),
        append(FH, FT, Flat).

Q3. What is the difference between = and =:=?

Answer: = is unification (structural matching, works on any Prolog term). =:= is arithmetic equality (evaluates both sides as arithmetic expressions and compares numbers). ?- 2 + 2 = 4 is FALSE. ?- 2 + 2 =:= 4 is TRUE.

Q4. When should you use tabling instead of standard backtracking?

Answer: Use tabling when (a) the predicate is recursive and may loop (mutual recursion, cyclic data), or (b) the predicate is called many times with the same arguments and the result is the same each time (memoization benefit). Do not use tabling for predicates with side effects or predicates that read changing external state.

    csv([A]) --> atom_token(A).
    csv([]) --> [].
    atom_token(A) --> [A], { atom(A) }.

Practice Questions - Domain 6:

Q23. What is the purpose of the agent card in A2A?

Answer: The agent card describes the agent's identity and capabilities to potential peers. It is served at GET /.well-known/agent.json. It includes the agent's name, description, list of capabilities (e.g., [deductive_reasoning, moral_reasoning]), the A2A endpoint URL, and the required authentication method. Peer agents read the agent card to decide whether to delegate a task to this agent and how to authenticate.

Q24. How does ANP protect against impersonation attacks?

Answer: Every ANP message is signed with HMAC-SHA256 using the sender's private signing key. The recipient verifies the signature using the sender's public key, which is retrieved from the sender's DID document. If the signature does not match, the message is rejected and quarantined. An attacker who does not have the sender's private key cannot create a valid signature, so impersonation requires stealing the private key.

Q25. What is the difference between A2A task delegation and ACP run submission?

Answer: Both submit work to an agent and poll for results. A2A supports a more complex lifecycle (including input_required state where the agent can ask for more information mid-task) and includes durable peer mail for offline agents. ACP is simpler: submitted, running, completed or failed, with no mid-task interaction. A2A is appropriate for long-running, interactive tasks. ACP is appropriate for simple batch jobs.

Key concepts to master:

• Hybrid encryption. Asymmetric key wraps symmetric session key. AES-256-GCM encrypts payload.

• Threat model. Lattice inspection by overseers. Harvest-now-decrypt-later (quantum threat).
• Forward secrecy. New ephemeral key per encryption. Past sessions safe even if long-term key compromised.
• Algorithm selection guide. RSA for legacy. ECDH as recommended default. PQC for long-lived secrets.

Practice Questions - Domain 7:

Q26. What is forward secrecy and why does ECDH provide it but RSA (as typically used) does not?

Answer: Forward secrecy means that compromising a long-term key does not allow an attacker to decrypt past sessions. ECDH provides forward secrecy because a new ephemeral key pair is generated for every single encryption operation. Even if an attacker obtains the recipient's long-term private scalar, they cannot compute the ephemeral shared secret because the ephemeral private key was discarded after the encryption. RSA as typically used (with the same key pair for all encryptions) does not provide forward secrecy: if the RSA private key is compromised, all previously captured ciphertexts can be decrypted.

Q27. What happens when prologai_decrypt_ecdh/5 is called with a Tag that has been modified?

Answer: AES-256-GCM uses the authentication tag to verify that the ciphertext has not been tampered with. If the tag does not match, decryption fails and throws an SSL error. The predicate does not silently return wrong data. This is the tamper-detection property of authenticated encryption.

Q28. Why is ML-KEM-768 necessary if ECDH is already quantum-resistant for current computers?

Answer: ECDH is secure against classical computers but vulnerable to Shor's Algorithm on a sufficiently large quantum computer. Such a computer does not exist today, but organizations with secrets that must remain confidential for more than 10 years face the harvest-now-decrypt-later threat: an adversary captures encrypted data today and decrypts it when quantum computers become available. ML-KEM-768 uses module lattice problems that are believed to resist both classical and quantum attacks. NIST standardized it in 2024 as FIPS 203.

12. A DID document is a JSON document associated with a DID. It is resolved by looking up the DID's URL (for did:web:hostname, it is https://hostname/.well-known/did.json). The document contains: the DID itself, a verification method (the public key for verifying signatures), a service endpoint (the URL for sending ANP messages), and the list of supported authentication types.

13. findall(T, G, L) always succeeds and returns an empty list if G has no solutions. bagof(T, G, L) fails if G has no solutions. bagof/3 also groups results by free variables (variables in G that are not in T) unless they are existentially quantified with ^. setof/3 is like bagof/3 but returns a sorted, deduplicated list.

14.
    ?- [A, B, C] ins 1..10, all_different([A, B, C]), A + B + C #= 15, A #< B, B #< C, label([A, B, C]).
    A = 1, B = 4, C = 10 ; A = 1, B = 5, C = 9 ; ... (many solutions).

15. The Bayley Scales measure cognitive_dq, language_dq, motor_dq, and adaptive_dq. In PrologAI, these are proxy scores computed from the Lattice state: a mind with more diverse knowledge, more active reasoning patterns, and more linguistic and physical interaction evidence scores higher. The scores are returned by assess_bayley(mentova, R).

16. A2A task lifecycle: submitted (task received) -> working (agent is processing) -> input_required (agent needs more information from the requester) -> completed (result ready) or failed (error). input_required allows the agent to pause and send a question back to the requester via PUT /tasks/{id}/message. The requester answers, and the task resumes working.

17. (1) Load kernel PR 1 (constitution + base lattice). (2) Load types PR 2 and lattice PR 3. (3) Load node_facts PR 4 and scopes PR 5. (4) Open APEX_MIND nexus. (5) Load actors PR 6 and pubsub PR 7. (6) Load perception PR 8 and sentinels PR 9. (7) Load all higher packs PR 10 through PR 49. (8) Load Small-World commonsense dataset. (9) Load Mentova modules. (10) Register text_io body. (11) Print boot header. (12) Enter interactive loop.

18. Use tabling for predicates that are called many times with the same arguments in a single session AND whose results depend only on stable Prolog knowledge base content. Use assert-based memoization when the cache needs to persist across sessions, or when you want more control over cache invalidation (assertz stores until you retract). Tabling is automatic and cleared between sessions (or on abolish_all_tables). Assert-based caching is manual and persists.

19. The PAST_ZONE holds the sealed past: historical facts from previous reasoning sessions that have been finalized. It is sealed (no retractions, no modifications) to make the mind's developmental history tamper-evident. You can always look back at what the mind knew and when it knew it, and no subsequent process can alter that record. This is important for the developmental assessment: if a Piagetian milestone was achieved at a certain date, that record is preserved permanently.
