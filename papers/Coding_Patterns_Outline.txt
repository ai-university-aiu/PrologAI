CODING (COMPUTER PROGRAMMING) ENGINEERING DESIGN PATTERNS

A COMPLETE AND COMPREHENSIVE TOPIC OUTLINE

Prepared for the textbook: Certified Coding (Computer Programming) Design Patterns Engineer (CCDPE)


HOW TO READ THIS OUTLINE

This document is a structured catalog of the design patterns used to write computer programs.

Coding, also called computer programming, is the act of writing instructions that a computer follows to do useful work.

A design pattern is a named, reusable solution to a problem that comes up again and again, written down so a programmer can apply a tested answer instead of inventing one each time.

This outline is written for a newcomer, so each part begins with a plain-language note before listing its subtopics.

Acronyms are spelled out the first time they appear, and technical terms are defined the first time they are used.

This is the companion to the separate coding principles outline: principles tell you what good code looks like, while patterns give you concrete, named structures to build with.

The outline is organized into parts, and each part is broken into numbered chapters and bulleted subtopics.

PART 2 - CREATIONAL PATTERNS

This part covers patterns that control how objects are created.

An object is a bundle of data and the operations that work on that data.

Creational patterns make object creation flexible, controlled, and decoupled from the rest of the code.

2.1 Factory Method

• Factory Method, meaning a method that decides which object to create

• Letting subclasses choose the concrete type

• Decoupling code from the exact classes it builds

• When creation logic should be centralized

2.2 Abstract Factory

• Abstract Factory, meaning a factory that creates families of related objects

• Producing matching sets of objects that work together

• Swapping whole families without changing client code

• The trade-off of added structure for added flexibility

2.3 Builder

• Builder, meaning step-by-step construction of a complex object

• Separating construction from the final representation

• Building different variations through the same process

• Avoiding constructors with too many parameters

2.4 Prototype

• Prototype, meaning creating new objects by copying an existing one

• Cloning, meaning making a copy of an object

• Avoiding expensive creation by duplicating a template

• Deep versus shallow copying, meaning whether nested data is also copied

2.5 Singleton

• Singleton, meaning ensuring only one instance of a class exists

• Providing a single global point of access

• The trade-offs and criticisms, such as hidden global state

• Safer alternatives in modern designs


PART 3 - STRUCTURAL PATTERNS

This part covers patterns for composing objects into larger structures.

Composition means assembling smaller parts into a larger whole.

Structural patterns describe how to combine objects to gain new capabilities while keeping the design flexible.

3.1 Adapter

• Adapter, meaning making two incompatible interfaces work together

• An interface, meaning the agreed way code interacts with a component

• Wrapping one component so it matches what another expects

• Connecting new code to old or external systems

3.2 Bridge

• Bridge, meaning separating an abstraction from its implementation so each can vary

• Avoiding an explosion of combined subclasses

• Letting two dimensions of a design change independently

3.3 Composite

• Composite, meaning treating individual objects and groups of objects the same way

• Building tree structures of parts and wholes

• Uniform handling of leaves and branches

• Common uses such as menus, files, and graphics

3.4 Decorator

• Decorator, meaning adding behavior to an object without changing its type

• Wrapping an object to extend its abilities at runtime

• Stacking multiple decorators for combined behavior

• An alternative to deep inheritance

3.5 Facade

• Facade, meaning a simple front for a complex subsystem

• Hiding internal complexity behind one clear entry point

• Reducing coupling between clients and subsystems

3.6 Flyweight

• Flyweight, meaning sharing common data among many objects to save memory

• Separating shared state from unique state

• Efficient handling of very large numbers of similar objects

3.7 Proxy

• Proxy, meaning a stand-in that controls access to another object

• Adding control such as lazy loading, caching, or security

• Lazy loading, meaning delaying work until it is actually needed

• Common uses such as remote, virtual, and protection proxies


PART 4 - BEHAVIORAL PATTERNS

This part covers patterns for how objects communicate and share responsibility.

Behavioral patterns describe the flow of control and the assignment of duties among objects.

4.1 Chain of Responsibility

• Chain of Responsibility, meaning passing a request along a line of handlers until one handles it

• Decoupling senders from receivers

• Flexible ordering and composition of handlers

4.2 Command

• Command, meaning packaging a request as an object

• Supporting undo, queuing, and logging of operations

• Decoupling the requester from the performer

4.3 Iterator

• Iterator, meaning stepping through a collection without exposing its internals

• A collection, meaning a group of items held together

• Uniform traversal across different collection types

4.4 Mediator

• Mediator, meaning a central object that coordinates how others interact

• Reducing direct connections among many objects

• Centralizing complex communication logic

4.5 Memento

• Memento, meaning capturing an object's state so it can be restored later

• Supporting undo and checkpoints

• Preserving encapsulation while saving state

4.6 Observer

• Observer, meaning notifying interested objects when something changes

• Publish and subscribe, meaning announcing events to many listeners

• Keeping parts in sync without tight coupling

4.7 State

• State, meaning changing an object's behavior as its internal state changes

• Replacing large conditional blocks with state objects

• Modeling clear transitions between states

4.8 Strategy

• Strategy, meaning selecting an interchangeable algorithm at runtime

• Encapsulating each algorithm behind a common interface

• Swapping behavior without changing the caller

4.9 Template Method

• Template Method, meaning defining the skeleton of an operation and letting steps vary

• Fixing the overall sequence while customizing parts

• Reusing shared structure across variations

4.10 Visitor

• Visitor, meaning adding new operations to objects without changing their classes

• Separating algorithms from the data they operate on

• Trade-offs when the object structure changes often

4.11 Interpreter

• Interpreter, meaning representing and evaluating a simple language or grammar

• Modeling rules as a structure that can be executed

• Suitable problems and its practical limits


PART 5 - SUPPLEMENTARY OBJECT-ORIENTED PATTERNS

This part covers widely used object patterns beyond the original catalog.

These patterns appear constantly in everyday application code.

5.1 Dependency Injection

• Dependency injection, meaning supplying a component's dependencies from outside

• Inversion of control, meaning a framework or caller decides what to wire together

• Improving testability and flexibility

5.2 Null Object

• Null Object, meaning a do-nothing object used instead of an empty value

• Avoiding repeated checks for missing values

• Simplifying calling code

5.3 Object Pool

• Object Pool, meaning reusing a set of ready objects instead of creating new ones

• Reducing the cost of expensive creation

• Managing limited resources such as connections

5.4 Other Common Patterns

• Lazy initialization, meaning creating something only when first needed

• Value object, meaning an object defined by its data rather than identity

• Data transfer object, meaning a simple object for carrying data between parts

• Module pattern, meaning grouping related code behind a clear boundary


PART 6 - CONCURRENCY PATTERNS

This part covers patterns for doing many things at once safely.

Concurrency means multiple tasks making progress at the same time.

6.1 Coordination Patterns

• Producer-consumer, meaning one part creates work and another processes it

• A queue, meaning a line where items wait to be handled in order

• Read-write lock, meaning allowing many readers but one writer

• Monitor, meaning safely guarding shared data with controlled access

6.2 Execution Patterns

• Thread pool, meaning reusing a fixed set of workers for many tasks

• Worker queue, meaning distributing tasks among workers

• Future or promise, meaning a placeholder for a result that is not ready yet

• Actor model, meaning independent units that communicate only by messages

6.3 Isolation and Safety Patterns

• Bulkhead, meaning isolating resources so one failure does not sink everything

• Isolated thread pools, meaning separate pools per dependency

• Immutability, meaning data that cannot change, to avoid conflicts

• Avoiding deadlocks, meaning tasks stuck waiting on each other forever


PART 7 - ARCHITECTURAL PATTERNS

This part covers patterns for organizing an entire application.

Architecture is the high-level structure of a system and its major parts.

7.1 Layered and Tiered Patterns

• Layered architecture, meaning stacking presentation, logic, and data layers

• Separating responsibilities across layers

• Controlling dependencies between layers

7.2 Presentation Patterns

• Model-View-Controller (MVC), meaning separating data, display, and input handling

• Model-View-ViewModel (MVVM), meaning binding the view to a view model

• Model-View-Presenter (MVP), meaning a presenter mediating view and model

7.3 Core-Centric Patterns

• Hexagonal architecture, also called ports and adapters, which isolates core logic behind adapters

• A port, meaning a defined entry or exit point of the core logic

• An adapter, meaning code connecting an external system to a port

• Clean and onion architectures, which keep dependencies pointing inward

7.4 Distributed Structure Patterns

• Monolith, meaning one deployable application containing everything

• Microservices, meaning building a system from small independent services

• Service-oriented architecture, meaning organizing around shared services

• Serverless, meaning running code without managing servers directly

7.5 Event-Driven Patterns

• Event-driven architecture, meaning components reacting to events

• Publish and subscribe communication

• Event streaming, meaning continuous flows of events

• Loose coupling through events


PART 8 - ENTERPRISE APPLICATION PATTERNS

This part covers patterns for large business applications.

Enterprise applications are large systems that manage important business data and processes.

8.1 Domain Logic Patterns

• Transaction Script, meaning organizing logic as straightforward procedures

• Domain Model, meaning rich objects that hold both data and behavior

• Service Layer, meaning a boundary that defines available operations

8.2 Data Source Patterns

• Table Data Gateway, meaning one object that accesses a database table

• Row Data Gateway, meaning an object that represents a single database row

• Active Record, meaning an object that carries both data and its own database access

• Data Mapper, meaning a layer that moves data between objects and the database

8.3 Object-Relational Patterns

• Unit of Work, meaning tracking changes and saving them as one batch

• Identity Map, meaning ensuring each record is loaded only once

• Lazy Load, meaning fetching related data only when needed

• Object-relational mapping, meaning translating between objects and tables

8.4 Web Presentation Patterns

• Front Controller, meaning one entry point handling all requests

• Page Controller, meaning a controller per page or action

• Template View, meaning generating pages from templates

• Application Controller, meaning centralizing screen flow


PART 9 - DATA AND PERSISTENCE PATTERNS

This part covers patterns for storing, reading, and changing data reliably.

Persistence means saving data so it survives after a program stops running.

9.1 Repository and Access Patterns

• Repository, meaning a collection-like interface that hides data access

• Query object, meaning representing a database query as an object

• Data access object, meaning an object dedicated to reading and writing data

9.2 Command Query Responsibility Segregation (CQRS)

• Command Query Responsibility Segregation (CQRS), meaning separating writes from reads

• A command, meaning an operation that changes state

• A query, meaning an operation that reads state

• Benefits and added complexity of separate models

9.3 Event Sourcing

• Event Sourcing, meaning storing all changes as a sequence of events

• Rebuilding current state by replaying events

• A built-in audit log of everything that happened

• Pairing event sourcing with CQRS

9.4 Consistency Patterns

• Optimistic locking, meaning detecting conflicts at save time

• Pessimistic locking, meaning preventing conflicts by holding a lock

• Caching, meaning keeping copies of data for fast access

• Eventual consistency, meaning data becomes consistent after a short delay


PART 10 - DISTRIBUTED SYSTEMS AND MICROSERVICES PATTERNS

This part covers patterns for systems spread across many services and machines.

A distributed system is software whose parts run on separate computers and cooperate over a network.

10.1 Communication Patterns

• Application Programming Interface (API) gateway, meaning a single entry point for many services

• Service discovery, meaning finding where services are located at runtime

• Backend for frontend, meaning a tailored gateway per client type

• Synchronous versus asynchronous communication

10.2 Reliability Patterns

• Circuit breaker, meaning stopping calls to a failing service to prevent cascades

• Retry with backoff, meaning trying again after waiting longer each time

• Bulkhead, meaning isolating resources so one failure stays contained

• Timeout and fallback, meaning giving up gracefully and using a backup

10.3 Data Coordination Patterns

• Saga, meaning coordinating a multi-service process with compensating undo steps

• Choreography, meaning services reacting to each other's events

• Orchestration, meaning a central coordinator directing the steps

• Outbox, meaning reliably publishing events alongside database changes

10.4 Decomposition Patterns

• Decompose by business capability or subdomain

• Aggregator, meaning combining results from several services

• Strangler fig, meaning gradually replacing a legacy system piece by piece

• Sidecar, meaning a helper deployed alongside a service


PART 11 - ENTERPRISE INTEGRATION AND MESSAGING PATTERNS

This part covers patterns for connecting separate applications.

Integration means making separate systems work together.

11.1 Messaging Foundations

• A message, meaning a packaged piece of data sent between systems

• A message channel, meaning the path a message travels

• Point-to-point versus publish and subscribe delivery

• Asynchronous messaging for loose coupling

11.2 Message Routing Patterns

• Content-based router, meaning sending a message based on its contents

• Message filter, meaning dropping unwanted messages

• Splitter and aggregator, meaning breaking apart and recombining messages

• Routing slip, meaning attaching a planned route to a message

11.3 Message Transformation Patterns

• Message translator, meaning converting between data formats

• Envelope wrapper, meaning adding and removing routing information

• Canonical data model, meaning one shared format all systems agree on

11.4 Endpoint and Reliability Patterns

• Competing consumers, meaning several workers sharing a queue

• Dead letter channel, meaning a place for messages that cannot be delivered

• Idempotent receiver, meaning safely handling duplicate messages

• Guaranteed delivery, meaning ensuring messages are not lost


PART 12 - CLOUD DESIGN PATTERNS

This part covers patterns for building software that runs in the cloud.

The cloud means computing resources rented over the internet rather than owned outright.

12.1 Reliability and Resilience Patterns

• Retry and circuit breaker applied in cloud services

• Health endpoint monitoring, meaning exposing checks of service health

• Throttling, meaning limiting load to protect a service

• Queue-based load leveling, meaning smoothing spikes with a queue

12.2 Scalability and Performance Patterns

• Auto-scaling, meaning adding or removing resources with demand

• Cache-aside, meaning loading data into a cache on demand

• Sharding, meaning splitting data across multiple stores

• Static content offloading, meaning serving fixed files from specialized storage

12.3 Operations and Management Patterns

• Infrastructure as code, meaning defining systems in version-controlled files

• Configuration management across environments

• External configuration store, meaning keeping settings outside the code

• Monitoring, logging, and observability in the cloud

12.4 Security Patterns in the Cloud

• Gatekeeper, meaning a protective front that validates requests

• Valet key, meaning granting limited, time-bound access

• Federated identity, meaning trusting an external identity provider

• Secrets management for keys and credentials


PART 13 - FUNCTIONAL AND REACTIVE PATTERNS

This part covers patterns from styles that avoid changing shared state.

Functional programming organizes software around pure functions and immutable data.

13.1 Functional Patterns

• Pure functions, meaning functions with no side effects

• A side effect, meaning a change to state outside the function

• Higher-order functions, meaning functions that take or return functions

• Map, filter, and reduce, meaning standard ways to transform collections

13.2 Composition Patterns

• Function composition, meaning combining small functions into larger ones

• Currying, meaning turning a multi-argument function into a chain of single-argument ones

• Pipelines, meaning passing data through a sequence of transformations

• Monads, meaning a structured way to chain computations with context

13.3 Reactive Patterns

• Reactive programming, meaning reacting to streams of data over time

• Observables and streams, meaning sequences of events you can subscribe to

• Backpressure, meaning controlling fast producers so consumers keep up

• Event loops and non-blocking input and output


PART 14 - LANGUAGE IDIOMS AND IMPLEMENTATION PATTERNS

This part covers smaller, language-level patterns and good habits.

An idiom is a common, language-specific way to write a small piece of code well.

14.1 Resource Management Idioms

• Resource Acquisition Is Initialization (RAII), meaning tying resource cleanup to object lifetime

• Try-with-resources and disposal patterns

• Guard clauses for early exit on bad input

14.2 Common Idioms

• Iterator and generator idioms for traversal

• Memoization, meaning caching the results of expensive calls

• Fluent interfaces, meaning chaining calls for readable configuration

• Immutable object construction

14.3 Pattern Implementation Notes

• How the same pattern looks different across languages

• Built-in language features that replace classic patterns

• Choosing idiomatic code over forced patterns


PART 15 - TESTING PATTERNS

This part covers reusable structures for testing code well.

A test is code that automatically checks whether other code behaves correctly.

15.1 Test Structure Patterns

• Arrange-act-assert, meaning set up, perform, then check

• Given-when-then, meaning describing a scenario and its expected result

• Test fixtures, meaning shared setup for a group of tests

15.2 Test Double Patterns

• A test double, meaning a stand-in used in place of a real dependency

• Mock, meaning a double that checks how it was called

• Stub, meaning a double that returns canned answers

• Fake and spy, meaning simplified or observed substitutes

15.3 Test Strategy Patterns

• The test pyramid, meaning many fast unit tests and fewer broad tests

• Test-driven development, meaning writing tests before the code

• Behavior-driven development, meaning describing behavior in plain language

• Contract testing, meaning verifying agreements between services


PART 16 - SECURITY PATTERNS

This part covers reusable structures for keeping software safe.

Security patterns build protection into the design rather than adding it later.

16.1 Access Control Patterns

• Authentication, meaning confirming who a user or system is

• Authorization, meaning controlling what they may do

• Role-based access control, meaning granting permissions by role

• Least privilege, meaning granting only the minimum access needed

16.2 Protection Patterns

• Input validation and sanitization at trust boundaries

• Output encoding to prevent injection

• Secure session management

• Secrets management for keys and credentials

16.3 Defensive Design Patterns

• Defense in depth, meaning multiple independent layers of protection

• Fail securely, meaning failing into a safe state

• Secure by default, meaning the safe option is the standard one

• Audit logging for accountability


PART 17 - ANTI-PATTERNS

This part covers tempting but flawed solutions to avoid.

An anti-pattern is a common solution that looks helpful but tends to cause more problems than it solves.

17.1 Structural Anti-Patterns

• Big ball of mud, meaning a system with no clear structure

• God object, meaning one unit that does far too much

• Spaghetti code, meaning tangled, hard-to-follow control flow

• Golden hammer, meaning forcing one familiar tool onto every problem

17.2 Process and Design Anti-Patterns

• Over-engineering, meaning adding complexity that is not needed

• Premature optimization, meaning tuning before measuring

• Copy-paste programming, meaning duplicating code instead of reusing it

• Reinventing the wheel, meaning rebuilding what already exists well

17.3 Distributed Anti-Patterns

• Distributed monolith, meaning services so tangled they must change together

• Chatty services, meaning too many small calls between services

• Shared database across services, meaning hidden coupling through data

• Ignoring failure in network calls

17.4 Avoiding Anti-Patterns

• Recognizing the warning signs early

• Refactoring toward clearer structure

• Choosing the simplest design that works

• Learning from reviews and real failures

APPENDIX A - GLOSSARY OF KEY TERMS

• Design pattern - a named, reusable solution to a recurring design problem

• Anti-pattern - a tempting but flawed solution that causes more problems than it solves

• Gang of Four (GoF) - the four authors of the foundational 1994 design patterns book

• Object - a bundle of data and the operations that work on that data

• Interface - the agreed way other code interacts with a component

• Creational pattern - a pattern that controls how objects are created

• Structural pattern - a pattern for composing objects into larger structures

• Behavioral pattern - a pattern for how objects communicate and share responsibility

• Model-View-Controller (MVC) - a pattern separating data, display, and input handling

• Repository - a collection-like interface that hides data access

• Command Query Responsibility Segregation (CQRS) - separating operations that write from those that read

• Event Sourcing - storing all changes as a sequence of events

• Saga - coordinating a multi-service process with compensating undo steps

• Circuit breaker - stopping calls to a failing service to prevent cascading failure

• Application Programming Interface (API) - a defined way for programs to interact
